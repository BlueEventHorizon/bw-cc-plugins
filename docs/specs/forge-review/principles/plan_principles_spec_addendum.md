## タスク受け入れ基準規範 [MANDATORY] (Appendix A.12 由来)

### 規範

1. 各タスクに受け入れ基準 (acceptance criteria) フィールドを必ず持たせる
2. 受け入れ基準は **Yes/No で判定可能** な記述にする (主観表現禁止)
3. 主要機能の実装タスクには対応するテストタスクを必ず計画書に含める
4. テストタスクは「実装タスクと同一グループ」または「直後の独立タスク」として配置する

### 受け入れ基準の主観表現禁止例

| ❌ 不許容 (主観表現)                       | ✅ 許容 (判定可能)                                                   |
| ------------------------------------------ | -------------------------------------------------------------------- |
| 「適切に動作する」                         | 「正常系 3 ケース・異常系 2 ケースで期待値と一致する」                |
| 「問題なく処理できる」                     | 「指定入力で例外を出さず指定 schema の出力を返す」                    |
| 「パフォーマンスが改善される」             | 「ベースライン処理時間に対して 50% 以内」(根拠 = 要件側で確定済み)    |
| 「ユーザビリティが向上する」               | (この粒度はタスクではなく要件側で受け入れ基準を定義する)              |

## テストタスク必須化規範 [MANDATORY] (Appendix A.12 由来)

### 規範

1. 主要機能 (要件 ID に直接紐づく機能) の実装タスクには、対応するテストタスクが計画書に必ず存在する
2. テストタスクの粒度は「単体テスト」「統合テスト」のいずれかを明示する
3. テストタスクは実装タスクの `build_check` 設計に組み込む (グループ完了時 or 独立)

### 例外 (許容範囲)

| 論点                                                         | ✅ 許容                                                | ❌ 不許容                                          |
| ------------------------------------------------------------ | ------------------------------------------------------ | -------------------------------------------------- |
| SKILL.md (テキスト規約) のみのタスク                          | テストタスク省略可 (SKILL.md は本リポジトリでテスト対象外) | なし                                                |
| Markdown 文書編集のみのタスク                                | テストタスク省略可                                       | なし                                                |
| Python スクリプト追加・修正                                  | 単体テスト必須                                          | テストタスクなし                                    |
| 既存テストで網羅済みの内部リファクタ                          | 「既存テストで網羅」と明示するメモを持つ                 | 何も書かずテストタスクを省略                        |

## 暗黙依存検出規範 [MANDATORY] (Appendix A.14 由来)

### 規範

1. タスクの `depends_on` フィールドに、実装上必要な先行タスクを必ず列挙する
2. 「フロントエンド実装」が「API 定義」に依存する等の典型的な暗黙依存パターンを見落とさない
3. 依存関係マップは作業メモとして使ってよいが、計画書本体に図として残さない (本文書「依存関係管理」の既存規範)

### 典型的な暗黙依存パターン

| 依存元タスク                       | 依存先タスク                                       | 重大度 (未記載時) |
| ---------------------------------- | -------------------------------------------------- | ----------------- |
| フロントエンド画面実装             | API スキーマ確定 / API 実装                        | 🔴 critical       |
| 統合テスト                         | 全機能実装完了                                     | 🔴 critical       |
| マイグレーション実行               | スキーマ定義タスク                                 | 🔴 critical       |
| デプロイ                           | ビルド / テスト / 環境変数設定                     | 🔴 critical       |
| 画面実装                           | デザイントークン定義 / デザインシステム整備        | 🟡 major          |
| 通知機能                           | 通知チャネル設定 / 認証情報整備                    | 🟡 major          |

## トレーサビリティ規範 [MANDATORY] (Appendix A.11 由来)

### 規範

1. 計画書の `requirements_traceability` に全要件 ID を載せ、対応タスク ID を必ず付与する
2. 対応タスクが無い要件 ID には除外理由を明示する (除外理由欄)
3. タスクの内容が要件・設計と矛盾しないこと

### グレーゾーン許容範囲

| 論点                                                         | ✅ 許容                                              | ❌ 不許容                                          |
| ------------------------------------------------------------ | ---------------------------------------------------- | -------------------------------------------------- |
| 要件 ID に対応タスクが無い                                  | 除外理由欄に「次フェーズで実装」「TBD-xxx と一体化」等を明示 | 除外理由なし                                        |
| 1 タスクが複数要件 ID をカバー                              | 各要件 ID への対応関係を明示 (主従関係を記載)         | 「複数要件をまとめて実装」のみの記述                |

## 重大度カタログ [MANDATORY]

`plugins/forge/docs/plan_principles_spec.md` 内の各規範を、違反時の重大度に対応付ける。

### 構成ルール (本文書既存)

| 違反パターン                                                       | 違反時の重大度 | 理由                                                              |
| ------------------------------------------------------------------ | -------------- | ----------------------------------------------------------------- |
| `requirements_traceability` / `design_traceability` / `tasks` / `revision_history` 以外のキー追加 | 🟡 major       | 計画書の目的逸脱 (タスク実行に不要な情報の追加)                     |
| YAML 形式以外で記述                                                | 🔴 critical    | ツールチェーンが解釈不能                                          |
| ファイル名が `{feature}_plan.yaml` 規約から逸脱                     | 🟡 major       | 自動解決経路が機能しない                                          |

### タスクの粒度 (本文書既存)

| 違反パターン                                                       | 違反時の重大度 | 理由                                                              |
| ------------------------------------------------------------------ | -------------- | ----------------------------------------------------------------- |
| タスク内容が 5〜10 項目を大幅に超える (例: 20 項目)                  | 🟡 major       | Agent 1 実行で完結しない粒度                                       |
| タスク完了時にビルド・テストが成功しない                            | 🔴 critical    | 計画書の中核要件違反                                              |
| 1 タスクが多数ファイル (4 ファイル以上) を操作                      | 🟢 minor       | 粒度が粗いがケースバイケース。グループ化で対応可                    |

### 「やるべき内容」記載原則 (本文書既存)

| 違反パターン                                                       | 違反時の重大度 | 理由                                                              |
| ------------------------------------------------------------------ | -------------- | ----------------------------------------------------------------- |
| 設計書から実装詳細 (プロパティ名 / 型 / メソッドシグネチャ) を転記  | 🟡 major       | 二重管理化・不整合源                                              |
| 設計書セクションを特定できないレベルの曖昧記述                      | 🟡 major       | 実装者が必要情報に辿り着けない                                    |

### 「必読」列 (本文書既存)

| 違反パターン                                                       | 違反時の重大度 | 理由                                                              |
| ------------------------------------------------------------------ | -------------- | ----------------------------------------------------------------- |
| 必読パスがファイル ID のみ (例: `DES-001`)                          | 🔴 critical    | 自動解決不能                                                      |
| 必読パスが glob (例: `specs/*/design/*.md`)                         | 🔴 critical    | 自動解決不能                                                      |
| 必読パスが曖昧名 (例: `foo_design`)                                 | 🔴 critical    | 自動解決不能                                                      |
| 必読列が空 (`-` も無い)                                            | 🟡 major       | 「必読なし」の明示が必要                                          |

### タスクグループ (本文書既存)

| 違反パターン                                                       | 違反時の重大度 | 理由                                                              |
| ------------------------------------------------------------------ | -------------- | ----------------------------------------------------------------- |
| グループ途中タスクで `build_check: per_task` を指定                  | 🟡 major       | ビルド失敗を想定したグループ化の意義を無効化                       |
| グループサイズが 10 タスクを大幅超過 (例: 20 タスク)                | 🟡 major       | 段階的分割の検討漏れ                                              |
| グループ最終タスクに `build_check: on_group_complete` 未指定        | 🟡 major       | グループ完了時のビルド確認漏れ                                    |

### 依存関係管理 (本文書既存 + 本 addendum 追加)

| 違反パターン                                                       | 違反時の重大度 | 理由                                                              |
| ------------------------------------------------------------------ | -------------- | ----------------------------------------------------------------- |
| 循環依存 (タスク A → B → C → A)                                    | 🔴 critical    | 実行順序決定不能                                                  |
| 依存される側より依存する側が先に実装される順序                      | 🔴 critical    | 実装不能                                                          |
| 暗黙依存の未記載 (Appendix A.14 由来。上記「典型パターン」表参照)   | 🔴 critical 〜 🟡 major | パターンごとに重大度差あり (上記表)                                |

### 受け入れ基準・テストタスク (本 addendum で追加)

| 違反パターン                                                       | 違反時の重大度 |
| ------------------------------------------------------------------ | -------------- |
| 主要機能タスクに受け入れ基準フィールドなし or 空                    | 🔴 critical    |
| 受け入れ基準が主観表現 (「適切に動作」「問題なく」等)               | 🟡 major       |
| 主要機能タスクに対応するテストタスクが計画書に存在しない             | 🔴 critical    |
| テストタスクが Markdown 文書編集等の例外条件に該当しないのに省略     | 🔴 critical    |

### トレーサビリティ (本 addendum で追加)

| 違反パターン                                                       | 違反時の重大度 |
| ------------------------------------------------------------------ | -------------- |
| `requirements_traceability` に未掲載の要件 ID がある                | 🔴 critical    |
| 対応タスクなしの要件 ID に除外理由が記載されていない                 | 🟡 major       |
| タスクが対応要件と矛盾する内容                                      | 🔴 critical    |
