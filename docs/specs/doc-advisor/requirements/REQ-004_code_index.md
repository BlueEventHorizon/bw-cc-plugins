# REQ-004: コードインデックス要件定義書

## 概要

ソースコードファイルから構造情報（import、export、ドキュメントコメント等）を抽出し、コードインデックスを生成する。AI エージェントが大規模プロジェクトでタスクに関連するソースファイルを高速に特定できるようにする。

## 背景・動機

- 現行 doc-advisor は文書（rules / specs）の検索に特化しており、ソースコードの検索機能がない
- ソースコードに Embedding を適用するアプローチは、セマンティックギャップ（自然言語クエリ vs コード）により精度が不十分（0.2.1 で文書 Embedding すら精度不足だった実績）
- AI の推論力を活かすには、機械的に抽出した構造情報を AI に渡して判断させる方が効果的
- 言語に特化することで、ドキュメントコメント（`///` 等）やプロトコル準拠など豊富な情報を抽出できる
- ファイル数が数千規模のプロジェクトでは、全ファイルのメタデータを AI に読ませるのはコスト過大。機械的な絞り込み（Stage 1）+ AI 推論（Stage 2）の2段階方式が必要

## 前提条件

| ID     | 要件 |
|--------|------|
| PRE-01 | プロジェクトルートが特定できること（`.git` ディレクトリまたは `CLAUDE_PROJECT_DIR` 環境変数） |
| PRE-02 | 対象はテキストベースのソースコードファイルのみ。バイナリファイルは対象外 |
| PRE-03 | doc-advisor プラグインとして既存のインフラ（`toc_utils.py` の共通関数）を再利用する |
| PRE-04 | 対応言語の MCP サーバーがプロジェクトに登録されていること（Swift の場合は Swift-Selena MCP） |

## スコープ

### 対象

- ソースコードファイル（初期実装は Swift。言語プラグイン構造により他言語を追加可能）
- テストファイル（検索対象に含める。影響範囲の特定に必要）

### 対象外

- 文書ファイル（rules / specs）— 既存の ToC で対応済み
- バイナリファイル、画像、ビルド成果物

## 関連システム

### Swift-Selena MCP サーバー

SwiftSyntax を用いた AST ベースの Swift コード解析 MCP サーバー。利用可能な場合、正規表現より高精度な抽出ができる。

| 機能 | MCP ツール名 | 提供する情報 |
|------|-------------|-------------|
| import 解析 | `analyze_imports` | モジュール依存関係（キャッシュ対応） |
| シンボル抽出 | `list_symbols` | Class / Struct / Enum / Function 等（行番号付き） |
| プロトコル準拠 | `list_protocol_conformances` | 継承関係・Protocol 準拠 |
| Extension 解析 | `list_extensions` | Extension の一覧と対象型 |
| シンボル検索 | `find_symbol_definition` | プロジェクト全体でのシンボル定義箇所 |

**利用条件**: Swift-Selena MCP がプロジェクトに登録されていること（必須前提条件）。未登録の場合はエラーとし、登録方法を案内する。

## 機能要件

### FR-01: インデックス対象の自動検出

| ID      | 要件 |
|---------|------|
| FR-01-1 | システムはプロジェクトルート配下のソースコードファイルを再帰的にスキャンする |
| FR-01-2 | システムは言語を拡張子から判定し、対応する言語プラグインにディスパッチする |
| FR-01-3 | システムは除外ディレクトリ（`.git`, `.build`, `DerivedData`, `Pods`, `node_modules`, `__pycache__`, `dist`, `build`, `vendor` 等）を自動的にスキップする |
| FR-01-4 | 除外ディレクトリのリストはコードデフォルトで定義し、設定ファイルによるカスタマイズは初期版では不要とする |
| FR-01-5 | `.doc_structure.yaml` は不要とする（ソースコードのルートディレクトリは自動検出で十分） |
| FR-01-6 | 言語プラグインが存在しない拡張子のファイルはパス・行数のみをインデックスに含める |

### FR-02: 言語プラグイン構造

| ID      | 要件 |
|---------|------|
| FR-02-1 | メタデータ抽出ロジックは言語別のプラグインモジュールとして分離する |
| FR-02-2 | 各言語プラグインは以下のインターフェースを実装する: 対応拡張子の宣言、MCP サーバーを介したメタデータ抽出 |
| FR-02-3 | core モジュールは拡張子から言語プラグインを自動選択する。未対応拡張子のファイルはパス・行数のみをインデックスに含める |
| FR-02-4 | 新しい言語の追加は、言語プラグインモジュールの追加のみで完結する。core モジュールの変更は不要 |

### FR-03: メタデータ抽出

| ID      | 要件 |
|---------|------|
| FR-03-1 | 全言語共通で以下を抽出する: ファイルパス（プロジェクトルート相対）、言語名、行数 |
| FR-03-2 | 言語プラグインが MCP サーバー経由で提供する情報: import 一覧、export 一覧（ドキュメントコメント付き）、プロトコル準拠、セクション構造 |
| FR-03-3 | MCP サーバー未登録の場合はエラーとし、登録方法を案内する |

### FR-04: Swift 言語プラグイン（初期実装）

| ID      | 要件 |
|---------|------|
| FR-04-1 | 対応拡張子: `.swift` |
| FR-04-2 | 必須前提: Swift-Selena MCP がプロジェクトに登録されていること |
| FR-04-3 | `analyze_imports` で import 依存関係を取得する |
| FR-04-4 | `list_symbols` で全シンボル（Class / Struct / Enum / Protocol / Function 等）を行番号付きで取得する |
| FR-04-5 | `list_protocol_conformances` でプロトコル準拠・継承関係を取得する |
| FR-04-6 | `list_extensions` で Extension 情報を取得する |
| FR-04-7 | AST ベースのため、ネストされた型定義やジェネリクスも正確に抽出できる |
| FR-04-8 | Swift-Selena の接続失敗・タイムアウト時はエラーとし、再接続を案内する |

### FR-05: import グラフの構築

| ID      | 要件 |
|---------|------|
| FR-05-1 | システムは全ファイルの import 情報から依存関係グラフを構築する |
| FR-05-2 | プロジェクト内モジュール import と外部フレームワーク import を区別する |
| FR-05-3 | 指定ファイルの「依存先」（このファイルが import しているもの）と「依存元」（このファイルを import しているもの）を取得できる |
| FR-05-4 | N ホップの連鎖探索（影響範囲の展開）をサポートする |

### FR-06: 2段階検索

| ID      | 要件 |
|---------|------|
| FR-06-1 | Stage 1（機械的絞り込み）: クエリからキーワードを抽出し、export 名・ドキュメントコメント・パス名・import 名に対して文字列マッチングを行う |
| FR-06-2 | Stage 1 は AI 呼び出しなしで完結し、ミリ秒オーダーで結果を返す |
| FR-06-3 | Stage 2（AI 推論）: Stage 1 の候補のメタデータを AI に渡し、最終的な関連ファイルを判断する |
| FR-06-4 | Stage 2 で AI が読むメタデータ量は、プロジェクト規模に関わらず概ね 30KB 以下とする |
| FR-06-5 | 影響範囲の検索: 指定ファイルの import グラフを N ホップ展開し、影響を受けるファイルを列挙する |

### FR-07: インデックスの差分更新

| ID      | 要件 |
|---------|------|
| FR-07-1 | システムは SHA-256 ハッシュでファイル変更を検出する（既存 `toc_utils.py` の `calculate_file_hash` を再利用） |
| FR-07-2 | 差分更新モード（デフォルト）: 追加・変更・削除されたファイルのみ処理する |
| FR-07-3 | 全体再構築モード（`--full`）: 全ファイルを再スキャンする |
| FR-07-4 | 差分更新・全体再構築は数秒で完了する |

### FR-08: インデックスの永続化

| ID      | 要件 |
|---------|------|
| FR-08-1 | インデックスは JSON 形式で `.claude/doc-advisor/code/code_index.json` に保存する |
| FR-08-2 | チェックサムファイルは `.claude/doc-advisor/code/.code_checksums.yaml` に保存する |
| FR-08-3 | インデックスファイルはスキーマバージョンを含む |

### FR-09: CLI インターフェース

| ID      | 要件 |
|---------|------|
| FR-09-1 | インデックス生成: `python3 build_code_index.py --diff | --mcp-data | --check`。`--full`（全ファイル再構築）はスキルレベルのオプションであり、`build_code_index.py` の CLI には含めない |
| FR-09-2 | キーワード検索: `python3 search_code.py --query "検索文字列"` |
| FR-09-3 | 影響範囲検索: `python3 search_code.py --affected-by <ファイルパス> [--hops N]` |
| FR-09-4 | 鮮度チェック: `python3 build_code_index.py --check` |
| FR-09-5 | 全コマンドの出力は JSON 形式（`{"status": "ok", ...}` / `{"status": "error", ...}`） |

## 非機能要件

### NFR-01: パフォーマンス

| ID       | 要件 |
|----------|------|
| NFR-01-1 | 全体再構築（`--full`）は 2000 ファイルのプロジェクトで妥当な時間内に完了する（MCP サーバーの応答速度に依存） |
| NFR-01-2 | 差分更新は変更ファイル数に比例する時間で完了する |
| NFR-01-3 | キーワード検索（Stage 1）は 2000 ファイルのインデックスに対して 100ms 以内に完了する |

### NFR-02: 依存関係

| ID       | 要件 |
|----------|------|
| NFR-02-1 | core モジュールは Python 標準ライブラリのみを使用する（pip 依存なし） |
| NFR-02-2 | 言語プラグインは対応する MCP サーバーを必須前提とする |

### NFR-03: 拡張性

| ID       | 要件 |
|----------|------|
| NFR-03-1 | 新しい言語の追加は、言語プラグインモジュールの追加のみで対応できる。core モジュールの変更は不要 |
| NFR-03-2 | 新しい言語は、対応する MCP サーバーと言語プラグインモジュールのペアで追加する |

## 用語定義

| 用語 | 定義 |
|------|------|
| コードインデックス | ソースコードファイルの構造情報を格納した検索用インデックス |
| 言語プラグイン | 言語固有の抽出ロジックを実装するモジュール。対応する MCP サーバーと組で動作する。`lang/` ディレクトリに配置 |
| ドキュメントコメント | ソースコード中の構造化されたコメント（Swift: `///`, `/** */`）。開発者が記述した自然言語の説明 |
| Stage 1 | 機械的な絞り込みフェーズ。キーワードマッチ + import グラフ展開。ミリ秒で完了 |
| Stage 2 | AI 推論フェーズ。Stage 1 の候補のメタデータを AI に渡し、最終的な関連ファイルを判断 |
| import グラフ | プロジェクト内ファイル間の依存関係グラフ。import 文から構築 |
| export | ファイルが外部に提供するシンボル。ドキュメントコメント、アクセスレベル、プロトコル準拠を含む |
| セクション | ファイル内の論理的な区切り（Swift の `// MARK: -` 等） |
