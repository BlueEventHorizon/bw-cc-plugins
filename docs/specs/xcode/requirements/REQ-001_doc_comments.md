# REQ-001: Swift ドキュメントコメント生成 要件定義書

## 概要

Swift ソースファイルに `///` ドキュメントコメント（DocC 形式）を自動生成・追加するスキル。AI がコードを読んで理解し、関数・クラス・プロトコル等の宣言に対して構造化されたドキュメントコメントを生成する。

## 背景・動機

- Swift の `///` ドキュメントコメントは Xcode の Quick Help に表示され、開発者の生産性を向上させる
- コードインデックス（doc-advisor REQ-004）が `///` コメントを抽出するため、コメントが充実しているほどコード検索の精度が上がる
- 新規開発時やリファクタリング時にドキュメントコメントを書く手間を削減する
- ファイルヘッダー（旧 create-code-headers）ではなく、個々の宣言に対するコメントを対象とする。ファイルヘッダーは AI 検索専用の情報を埋め込むものだったが、`///` コメントは開発者・AI 双方に有用な標準的なドキュメント手法である

## 前提条件

| ID     | 要件 |
|--------|------|
| PRE-01 | 対象は Swift ソースファイル（`.swift`）のみ |
| PRE-02 | xcode プラグインのスキルとして提供する |

## スコープ

### 対象

- `public` / `open` 修飾子を持つ宣言（外部に公開される API）
- `internal` 宣言のうち、プロトコル定義・クラス定義・構造体定義（型の目的の理解に重要）
- 既存の `///` コメントがない宣言（上書きしない）

### 対象外

- `private` / `fileprivate` 宣言（内部実装の詳細）
- 既に `///` コメントが存在する宣言（既存コメントを尊重）
- テストファイル（`*Test.swift`, `*Tests.swift`）
- モックファイル（`Mock*.swift`）
- プロパティの `///` コメント（自明なケースが多い。ただし `--all` オプションで含めることは可能）

## 機能要件

### FR-01: コメント生成対象の検出

| ID      | 要件 |
|---------|------|
| FR-01-1 | スキルは対象ファイル内の `///` コメントがない宣言を検出する |
| FR-01-2 | 検出対象: `class`, `struct`, `enum`, `protocol`, `func`, `init`, `subscript`, `typealias` |
| FR-01-3 | 既に `///` または `/** */` コメントが存在する宣言はスキップする |
| FR-01-4 | `private` / `fileprivate` 宣言はデフォルトでスキップする |

### FR-02: DocC 形式のコメント生成

| ID      | 要件 |
|---------|------|
| FR-02-1 | 生成するコメントは Swift DocC 形式（`///` 行コメント）とする |
| FR-02-2 | 関数・メソッドには以下を含む: 概要（1行）、`- Parameter`: 各パラメータの説明、`- Returns`: 戻り値の説明（`Void` 以外）、`- Throws`: 例外の説明（`throws` の場合） |
| FR-02-3 | 型宣言（class / struct / enum / protocol）には概要（1-2行）を生成する |
| FR-02-4 | AI はコードの実装内容を読んで理解した上でコメントを生成する。シグネチャの機械的な言い換えではなく、**なぜ・何のために** を記述する |
| FR-02-5 | テストファイルが存在する場合はそれも参考にし、使用例や意図を反映する |

### FR-03: 動作モード

| ID      | 要件 |
|---------|------|
| FR-03-1 | デフォルト: 未コミットの変更ファイルを対象とする（`git status` で検出） |
| FR-03-2 | `--full`: 全 Swift ファイルを対象とする（初回セットアップ用） |
| FR-03-3 | `--force`: 既存の `///` コメントも上書き再生成する |
| FR-03-4 | ファイル/パターン指定: 特定ファイルまたはグロブパターンで対象を指定できる |

### FR-04: 安全性

| ID      | 要件 |
|---------|------|
| FR-04-1 | 既存の `///` コメントはデフォルトで変更・削除しない |
| FR-04-2 | 既存の実装コメント（`//`）は変更・削除しない |
| FR-04-3 | コード本体（コメント以外）は変更しない |
| FR-04-4 | ファイル先頭の Copyright ヘッダーは変更しない |

### FR-05: 並列処理

| ID      | 要件 |
|---------|------|
| FR-05-1 | 複数ファイルを並列で処理する（Agent 並列起動） |
| FR-05-2 | 各ファイルは独立した Agent が処理する。ファイル間の依存はない |
| FR-05-3 | エラー発生時は該当ファイルをスキップし、他ファイルの処理を継続する |

### FR-06: 自動トリガー

| ID      | 要件 |
|---------|------|
| FR-06-1 | Swift ファイルを新規作成した直後、AI がスキルを自動的に呼び出す |
| FR-06-2 | 自動トリガー時の対象は作成・修正したファイルのみ。Agent 起動は不要で直接処理する |
| FR-06-3 | 自動トリガーは除外条件（テストファイル、モックファイル等）を適用する |

### FR-07: CLI インターフェース

| ID      | 要件 |
|---------|------|
| FR-07-1 | `/xcode:doc-comments` — 未コミットファイル対象 |
| FR-07-2 | `/xcode:doc-comments --full` — 全ファイル対象 |
| FR-07-3 | `/xcode:doc-comments --force` — 既存コメント上書き |
| FR-07-4 | `/xcode:doc-comments Sources/Auth/*.swift` — パターン指定 |

### FR-08: 完了報告

| ID      | 要件 |
|---------|------|
| FR-08-1 | 処理完了時に以下を報告する: 対象ファイル数、コメント生成数、スキップ数（既存コメントあり）、エラー数 |

## 非機能要件

### NFR-01: コメント品質

| ID       | 要件 |
|----------|------|
| NFR-01-1 | 生成されるコメントは DocC 形式に準拠し、Xcode の Quick Help で正しく表示される |
| NFR-01-2 | コメントは簡潔かつ十分な情報を含む。冗長な表現やシグネチャの単純な言い換えは避ける |
| NFR-01-3 | 日本語と英語の両方に対応する。プロジェクトの既存コメントの言語に合わせる |

### NFR-02: パフォーマンス

| ID       | 要件 |
|----------|------|
| NFR-02-1 | 1ファイルあたりの処理時間は AI モデルの応答速度に依存する。並列処理で全体の所要時間を短縮する |
| NFR-02-2 | 並列数は 5-10 Agent とする（システム負荷に応じて調整） |

### NFR-03: 既存スキルとの関係

| ID       | 要件 |
|----------|------|
| NFR-03-1 | 旧 create-code-headers スキル（ファイルヘッダー生成）とは独立したスキルとする |
| NFR-03-2 | create-code-headers の `[Code Header Format]` マーカーとは無関係。既存のヘッダーがあっても影響しない |

## 用語定義

| 用語 | 定義 |
|------|------|
| ドキュメントコメント | Swift の `///` 行コメントまたは `/** */` ブロックコメント。DocC が解析する |
| DocC | Apple のドキュメント生成ツール。`///` コメントから API リファレンスを生成する |
| Quick Help | Xcode で Option+クリックで表示されるインラインドキュメント。`///` コメントが表示される |
| 自動トリガー | ファイル作成・修正時に AI が自動的にスキルを呼び出す機能 |
