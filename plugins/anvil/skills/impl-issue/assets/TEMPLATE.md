# Issue 更新テンプレート

以下のテンプレートを使用して Issue 本文を更新する。
実装スコープ（UI 実装のみ、データ基盤整備のみ等）に応じて不要なセクションを削除すること。

`<owner>/<repo>` は Phase 0 で解決した値、レイヤー名・ディレクトリパス・チェックコマンドはプロジェクト規約に従う（`/forge:query-db-rules` で「ディレクトリ構造」「チェックコマンド」を検索）。

## テンプレート

```markdown
## 実装計画

### 関連ドキュメント

**仕様書**

| 種類 | ドキュメント |
| ---- | ------------ |
| API 設計書 | [<API 名>](URL) |
| 画面設計書 | [<画面名>](URL) |

**参考 PR（実装パターン）**

| PR | 参考ポイント |
| -- | ------------ |
| [#<番号> <タイトル>](URL) | <参考にした実装パターン> |

### 実装スコープ

| レイヤー | 実装内容 | ファイルパス |
| -------- | -------- | ------------ |
| <レイヤー名> | <クラス名 / モジュール名> | `<path>` |

### スコープ外

| 項目 | 理由 | 担当 |
| ---- | ---- | ---- |

## TODO

### 着手前確認

- [ ] 仕様書の確認完了
- [ ] 実装ルールの確認完了
- [ ] 参考 PR のパターン確認完了

### 実装する

- [ ] <実装項目 1>
- [ ] <実装項目 2>

### テストを書く

プロジェクトのテスト方針に従い、**実装した層のうちテスト可能なロジックがある対象のみ** テストを追加する（実装していない層・該当ロジックが無い項目はチェック項目ごと削除する）。

- [ ] <対象 1>: <テスト内容>
- [ ] <対象 2>: <テスト内容>

### 確認・レビュー準備を行う

- [ ] コード生成コマンドを実行する
- [ ] 静的解析を実行する
- [ ] フォーマットチェックを実行する
- [ ] テストを実行する

### 完了する

- [ ] PR を作成する
- [ ] レビューに対応する
```

## gh コマンド例

特殊文字（バッククォート・`$` 等）に強い `--body-file` 経由で更新する。詳細手順とエンコード規約は [`references/phase-09-issue-update.md`](../references/phase-09-issue-update.md) を参照。

```bash
tee /tmp/issue_body.md <<'BODY'
<上記テンプレートの内容>
BODY

gh issue edit <issue番号> --repo <owner>/<repo> --body-file /tmp/issue_body.md
```
