# 実装設計書 作成ルール

デザイン仕様書と既存ソースコードを照合し、実装設計書（How: どう作るか）を作成するためのルール。

## ドキュメントの役割

| ドキュメント   | 目的              | 含める内容                                     | 含めない内容 |
| -------------- | ----------------- | ---------------------------------------------- | ------------ |
| **実装設計書** | どう作るか（How） | 使用コンポーネント、デザイントークン、状態管理 | ソースコード |

## 重要な原則

### 1. 設計書だけで誰でも実装できる

- 設計書を読めば、ソースコードがなくても実装方法が分かる
- 曖昧な記述を避け、具体的に記載する

### 2. 実装設計書にソースコードは書かない（原則）

- **例外**: 使用方法を示す必要がある場合の短いコード例は許容
  - デザイントークンの参照方法（プロジェクト固有の呼び出し例）
  - これらは「設計上の決定事項」であり「実装コード」ではない
- 上記以外で特殊な事情がある場合は、人間に問い合わせて判断を仰ぐ

### 3. 既存実装の共通パターンを確認する

- 設計書を作成する前に、類似機能・画面の既存実装を必ず確認する（Phase 5 / Phase 6）
- 共通パターン（画面構成、状態管理、コンポーネントの使い方）に従う
- デザインと既存実装で差分がある場合は、設計書に明記して確認を促す
- 参照すべき既存コードを設計書に記載する

### 4. 共通化の判断と設計での実施

- 既存コード調査で**共通化できるコード**を発見した場合は、設計フェーズで共通化方針を決定する
- 共通化の対象（コンポーネント / ユーティリティ / マッパー等）を明記する
- 既存の共通実装がある場合は**新規作成せず再利用**する
- 新規共通化が必要な場合は、理由・影響範囲・適用箇所を設計書に記載する

## 作成前に確認すべきもの

### 既存コンポーネント・ユーティリティの確認

swift-selena MCP または Glob / Grep でプロジェクトの共通コンポーネント配置を検索する。
共通コンポーネントの配置先は CLAUDE.md またはプロジェクト規約から確認する。

### デザイントークン・テーマの確認

プロジェクトのデザインシステム定義（カラー、タイポグラフィ、スペーシング等）を確認する。
ハードコードの値は使用せず、プロジェクトのデザイントークンを使用する。

### ドメインモデル・マッパーの確認

- 画面で表示するデータに対応するドメインモデルが存在するか
- ドメインモデルから UI モデルへの変換メソッドが存在するか

## よくある失敗パターン

1. **既存コンポーネントの検索が不十分** — 1つの検索パターンだけで諦めない
2. **実装設計書にソースコードを書いてしまう** — 特殊な事情がある場合は人間に確認
3. **デザイントークンをハードコード** — プロジェクトのテーマ定義を使用する
4. **既存のモデルやユーティリティを確認せずに新規作成** — Phase 6 で調査済みの実装を再確認
5. **類似実装を確認しない** — 同じ種類の機能・画面の既存実装を確認し、共通パターンに従う
