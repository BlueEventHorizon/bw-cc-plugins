# UI 実装ルール

## Design Implementation Policy [MANDATORY]

- **デザイン仕様書 = 構造の正**（順序・サイズ・色・padding・font・条件分岐をそのまま転記）
- **Figma = ビジュアル詳細の正**（アイコン形状、デザイン仕様書に無い情報の補完）
- **既存パターンよりデザイン仕様書優先**（仕様書と整合しない既存構造は流用しない）
- **フォントファミリーのみ例外**（システムフォント等、プロジェクト規約に従う）
- **トークン使用 ≠ デザイン準拠** — トークン**名**まで仕様書と照合する（[Typography 照合ルール](phase-11-typography-mapping.md)）

## Typography 照合 [MANDATORY]

**詳細手順・変換ルール**: [phase-11-typography-mapping.md](phase-11-typography-mapping.md)

最低限:

1. デザイン仕様書の全テキストノード（および Typography テーブル）を列挙
2. 各ノードについて `size / weight / line_height / color / 言語` を抜き出す
3. プロジェクトのトークン定義ファイルで `fontSize + fontWeight + lineHeight` が一致するトークンを選ぶ
4. hex → プロジェクトのカラートークンへ変換（タップ動作があればリンク色用トークン）
5. 対応表を実装設計書 or コードコメントに書いてから実装

**禁止**: 「小さいラベルだから〜」「リンクっぽいから〜」等の推測選択。

## デザイン仕様書の逐次参照 [MANDATORY]

実装コードを書く前に、対応するノードを Read し、コメントへ抜粋転記してから書く。

**禁止**: 仕様書未確認での実装 / 既存画面のコピペ / フレームワーク標準値の流用。

## 既存コンポーネント流用時の差分照合 [MANDATORY]

1. デザイン仕様書の `font` / `color` / `size` / `padding` を抜き出す
2. 既存コンポーネントの対応値を Grep で確認
3. 差分があれば: 既存を仕様書に揃える / 派生バリアント追加 / 新規作成 / 例外理由をコメント

「既存流用 = 仕様準拠」ではない。近似色も完全一致させる。

## 共用コンポーネントの変更禁止 [MANDATORY]

2 画面以上から参照されるコンポーネントを **1 画面の typography に合わせて書き換えない**。

**判断基準**: Grep / コードシンボル検索で **2 画面以上**から参照されていたら共用。

| 状況 | 対応 |
| ---- | ---- |
| 共用コンポーネントの typography が仕様書と不一致 | 画面専用コンポーネントを新規作成 |
| 共用コンポーネントで十分 | Typography 対応表で**既存値と仕様書の diff**を明示し、一致を確認して流用 |
| 1 画面だけ特殊 | 画面ローカル配置に画面専用コンポーネントを置く |

## インタラクション同時実装 [MANDATORY]

デザイン仕様書「アクション一覧」「状態バリエーション」は見た目と**同時**に実装。後回し・TODO 禁止。

実装パターンの選択（タップ → 遷移、お気に入りトグル、外部リンク等）はプロジェクトの実装規約に従う（`/forge:query-db-rules` で「UI 実装パターン」を検索）。

## レイアウト抽出ノウハウ [MANDATORY]

デザイン仕様書の数値をコードに転記する際、暗黙の前提を踏み外すとオーバーフロー / 隙間 / 色違いが発生する。以下を必ず守る。

### 寸法・サイズ

- **スクロール領域は固定高さで囲まない**: 子の内在高さに任せる。フォントレンダリングや画像スケール誤差で常に数 px ズレが出るため、固定値は筋悪
- **スクロール可否の前提を確認**: データ依存で画面高さが変動する画面では、コンテンツが画面内に収まる場合でもスクロール可能にしておく方がバグに見えにくい（プロジェクトの UI ガイドラインに従う）
- **下部ナビゲーション等の重なりに注意**: 固定 UI が画面下部に重なる構成の場合、最下部コンテンツが隠れないよう必要な余白を加える
- **コンテナ高は積算で検算**: 固定値が必要な場合は `親.height == 子要素の height 合計 + gap + padding` で必ず検算する
- **行高 (`line_height`) を必ず加算**: テキスト部の実高さは `font.line_height`。`fontSize × 倍率` で勝手に推測しない
- **スケール対象を区別**: 画像幅・カードサイズなどデザイン基準幅前提の絶対サイズ（スケール対象）と、フォントサイズ・gap・padding・border 幅（スケール非対象）を混在させない
- **`width: fill` / `hug`、`height: fill` / `hug`** はプロジェクトのレイアウト規約に従って正しいプリミティブに置き換える

### フォントサイズが異なるテキストの横並び（baseline 揃え）

異なるフォントサイズのテキストを同じ行に並べる場合、**ベースラインで揃える** ことが基本（行ボックス下端で揃えると小さい文字が下にずれる）。
プロジェクトの UI ライブラリでベースライン揃えに対応するプリミティブを使う。折り返しが必要な場合、ベースライン揃えと折り返しが両立しない実装は早めに見抜き、フォールバック方針を設計書に明記する。

### Padding / Gap

- **`padding: { top, right, bottom, left }` を実装ライブラリの引数順に正しくマッピング**（順番を間違えやすい）
- **垂直レイアウトの `gap: N`、水平レイアウトの `gap: N`** を正しい方向のスペーサーに変換
- **左右非対称 padding**（例: 右だけ 0）に注意。`all` / `symmetric` で雑にまとめない

### 色

- **同じ hex でも文脈で対応トークンが変わる**:
  - リンク（タップ動作あり） → リンク色トークン
  - ラベル / 非インタラクティブ → 補助色トークン
  - ブランド強調 → ブランド色トークン
- **インタラクション表で「タップ動作」が定義されているか確認** してから色トークンを選ぶ
- **デザイントークン側の色と仕様書の色の微差** は許容（トークン優先）。ただし完全に違う色（青 vs 赤）なら仕様書 / トークンどちらが正か確認

### 条件分岐

- **デザイン仕様書の備考や仕様文言（例: 「100 文字以上で表示」）は仕様と等価**。コードの条件と 1 対 1 で照合
- **状態テーブル**（「条件 X 時非表示」「条件 Y 時無効」等）はすべて実装する。空ハンドラや TODO で放置しない
- **区切り線・装飾要素**: 仕様書にある区切り線・縦線を、隙間（gap）だけで代用しない。専用プリミティブを使う

### 検算チェック

実装後、以下を仕様書と突き合わせる:

- 全体高さ = 各子要素の高さ + gap + padding の合計
- スケール対象の値は実機幅 = 基準幅以外で破綻しないか
- 色トークンが意図通り（リンク → リンク色、非リンク → 補助色）になっているか

## 必須参照ドキュメント

実装を開始する前に、以下のドキュメントを**必ず読み込む**（取得方法は `/forge:query-db-rules` を使用）:

| ドキュメント | 内容 | 必須度 | 取得方法 |
| ------------ | ---- | ------ | -------- |
| UI 実装パターン | 画面構成、コンポーネント、状態管理、シート・モーダル等のパターン | **必須** | `/forge:query-db-rules` で「UI 実装パターン」を検索 |
| コーディング規約 | 命名規則、ディレクトリ構造、Figma 値取得ルール | **必須** | `/forge:query-db-rules` で「コーディング規約」を検索 |
| コンポーネントカタログ | 既存コンポーネント一覧（存在する場合） | **必須** | プロジェクト規約を確認 |
| 特殊 UI 設計ガイド | フィルターシート・カレンダー等の専用ガイド | 該当時 | `/forge:query-db-rules` で対象 UI 名を検索 |

## 必須実装ルール

| ルール | 説明 |
| ------ | ---- |
| 既存コンポーネント優先 | 新規作成前に必ず既存コンポーネントを検索 |
| デザイントークン使用 | 色・フォント・スペーシングのハードコード禁止 |
| アセット管理の遵守 | アイコン・画像はプロジェクト規約のアセット参照方法を使用（標準セットや直接参照を避ける） |
| UI 文字列の管理 | UI 文字列のハードコード禁止。プロジェクトの i18n / 文字列管理方法を使用 |
| Figma / 仕様書の値を使用 | 仕様書にない値は空想で実装しない |
| 共通化方針の遵守 | 設計書に記載された共通化方針に従い、既存共通実装を優先 |

## 状態管理の選択基準

| ケース | 考慮事項 |
| ------ | -------- |
| シート / ダイアログ内の一時的な選択状態 | 画面ローカルの状態として管理 |
| フォーム入力の一時状態 | 画面ローカルの状態として管理 |
| API から取得したデータ | 永続化・共有が必要な状態として管理 |
| 画面間で共有される状態 | グローバル状態として管理 |

具体的な状態管理ライブラリ・パターンはプロジェクト規約に従う（`/forge:query-db-rules` で「状態管理」を検索）。

## 設計書にない値の取得 [MANDATORY]

**設計書に記載のない値は空想で実装しない。**

設計書に値がない場合:

1. Figma MCP で `get_design_context` を実行
2. **`get_screenshot` で写っているか確認**（node ツリーにあっても `visible=false` の未使用パーツは実装しない）
3. `get_variable_defs` で変数定義を取得
4. `get_metadata` でノードのサイズ情報を取得

**禁止事項**:

- 「だいたいこのくらいだろう」と推測して実装する
- 他の類似コンポーネントから値をコピーして流用する
- よくある値を根拠なく使用する

## 実装ファイルの配置

プロジェクトのディレクトリ規約に従う（`/forge:query-db-rules` で「ディレクトリ構造」を検索）。

## Figma MCP エラー時の対応

1. **必ずユーザーに報告する** — 何が起きたかを明示する
2. **不確かな情報で進めない** — MCP が失敗した場合、推測で実装しない
3. **代替手段を試す** — REST API での取得を試みる
4. **検証が不十分なら中断** — 「Figma 情報を取得できませんでした」と報告する

## よくある失敗パターン

- 色・フォントをハードコード → デザイントークン使用
- 標準アイコンセット直接使用 → プロジェクト規約のアセット参照を使用
- UI 文字列をハードコード → プロジェクトの i18n / 文字列管理方法を使用
- 既存コンポーネントを探さず新規作成 → コンポーネントカタログと Grep / シンボル検索で検索
- 設計書にない値を空想で実装 → Figma MCP で取得
- 状態バリエーション実装漏れ → Figma で全状態確認
- **仕様書を一度しか見ない** → コンポーネント単位で逐次参照・コメント転記
- **既存流用時に値差分未確認** → Grep で `font` / `color` / `size` / `padding` を確認
- **既存画面パターンに引きずられて仕様書構造を曲げる** → 仕様書を優先
- **実装難易度で仕様書構造を省略** → 妥協時は `AskUserQuestion`
- **lint 回避で仕様書を妥協** → 必要なら `// ignore` 等で設計を貫く
- **トークン名を推測で選んだ** → [Typography 照合ルール](phase-11-typography-mapping.md) で対応表を作る
- **共用コンポーネントを 1 画面用に変更** → revert、画面専用コンポーネントを作る
- **アクション一覧を後回し** → 見た目と同時にハンドラ・遷移を実装
- **区切り線・装飾要素の欠落** → 類似コンポーネントを Grep してパターン流用

## 実装後セルフチェック [MANDATORY]

Phase 12 へ進む前に確認。不合格があれば修正してから次へ。
妥協する場合は `AskUserQuestion` で確認を取る。

- [ ] 子要素の順序がデザイン仕様書のレイアウト定義と一致
- [ ] **Typography 対応表**の行数 = デザイン仕様書のテキストノード数（[phase-11-typography-mapping.md](phase-11-typography-mapping.md)）
- [ ] font / color / size / padding / gap が仕様書と一致（推測トークン禁止。トークン定義ファイルで検証済み）
- [ ] **アクション一覧**の全項目が実装済み（空ハンドラ / TODO なし）
- [ ] **共用コンポーネント**を 1 画面の typography に書き換えていない
- [ ] **コンテナ高は内部要素の積算で検算**（画像高 + gap + テキスト行高、スケール対象 / 非対象を区別）
- [ ] 既存流用箇所の値差分を Grep 確認済み
- [ ] 仕様書要求の構造を採用（実装難易度を理由に省略していない）
- [ ] lint 回避で仕様妥協していない
- [ ] 条件分岐は仕様書文言通り（例: 「100 文字以上」→ 数値で照合）
- [ ] スクロール挙動が実機で仕様書通り
