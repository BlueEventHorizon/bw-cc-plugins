# Typography → デザイントークン照合ルール

デザイン仕様書の `font: { size, weight, color, line_height }` を **推測せず** プロジェクトのデザイントークン（テキストスタイル / カラー）に機械的に変換するためのルール。

> **原則**: トークン名を「だいたい」選ぶと font / color が大量にズレる。
> **実装コードを 1 行も書く前に、Typography 対応表を必ず作る。**

## 手順 [MANDATORY]

### Step 1: 全テキストノードを抽出

デザイン仕様書の YAML / レイアウト定義および Typography テーブルを Grep し、**すべてのテキストノード**を列挙する。

対象: `type: text` ノード、`content` 付きラベル、Typography テーブルの各行。

### Step 2: Typography 対応表を作成

実装設計書（Phase 8）または実装ファイルの先頭コメントに、**全ノード分**の表を書く。

| ノード名 | size | weight | line_h | color (hex) | 言語 | textStyle トークン | color トークン | 根拠 |
| -------- | ---- | ------ | ------ | ----------- | ---- | ------------------ | -------------- | ---- |

**禁止**: 表の行数 < テキストノード数 で実装開始 / 「似たトークン」で埋める。

### Step 3: トークン定義ファイルで検証

対応表の各行について、プロジェクトのデザイントークン定義ファイル（`/forge:query-db-rules` で「デザイントークン」「テキストスタイル」を検索）を Read し、`fontSize` / `fontWeight` / `lineHeight` がデザイン仕様書の値と一致するか確認する。

### Step 4: コードコメントへ転記してから実装

実装コードのテキスト要素ごとに、対応するノード名 → 採用トークンの対応をコメントで残してから書く。

---

## トークン選択の汎用ルール

### color (hex) → カラートークン

色トークン選択の優先順位：

1. デザイン仕様書「アクション一覧」— タップ動作が定義されていればリンク色用トークン
2. デザイン仕様書 `font.color` の hex → プロジェクトのカラートークン定義と照合
3. Figma 変数（MCP `get_variable_defs`）

**禁止**: hex を見ずに「リンクっぽいから link」「小さいから tertiary」と推測する。

### 言語別テキストスタイル

プロジェクトが言語別にテキストスタイルを分けている場合（日本語用 / 英数字用 等）：

| 内容                  | 使用                               |
| --------------------- | ---------------------------------- |
| 自国語 UI 文言        | 自国語用テキストスタイル           |
| 英数字・単位・ID 等   | 英数字用テキストスタイル           |
| 混在（名前 + 数値等） | 要素ごとに分離して別トークンを適用 |

判別ルールはプロジェクト規約に従う（`/forge:query-db-rules` で「テキストスタイル 言語」を検索）。

### `body` / `label` / `title` 等のスタイル系統の見分け

同じ size / weight でも `line_height` や letter spacing が異なる別系統トークンがある場合、`size + weight` だけで選ばずに **line_height / letter spacing を含めて完全一致するトークン**を選ぶ。

---

## トークンの再発明禁止 [MANDATORY]

> **原則**: Figma のスタイル名・Variable 名は、概ねプロジェクトのトークン名に 1:1 対応する。スタイル名から該当トークンを探さずに、`copyWith(fontSize:, letterSpacing:)` や独自定数を新規定義しないこと。

**値（fontSize / letterSpacing / weight / 色）をコード内で直接書く前に、対応するトークンが既に無いか必ず探す。**

**手順**:

1. Figma のスタイル名 / Variable 名を控える
2. プロジェクトのテキストスタイル / カラー定義ファイルを Grep
3. あればトークンを使い、差分のみ最小に上書きする
4. 無い場合のみ定数化（出典コメント必須）

**禁止**: 既存トークンと同値の magic number を新規定義する / `copyWith` でトークンを実質再現する。

---

## 共用コンポーネントの変更禁止

Typography がデザイン仕様書と不一致でも、共用コンポーネント本体を 1 画面のデザイン仕様書に合わせて書き換えない。
詳細・判断基準・対応表は [phase-11-ui-implementation.md の「共用コンポーネントの変更禁止」](phase-11-ui-implementation.md#共用コンポーネントの変更禁止-mandatory)を参照。

## インタラクション同時実装

デザイン仕様書「アクション一覧」「状態バリエーション」は見た目と同時に実装する。
詳細パターンは [phase-11-ui-implementation.md の「インタラクション同時実装」](phase-11-ui-implementation.md#インタラクション同時実装-mandatory)を参照。

## 類似コンポーネントから取るべきパターン（Grep 必須）

実装前に類似画面・機能のコンポーネントを Read し、デザイン仕様書の対応表と突合する。

**禁止**: 類似コンポーネントをコピペして typography を確認しない。

---

## セルフチェック（Typography 特化）

実装レビュー前に追加確認:

- [ ] Typography 対応表の行数 = デザイン仕様書のテキストノード数
- [ ] 各行をプロジェクトのトークン定義ファイルで検証済み
- [ ] 共用コンポーネントを 1 画面専用 typography に書き換えていない
- [ ] アクション一覧の項目がすべて実装済み（TODO / 空ハンドラなし）
- [ ] コード内コメントにノード → トークンの対応が書いてある

---

## よくある失敗

| 失敗                                       | 正しい対応                                            |
| ------------------------------------------ | ----------------------------------------------------- |
| サイズと weight だけで近いトークンを選んだ | line_height / letter spacing まで含めて完全一致を確認 |
| 共用コンポーネントを 1 画面用に書き換え    | revert、画面専用コンポーネントで対応                  |
| hex を見ず「リンクっぽいから link」        | 仕様書のアクション定義 + hex で正しい色トークンを選ぶ |
| トークン使用 = デザイン準拠と錯覚          | **トークン名まで**対応表で検証                        |
| インタラクション要素を後付け               | 類似コンポーネント Grep → 初回実装に含める            |
