# Phase 6: 既存コード調査ルール

実装に再利用できる既存クラス・コンポーネントを特定する。

## ツール選択

**swift-selena MCP が利用可能な場合**（推奨）:

| 操作                    | 用途                     |
| ----------------------- | ------------------------ |
| `find_symbol`           | クラス名・関数名で検索   |
| `get_symbol_definition` | クラスの詳細な実装を確認 |
| `find_references`       | シンボルの使用箇所を確認 |

**MCP が利用不可の場合**: Glob / Grep で代替する。

## [MANDATORY] 同種の実装パターン調査

実装に入る前に、**同種の機能・画面がどう実装されているか**を徹底的に調査する。
表面的な検索で済ませず、実際にコードを Read して実装パターンを理解すること。

- 同種の機能（一覧取得、詳細表示、フォーム、状態管理等）の既存実装を **3件以上** 特定し Read する
- レイヤー構成（Domain / Infrastructure / DI 等）の実装パターンを把握する
- 命名規則・ファイル配置パターンを確認する

## [MANDATORY] 共通コンポーネントの調査・採用

新規にコンポーネントやロジックを作る前に、**既存の共通実装で代替できないか**を必ず確認する。

- プロジェクトの共通コンポーネント・ユーティリティの配置は CLAUDE.md またはプロジェクト規約から確認する
- 特定した共通実装は実際に Read して以下を把握する：
  - 公開 API（コンストラクタ引数・カスタマイズポイント）
  - 使用例（他の箇所でどう使われているか）
  - 今回の実装で**そのまま使えるか、拡張が必要か、新規作成が必要か**を判断する

## その他の調査ポイント

- 同じドメイン領域の既存モデル・リポジトリ・サービス
- クラス・インターフェースのシグネチャ
- 既存コードと新規実装の依存関係
- 動作確認導線の有無（未実装導線がある場合は一時的なデバッグ導線を実装計画に含める）
