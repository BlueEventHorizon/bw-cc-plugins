# Phase 5: 既存コード調査ルール

Grep / Glob（および利用可能であれば Serena MCP 等のコードシンボル検索ツール）を使い、実装に再利用できる既存クラス・コンポーネント・ユーティリティを特定する。

## [MANDATORY] 新規作成回避の原則（最優先）

このフェーズの本質は「既存資産を徹底的に洗い出すこと」であり、新規作成の検討はその**後**に行う。
Issue 本文に「新規作成しない」と明記されていなくても、本原則は常に最優先で適用する。

### 真意の定義

「新規作成しない」とは:

- **「既存を探さずに楽に新規を作るな」**
- **「整合を取れ。兄弟機能が確立したパターンを破壊するな」**
- **「スコープ境界を口実に、楽な道を選ぶな」**

を意味する。

これは「特定レイヤーにファイルを 1 つも増やすな」という字義通りの禁止ではない。
**徹底調査もせず、整合性を考えず、楽だから新規にする**判断を禁じている。

### 検証チェックリスト（実装計画策定前に必ず実施）

1 つでも未確認の項目を残したまま新規作成を計画してはならない。

- [ ] **同一目的の状態管理・Provider・Notifier** が既に存在しないか
  - 機能名（「並び順」「ソート」「フィルタ」「リスト状態」等）で全体 Grep
  - 既存がある場合は **改修 > 新規** を原則とする
- [ ] **同一目的のドメインモデル / enum / 値オブジェクト** が既に存在しないか
  - プロジェクトのドメインモデル配置場所（`/forge:query-db-rules` で「ディレクトリ構造」を検索）を Glob / Grep
  - 既存があれば、活用するか、整合上廃止するかを設計判断する（既存を残したまま新規型を並べるのは原則禁止）
- [ ] **既存ファイル（仮実装含む）が同じ場所に既に存在し、書き換えれば済まないか**
  - 同名ファイルがある場合は、ほぼ確実に書き換え
- [ ] **兄弟機能（同種パターンを持つ他画面・他機能）が確立した実装パターン**を破壊しないか
  - 直近 PR（同種・同レイヤー）を最低 2 件特定し、Read レベルでパターンを把握
  - 新規実装が兄弟と異なる流派になる場合、それは「整合破壊」の警報
- [ ] **既存 extension / utility / helper** で代替できないか
- [ ] **i18n / debug / route エントリ**で既に登録済みの口がないか
- [ ] **既存 API 連携部が新パターンに整合しているか**
  - 整合していなければ、本 Issue で型整合のための最小修正をする必要があるか判断

### 「楽な道」を見抜く自己検証シグナル

以下に該当する判断が出てきたら、必ず立ち止まり、整合的な代替案を提示する。

- **スコープ境界を口実に、既存と並列で新規を作る案**
  - 例:「既存は別 PR のスコープなので、新規 Provider を併設して既存は触らない」
  - これは原則禁止。既存と新規が並列共存する状態は整合破壊
- **兄弟機能と異なる実装を採用し、その理由が「既存を変えるのが面倒」「影響範囲を狭めたい」だけ**
  - 影響範囲の狭さは、整合性を犠牲にする理由にはならない
- **整合的な書き換えなら不要になる既存資産（enum / extension 等）を残したまま、新パターンを並べる**
  - 二流派並存は将来のバグ温床
- **「新規作成しない」制約を「既存削除もしない」と拡大解釈する**
  - 制約の真意は「無駄な新規を作るな、既存を活かせ、楽するな」
  - **整合確保のために必要な削除は「楽な道」ではなく「必要な労力」**

### 既存資産の「廃止判断」基準

既存資産（enum、ファイル、extension 等）を削除する判断は、以下を**すべて**満たすときに行う:

1. 兄弟機能が確立したパターンと整合させる必要がある
2. 既存資産を残すと、新旧 2 流派が混在する状態になる
3. 影響範囲を Grep / Read で全件特定し、書き換え計画が立てられる
4. Issue 制約（例: 「特定レイヤーを変更しない」）に抵触しないことを使用箇所で確認済み

削除は工数を増やす。それでも整合のために必要なら実施する。
**「削除は楽じゃないから新規併設のほうが安全」は逆。整合破壊のほうが将来コストが高い。**

## [MANDATORY] 他の画面・機能の実装パターン調査

実装に入る前に、**同種の画面・機能がどう実装されているか**を徹底的に調査する。
表面的な検索で済ませず、実際にコードを Read して実装パターンを理解すること。

- 同種の画面・機能（一覧、詳細、フォーム等）の既存実装を **3 件以上** 特定し Read する
- 画面・機能の構成（モジュール分割、状態管理、データ取得パターン）を把握する
- 状態管理スタイルを確認する
- ナビゲーション・ルーティングのパターンを確認する
- エラーハンドリング・ローディング表示のパターンを確認する

特に重要なのは **直近マージ・直近進行中の PR** で確立されたパターン。
ベースブランチが切り替わっている可能性があるため、`git log` で最近の変更を必ず確認する。

## [MANDATORY] 共通コンポーネントの調査・採用

新規にコンポーネントやロジックを作る前に、**既存の共通コンポーネントで代替できないか**を必ず確認する。

共通コンポーネント・ユーティリティの配置先はプロジェクト規約から取得する（`/forge:query-db-rules` で「ディレクトリ構造」「共通コンポーネント」を検索）。

特定した共通コンポーネントは実際に Read して以下を把握する：

- 公開 API（コンストラクタ引数、カスタマイズポイント）
- 使用例（他の画面・機能でどう使われているか）
- 今回の実装で**そのまま使えるか、拡張が必要か、新規作成が必要か**を判断する

## その他の調査ポイント

- 同じドメイン領域の既存モデル・リポジトリ・状態管理
- 実装の参考になる類似のユースケース層
- クラス・インターフェースのシグネチャ
- 命名規則・ファイル配置パターン
- 既存コードと新規実装の依存関係
- 動作確認手段があるか（debug 画面・カタログ等のプロジェクト固有導線）
- 動作確認導線が無い場合は、必要に応じて debug 導線追加を実装計画に含める
- debug 導線に追加する場合、ボタン名は仮名ではなく **画面設計書 ID ベース** で命名する
