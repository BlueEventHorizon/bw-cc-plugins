# UI 実装レビュールール

実装後に「デザイン仕様書」「実装設計書」「Figma」の三点突合を行い、正しい実装になっているか確認するためのルール。

## 前提条件

1. **実装設計書が作成済みであること** — `specs/design/{id}_実装設計書.md`
2. **デザイン仕様書が作成済みであること** — `specs/design/{id}_デザイン仕様書.md`
3. **Figma MCP が利用可能であること** — Figma Desktop 起動 + Dev Mode（`Shift + D`）

## レビュー手順

### 1. 設計書の読み込み

- **実装設計書**: 画面構成、コンポーネント対応表、デザイントークン、状態管理方針、共通化方針
- **デザイン仕様書**: レイアウト仕様、カラーパレット、フォントスタイル、動作・インタラクション

### 2. Figma でデザインを確認

**Figma MCP を使用して確認**:

1. `get_design_context` — デザインコンテキストを取得
2. `get_screenshot` — スクリーンショットで視覚的に確認
3. `get_metadata` — 個別要素のノードタイプを確認

**確認すべき項目**:

| 項目               | 確認内容                       |
| ------------------ | ------------------------------ |
| レイアウト         | 余白、配置、サイズ             |
| 色                 | 背景色、テキスト色、ボーダー色 |
| タイポグラフィ     | フォントサイズ、行間、ウェイト |
| 状態バリエーション | 選択時、非選択時、無効時など   |
| インタラクション   | タップ領域、アニメーション     |
| アイコン・画像     | 使用アセット、サイズ、色       |

**個別要素のノードタイプ確認**:

`get_design_context` の生成コードは**マスク/クリッピング情報を正確に反映しない場合がある**。

| 要素            | 確認内容               | get_metadata で確認                    |
| --------------- | ---------------------- | -------------------------------------- |
| 画像/サムネイル | 円形 or 四角形 or 角丸 | `ellipse` = 円形, `rectangle` = 四角形 |
| アイコン        | 形状とサイズ           | `vector`, `group`, `frame`             |
| ボタン/カード   | border-radius          | `rectangle` の cornerRadius            |

**node-id の鮮度に関する注意**:

- Desktop MCP の場合は **選択中ノードの自動認識** を優先し、node-id なしで `get_design_context` を実行する
- Remote MCP の場合は **最新のリンク** から node-id を再取得して使用する

### 3. 三点突合（必須）

以下の 3 点を突き合わせて差異を確認:

1. **デザイン仕様書** — レイアウト、色、フォント、アセット
2. **実装設計書** — コンポーネント対応、デザイントークン、状態管理
3. **Figma** — 実際のデザイン（スクリーンショット + 構造データ）

**差異があれば必ず修正し、再度突合する。**
いずれかが取得できない場合は、その旨を報告して中断する。

### 4. 類似画面との実装パターン比較

同じ種類の画面（一覧、詳細、シート）の既存実装と比較:

| 確認項目           | 内容                                         |
| ------------------ | -------------------------------------------- |
| コンポーネント使用 | 同種画面と同じコンポーネントを使用しているか |
| 状態管理パターン   | 同種画面と同じ状態管理パターンか             |
| レイアウト構造     | 同種画面と一貫した構造か                     |

## 実装ルール確認チェックリスト

### 必須ルール

| # | ルール                 | 確認内容                                                   |
| - | ---------------------- | ---------------------------------------------------------- |
| 1 | デザイントークン使用   | 色・フォントがハードコードされていないか                   |
| 2 | アセット管理の遵守     | プロジェクト規約に従ったアセット参照を使用しているか       |
| 3 | UI 文字列の管理        | 文字列がハードコードされていないか                         |
| 4 | 既存コンポーネント優先 | コンポーネントカタログにあるコンポーネントを使用しているか |
| 5 | Figma 値を使用         | 設計書にない値を空想で実装していないか                     |
| 6 | 共通化方針の遵守       | 設計書に記載された共通化方針に従っているか                 |

### プロジェクト固有ルールの確認

`/doc-advisor:query-rules` で「UI実装パターン」「コーディング規約」を検索し、該当するルールを読み込んで準拠しているか確認する。

## Figma MCP エラー時の対応

1. **必ずユーザーに報告する** — 何が起きたかを明示する
2. **不確かな情報で進めない** — MCP が失敗した場合、推測でレビューしない
3. **代替手段を試す** — REST API での取得を試みる
4. **検証が不十分なら中断** — 「Figma 情報を取得できませんでした」と報告する

## よくある失敗パターン

1. **設計書だけ見て実装し、Figma を確認しなかった** — 実装前・中・後に Figma を確認
2. **色やフォントをハードコードした** — デザイントークンを使用
3. **既存コンポーネントを探さず新規作成した** — コンポーネントカタログと swift-selena MCP / Grep で検索
4. **プロジェクト規約のアセット管理を使わなかった** — プロジェクト規約に従ったアセット参照を使用
5. **UI 文字列をハードコードした** — プロジェクトの i18n / 文字列管理方法を使用
6. **状態バリエーションを実装し忘れた** — Figma で全状態を確認
7. **設計書にない値を空想で実装した** — Figma MCP で正確な値を取得
8. **get_design_context の生成コードを鵜呑みにした** — 画像・アイコンは get_metadata でノードタイプ確認
9. **類似画面の実装パターンを確認しなかった** — 同種画面と比較して一貫性を確認

## 完了後

- コード生成: プロジェクトのコード生成コマンドを実行（例: `melos run gen` / `swift build` など、プロジェクト規約を確認）
- 新規コンポーネント作成時: プロジェクトのコンポーネントカタログを更新する（存在する場合）
