# Phase 2: 仕様書調査ルール

`/forge:query-db-specs` を使い、Issue のタイトル・本文から **抽出した検索キーワード** または **短い自然文のタスク記述** を `args` として渡し、関連仕様書を特定する。Issue 本文をそのまま貼り付けない。

## 必須ルール

- 特定した仕様書は**すべて**実際に Read tool で読み込む（検索結果の見出しだけで済ませない）
- API レスポンス構造、画面要件、制約・バリデーションルールを完全に把握する
- 関連する可能性がある周辺ドキュメントも積極的に読む

## 外部リポジトリ・symlink への対応

仕様書ディレクトリが symlink で別 Git 管理リポジトリを指している場合がある。

- ローカル相対パスをそのまま Issue に貼らず、**実体の GitHub URL** を確認して記録する
- 対象パスがローカルに存在しない場合でも、**仕様書が無いと判断してはいけない**。外部リポジトリを直接参照して対象ドキュメントの有無を確認する
- 外部リポのリポジトリ名・URL 形式はプロジェクト規約から取得（`/forge:query-db-rules` で「外部仕様書リポジトリ」「symlink」等を検索）

## UI Issue の場合の追加作業

画面設計書・確認/調整事項ドキュメントが存在する場合、必ず読み、以下を記録する（Phase 6 で `/anvil:prepare-figma` に渡す）：

- 画面設計書ファイルパス
- 画面設計書の GitHub URL（Issue 記載用）
- 確認・調整事項ファイルパス（存在する場合）
- Figma URL（画面設計書に記載されているもの）
- 画面 ID・画面名
