# 実装設計書 作成ルール

デザイン仕様書と既存ソースコードを照合し、実装設計書（How: どう作るか）を作成するためのルール。

## ドキュメントの役割

| ドキュメント | 目的 | 含める内容 | 含めない内容 |
| ------------ | ---- | ---------- | ------------ |
| **実装設計書** | どう作るか（How） | 使用コンポーネント、デザイントークン、状態管理、データフロー、アクション | ソースコード |

## 重要な原則

### 1. 設計書だけで誰でも実装できる

- 設計書を読めば、ソースコードがなくても実装方法が分かる
- 曖昧な記述を避け、具体的に記載する

### 2. 実装設計書にソースコードは書かない（原則）

- **例外**: 使用方法を示す必要がある場合の短いコード例は許容
  - デザイントークンの参照方法（プロジェクト固有の呼び出し例）
  - アイコン・色トークンの適用方法
  - これらは「設計上の決定事項」であり「実装コード」ではない
- 上記以外で特殊な事情がある場合は、人間に問い合わせて判断を仰ぐ

### 3. 既存実装の共通パターンを確認する

- 設計書を作成する前に、類似画面・機能の既存実装を必ず確認する（Phase 4 / Phase 5）
- 共通パターン（画面構成、状態管理、コンポーネントの使い方）に従う
- デザインと既存実装で差分がある場合は、設計書に明記して確認を促す
- 参照すべき既存コードを設計書に記載する

### 4. 共通化の判断と設計での実施

- 既存コード調査で **共通化できるコード** を発見した場合は、設計フェーズで共通化方針を決定する
- 共通化の対象（UI コンポーネント / Mapper / UseCase / Utility 等）を明記する
- 既存の共通実装がある場合は **新規作成せず再利用** する
- 新規共通化が必要な場合は、理由・影響範囲・適用箇所を設計書に記載する
- **共通化コードの配置先はプロジェクト規約に従う**（`/forge:query-db-rules` で「ディレクトリ構造」「共通コンポーネント」を検索）

### 5. アイコン・画像はプロジェクト規約に従う

- プロジェクトのアセット管理方法（共通アセット定義 / アセットジェネレーター等）を使用する
- ハードコードされた素材参照や、フレームワーク標準のアイコンを安易に使わない
- 使用可能なアセット一覧・参照方法は `/forge:query-db-rules` で「アセット管理」を検索

### 6. テーマ・デザイントークンを使用する（ハードコードしない）

- 色・フォント・スペーシングをハードコードしない
- プロジェクトのデザイントークン（カラー / テキストスタイル / スペーシング）を使用する
- トークン参照方法・命名規則は `/forge:query-db-rules` で「デザイントークン」を検索
- 言語別テキストスタイルがある場合は、対象テキストの言語に応じて正しく選択する

### 7. Typography 対応表を実装設計書に含める [MANDATORY]

デザイン仕様書の全テキストノードについて、デザイントークンへの変換表を **実装設計書に必ず記載** する。
手順・変換ルール: [phase-11-typography-mapping.md](phase-11-typography-mapping.md)

| ノード名 | size | weight | line_h | color (hex) | 言語 | textStyle トークン | color トークン |
| -------- | ---- | ------ | ------ | ----------- | ---- | ------------------ | -------------- |

**禁止**: 「使用デザイントークン」セクションに数行だけ書いて表を省略 / 実装時に推測で選ぶ。

### 8. アクション一覧を実装設計書に含める [MANDATORY]

デザイン仕様書「アクション一覧」の各行について、実装方針（遷移先・使用コンポーネント・条件）を表にする。
見た目だけ先に実装し、タップ・クリック挙動を後回しにしない。

### 9. API 定義を確認してモデリングに活用する

- API を使用する画面・機能では、**必ず API 定義を確認** してから設計書を作成する
- API 定義の場所はプロジェクト規約から取得（`/forge:query-db-rules` で「API 定義」「OpenAPI」を検索）
- **活用方法**:
  - レスポンスフィールドからドメインモデルのプロパティを決定
  - ページネーション情報を件数表示・追加読み込みに使用
  - 各フィールドの型・nullable 情報を null 安全な実装に反映

## 作成前に確認すべきもの

### 既存コンポーネントの確認

Glob / Grep でプロジェクトの共通コンポーネント配置を検索する。
Serena MCP 等のコードシンボル検索ツールが利用可能であれば併用する：

| 操作 | 用途 |
| ---- | ---- |
| シンボル検索 | コンポーネントクラス名で検索 |
| シンボル定義取得 | クラスの詳細な実装を確認 |
| 参照検索 | コンポーネントの使用箇所を確認 |

### デザイントークンの確認

プロジェクトのデザインシステム定義（カラー、タイポグラフィ、スペーシング等）を確認する。
ハードコードの値は使用せず、プロジェクトのデザイントークンを使用する。

### ドメインモデル・Mapper の確認

- 画面で表示するデータに対応するドメインモデルがあるか
- ドメインモデルから UI モデルへの変換メソッドがあるか
- 配置先はプロジェクト規約に従う

## 用語定義（混同防止）

- **パーツ名**: デザイン仕様書で定義する画面内の領域・要素の論理名（画面固有の名前）
- **コンポーネント名**: 既存または新規のコンポーネント（クラス名・ファイル名）

設計書では「パーツ名 → 使用コンポーネント名」の対応が一意に追えるように記載する。

## よくある失敗パターン

1. **既存コンポーネントの検索が不十分** — 1 つの検索パターンだけで諦めない
2. **実装設計書にソースコードを書いてしまう** — 特殊な事情がある場合は人間に確認
3. **アイコンを直接定数 / 標準セット参照で書いてしまう** — プロジェクト規約のアセット参照を使用
4. **色やテキストスタイルをハードコードしてしまう** — テーマ定義を使用する
5. **既存のモデルやユーティリティを確認せずに新規作成** — Phase 5 の調査結果を再確認
6. **類似画面・機能の実装を確認しない** — 同種の既存実装を確認し、共通パターンに従う
7. **デザイン仕様書のパーツ名と不一致** — パーツ名は必ず一致させる
8. **API 定義を確認せずにモデリング** — レスポンスフィールドを確認せずにドメインモデルを設計すると、マッピング漏れが発生
9. **言語別テキストスタイルを間違える** — 対象テキストの言語に合うスタイルを選択
10. **Typography 対応表を省略** — 全テキストノード → トークンの表が無いと font 修正が大量発生する（[phase-11-typography-mapping.md](phase-11-typography-mapping.md)）
11. **共用コンポーネントを 1 画面の typography に変更** — 画面専用コンポーネントを作る
12. **アクション一覧を実装設計書に書かない** — タップ挙動の後回し・漏れの原因
