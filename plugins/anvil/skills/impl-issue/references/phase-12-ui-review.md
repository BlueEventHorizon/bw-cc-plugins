# UI 実装レビュールール

実装後に Figma SS・実装キャプチャ・コード/設計書の三点突合を行い、正しい実装になっているか確認するためのルール。

## 前提条件

1. **実装設計書が作成済みであること**
2. **デザイン仕様書が作成済みであること**
3. **Figma MCP が利用可能であること** — Figma Desktop / Remote のいずれか、Dev Mode 起動
4. **実機キャプチャを取得できること** — 実装後のアプリ画面を Emulator / Simulator / Web ブラウザ等プロジェクトのプラットフォームに応じた手段でキャプチャできること

## レビュー手順

### 1. 設計書の読み込み

- **実装設計書**: 画面構成、コンポーネント対応表、デザイントークン、状態管理方針、共通化方針
- **デザイン仕様書**: レイアウト仕様、カラーパレット、フォントスタイル、動作・インタラクション

### 2. Figma でデザインを確認

**Figma MCP を使用して確認**:

1. `get_design_context` — デザインコンテキストを取得
2. `get_screenshot` — スクリーンショットで視覚的に確認（**採用判断はここが基準**。node ツリーにあっても写っていなければ未使用）
3. `get_metadata` — 個別要素のノードタイプを確認

**確認すべき項目**:

| 項目               | 確認内容                       |
| ------------------ | ------------------------------ |
| レイアウト         | 余白、配置、サイズ             |
| 色                 | 背景色、テキスト色、ボーダー色 |
| タイポグラフィ     | フォントサイズ、行間、ウェイト |
| 状態バリエーション | 選択時、非選択時、無効時など   |
| インタラクション   | タップ領域、アニメーション     |
| アイコン・画像     | 使用アセット、サイズ、色       |

**個別要素のノードタイプ確認**:

`get_design_context` の生成コードは**マスク / クリッピング情報を正確に反映しない場合がある**。

| 要素              | 確認内容               | get_metadata で確認                    |
| ----------------- | ---------------------- | -------------------------------------- |
| 画像 / サムネイル | 円形 or 四角形 or 角丸 | `ellipse` = 円形, `rectangle` = 四角形 |
| アイコン          | 形状とサイズ           | `vector`, `group`, `frame`             |
| ボタン / カード   | border-radius          | `rectangle` の cornerRadius            |

**node-id の鮮度に関する注意**:

- Desktop MCP の場合は **選択中ノードの自動認識** を優先し、node-id なしで `get_design_context` を実行する
- Remote MCP の場合は **最新のリンク** から node-id を再取得して使用する

### 3. 実装後キャプチャを取得

実装した画面を実機キャプチャする。これはデザイン仕様書のプレビュー画像とは別物で、**最新コード反映後の実アプリ描画**を確認するためのキャプチャ。

#### 取得手段の選択 [MANDATORY]

サイレントスキップ禁止。以下の順で選択する：

1. **プロジェクトのキャプチャ SKILL が存在するか確認**:
   - `/forge:query-db-rules` で「キャプチャ」「screenshot」「capture」を検索
   - プロジェクト固有 SKILL（例: `capture-emulator-screen`、`screenshot-web` 等）がヒットすれば `Skill` ツールで委譲する

2. プロジェクト SKILL が存在しない場合は、対象プラットフォームに応じて手段を選ぶ:

   | プラットフォーム     | 取得手段の例                                                                                                                         |
   | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
   | iOS / Android アプリ | Emulator / Simulator のスクリーンショット機能（`xcrun simctl io booted screenshot <path>`、`adb exec-out screencap -p > <path>` 等） |
   | Web                  | ブラウザ自動化ツール（`playwright`、`puppeteer` 等）でのスクリーンショット                                                           |
   | Desktop アプリ       | OS のスクリーンショット機能、または自動化ツール                                                                                      |
   | TUI / CLI            | ターミナル出力のテキストキャプチャ                                                                                                   |

3. **AI が単独でキャプチャを取れない場合**は `AskUserQuestion` で以下の 3 択を提示する:

   - **ユーザー依頼**: 「ユーザーが手元で画面をキャプチャし、画像パスを返す」を案内し、画像パスを受け取って続行
   - **スキップ**: キャプチャ無しでレビューを進める。ただし **レビュー結果に「実機キャプチャ未実施」を明示** し、三点突合が不完全であることを Phase 13 の commit メッセージ・PR 本文に反映する
   - **中断**: ここで中断

#### 取得時に必ず記録すること

1. ホットリロード / 再起動 / 完全リビルドのどれで反映したか
2. 対象画面までの到達手順
3. キャプチャ画像の保存先（推奨: `specs/design/{id}/captures/<state>.png`）
4. 未確認の状態バリエーション（どのバリエーションをキャプチャできなかったか）

### 4. 三点突合（必須）

以下の 3 点を突き合わせて差異を確認:

1. **Figma SS** — 実際のデザイン（スクリーンショット + 構造データ）
2. **実装キャプチャ** — 実機上で実際に描画された画面
3. **コード / 設計書** — 値・条件・構造の根拠（デザイン仕様書、実装設計書、実装コード）

**差異があれば必ず修正し、再度突合する。**
いずれかが取得できない場合は、その旨を報告して中断する。

#### 仕様書ベースの差分検証 [MANDATORY]

| 観点                          | 確認手順                                                                                                                                                                |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 構造順序                      | デザイン仕様書のレイアウト定義と実装コードの子要素順を 1 行ずつ目視照合                                                                                                 |
| **Typography**                | 実装設計書の **Typography 対応表** とコードのテキストスタイル参照を Grep 照合（[phase-11-typography-mapping.md](phase-11-typography-mapping.md)）                       |
| 値の一致                      | デザイン仕様書の `font` / `color` / `size` / `padding` / `gap` / `radius` を Grep で照合                                                                                |
| 条件分岐                      | 仕様書備考の文言（「100 文字以上」等）と条件式が一致                                                                                                                    |
| **アクション**                | デザイン仕様書「アクション一覧」の全行が実装済み（空ハンドラ / TODO なし）                                                                                              |
| **共用コンポーネント**        | 共用コンポーネントが 1 画面専用 typography に書き換わっていない                                                                                                         |
| スクロール構造                | 仕様書の `scroll` / `sticky` と実装の対応プリミティブが一致                                                                                                             |
| 既存流用箇所                  | 既存コンポーネントの値が仕様書と一致（Grep 確認）                                                                                                                       |
| **件数（一覧 / ページング）** | 仕様書の「初期表示件数 / 追加件数 / 全体上限」と実装値が一致（後述「観点 ❶」）                                                                                          |
| **Figma 実値 vs トークン**    | Figma API で取得した `style` の fontSize / fontWeight / lineHeightPx / fills.color と採用トークンの実体が一致、または乖離が実装設計書に明記されている（後述「観点 ❸」） |
| **直書き禁止**                | hex / 数値を直書きせず、トークン or 名前付き定数経由（後述「観点 ❹」）                                                                                                  |

### 5. 類似画面との実装パターン比較

同じ種類の画面（一覧、詳細、シート）の既存実装と比較:

| 確認項目           | 内容                                         |
| ------------------ | -------------------------------------------- |
| コンポーネント使用 | 同種画面と同じコンポーネントを使用しているか |
| 状態管理パターン   | 同種画面と同じ状態管理パターンか             |
| レイアウト構造     | 同種画面と一貫した構造か                     |

### プロジェクト固有ルールの確認

`/forge:query-db-rules` で「UI 実装パターン」「コーディング規約」を検索し、該当するルールを読み込んで準拠しているか確認する。

## 最終レビューで特に注意すべき観点

過去の実装で見落とされがちで、レビュー時に意識的に確認すべき観点を以下にまとめる。
これらは三点突合の **追加チェック項目** として扱う。

### ❶ 件数（初期表示 / 追加 / 全体上限）の明示確認

仕様書（画面設計書）には「最大表示件数 N 件」「もっと見るで追加 N 件」と書かれていても、**「初期表示件数（1 ページ目で取得する件数）」が明示されていないケース**が多い。実装値はどこから来ているか、根拠を必ず突き止める。

| 確認手順                                                                                           | 失敗例                                                           |
| -------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| 1. 画面設計書を Read し「初期 N 件」「最大 N 件」「追加 N 件」がそれぞれ明示されているか確認       | 「最大 N 件」「追加 N 件」だけ書いてあり初期件数が不明なまま実装 |
| 2. Figma サンプル数（カードの並び）を**件数指定と勘違いしない**                                    | 見た目だけで「初期 N 件」と決めつける                            |
| 3. 直近 PR の前例を確認し、同じ判断基準を踏襲                                                      | 前例と異なる解釈で実装し、レビューで揉める                       |
| 4. 不明な値は **PdM / デザイナー / 開発リーダ** に確認し、出典（口頭・チケット・チャット等）を記録 | 出典を残さないため、後から「誰がいつ決めたか」分からなくなる     |
| 5. 実装設計書 §件数の決定経緯 / Issue 本文に **値・根拠・出典** を表形式で残す                     | 設計書に書かず、実装コードのコメントだけに走り書き               |

### ❷ Figma の「画面外」のデザイナー作業メモ・コメントの確認

Figma の画面フレームの中だけを `get_design_context` で見ると、**デザイナーがフレーム外に置いた作業メモやレビューコメントを見落とす**ことがある。

| 確認手順                                                                                                                                          | 失敗例                                                             |
| ------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| 1. メインフレームの **半径 5,000–10,000 px 以内のページ直下フレーム**を抽出し、`メモ` / `Memo` / `Note` / `仕様` 等の名前を持つインスタンスを確認 | 隣のフレームやセクションの中に置かれた「メモ」インスタンスを見逃す |
| 2. **Figma コメント機能**で pinned されたコメントを取得（`/v1/files/{key}/comments` 経由）し、対象フレーム内/近傍のものを精査                     | 「件数追加」「最大個数：X 個でお願いします」等の指示を見逃す       |
| 3. 名前ベースのキーワード検索（`仕様`、`件数`、`最大`、`もっと`、`追加`、画面 ID 等）をフレーム周辺に対して行う                                   | 命名が `Frame XXXXXX` のようなデフォルトのため気付かない           |
| 4. 隣接する **バリエーションフレーム**（タブ切替後 / 状態違い）も確認                                                                             | メインフレームしか見ず、隣のフレームにある別タブの注釈を見逃す     |

> 参考: ページ全体を `depth=2` で取得 → bounding box から距離計算でメインフレーム周辺を抽出するスクリプトを書くのが手早い。

### ❸ Figma 実値と採用トークン実体の突合

「実装設計書にトークン X を採用」と書いていても、**そのトークンの実体（fontWeight / lineHeight / hex）が Figma 値と一致しているとは限らない**。プロジェクトのデザインシステムが Figma と乖離しているケースがある。

| 確認手順                                                                                                                          | 失敗例                                                                |
| --------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| 1. Figma API で対象 TEXT ノードの `style` を取得（`fontSize` / `fontWeight` / `lineHeightPx` / `lineHeightUnit` / `fills.color`） | 仕様書の値（人手で書かれた近似値）だけを見て突合した気になる          |
| 2. 採用トークンの **実体ファイル**を Read                                                                                         | トークン名から「Regular = w400」と推測し、実体（w300 等）を確認しない |
| 3. 「Figma 実値 / トークン実体」併記の表で乖離を可視化（実装設計書 §テキストスタイル）                                            | 表が「仕様書の値」しか書いておらず、実装時にギャップが見えない        |
| 4. 乖離があれば **直近 PR の前例と同じ判断**で揃える。採用方針をコードコメント（クラス文書）にも書く                              | 前例と異なる判断で勝手にトークンを上書き or 直接 hex で書く           |

### ❹ 前例 PR との名前付き定数の整合（直書き禁止の徹底）

プロジェクトに Spacing 用のデザイントークンが無い場合、前例 PR ではコンポーネント内の名前付き定数（言語に応じて `static const` / `const` / `private static final` 等）で命名している場合がある。新規コンポーネントでも **同名・同値で踏襲**すること。

| 確認手順                                                                                       | 失敗例                                            |
| ---------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| 1. 直近 PR の同種コンポーネント（Phase 4 で特定した参考 PR）を Read し、名前付き定数一覧を確認 | 前例を参照せず、独自命名の定数を作る              |
| 2. **同じ用途の定数は同名・同値で踏襲**。新規追加は前例の命名規則に揃える                      | 微妙に違う命名や同値の別定数を作る                |
| 3. ファイル内に padding / gap / size の **数値直書き** が残っていないか Grep で検証            | 数値直書きが残ったままマージ                      |
| 4. 例外として前例でも直書きされている値は踏襲                                                  | 前例を超えて const 化しようとして余計な改変が入る |
| 5. ページサイズ / 最大件数も **名前付き定数で命名**                                            | コード内に数値を直書き                            |

### ❺ プレビュー画像の存在確認

デザイン仕様書の §視覚比較セクションは「Figma SS とプレビュー画像を並べて見比べる」前提で書かれているため、プレビュー画像が存在しないと視覚比較が機能しない。

| 確認手順                                                           | 失敗例                                                         |
| ------------------------------------------------------------------ | -------------------------------------------------------------- |
| 1. デザイン仕様書で参照されているプレビュー画像の存在を確認        | デザイン仕様書には参照があるが画像が存在せず、視覚比較できない |
| 2. 不在なら `/anvil:prepare-figma` 経由で再生成                    | 「再生成すれば良い」と知らず、画像なしのまま実装が進む         |
| 3. 生成後、Figma SS と並べて目視比較し、構造誤りが無いことを再確認 | プレビュー生成だけして比較しない                               |

## Figma MCP エラー時の対応

1. **必ずユーザーに報告する** — 何が起きたかを明示する
2. **不確かな情報で進めない** — MCP が失敗した場合、推測でレビューしない
3. **代替手段を試す** — REST API での取得を試みる
4. **検証が不十分なら中断** — 「Figma 情報を取得できませんでした」と報告する

## よくある失敗パターン

1. **設計書だけ見て実装し、Figma を確認しなかった** — 実装前・中・後に Figma を確認
2. **色やフォントをハードコードした** — デザイントークンを使用
3. **既存コンポーネントを探さず新規作成した** — コンポーネントカタログと Grep / シンボル検索で検索
4. **標準アイコンセットを直接使用した** — プロジェクト規約のアセット参照を使用
5. **UI 文字列をハードコードした** — プロジェクトの i18n / 文字列管理方法を使用
6. **状態バリエーションを実装し忘れた** — Figma で全状態を確認
7. **設計書にない値を空想で実装した** — Figma MCP で正確な値を取得
8. **get_design_context の生成コードを鵜呑みにした** — 画像・アイコンは get_metadata でノードタイプ確認
9. **類似画面の実装パターンを確認しなかった** — 同種画面と比較して一貫性を確認
10. **仕様書の構造照合省略** → レイアウト定義と実装コードの子要素順を 1 行ずつ目視照合
11. **既存流用箇所の値未確認** → Grep で font / color / size / padding を仕様書と突合
12. **lint 回避で仕様書を妥協** → `// ignore` 等で貫く、共通基盤拡張は別 Issue
13. **条件分岐の仕様書文言逸脱** → 文言と条件式を 1 対 1 で照合
14. **Typography 対応表なし / 推測トークン** → [phase-11-typography-mapping.md](phase-11-typography-mapping.md) で全ノード照合
15. **共用コンポーネントの typography 変更** → revert、画面専用コンポーネント化
16. **アクション一覧の未実装** → タップハンドラ・遷移の漏れ確認
17. **件数（初期 / 追加 / 上限）の根拠未確認** → 仕様書の明示性をチェック、Figma サンプル数を件数指定と勘違いしない、出典を実装設計書に表形式で残す（前述「観点 ❶」）
18. **Figma の画面外メモ・コメント未確認** → ページ直下のフレーム周辺 / `/v1/files/{key}/comments` を必ず精査（前述「観点 ❷」）
19. **トークン名から推測してしまう** → トークンの実体ファイルを Read し、Figma API の `style` と突合（前述「観点 ❸」）
20. **デザインシステム既存乖離を勝手に上書き / 直書きで回避** → 直近 PR の前例と同じ判断で揃え、コードコメントに乖離を併記（前述「観点 ❸」）
21. **Spacing を直書き / 独自命名** → 前例 PR の名前付き定数を踏襲、ファイル内に数値直書きが残らないよう Grep 検証（前述「観点 ❹」）
22. **プレビュー画像が無い状態で進める** → `/anvil:prepare-figma` 等で生成し、Figma SS と並べて視覚比較（前述「観点 ❺」）

## 完了後

- コード生成: プロジェクトのコード生成コマンドを実行（プロジェクト規約を確認）
- 新規コンポーネント作成時: プロジェクトのコンポーネントカタログを更新する（存在する場合）
