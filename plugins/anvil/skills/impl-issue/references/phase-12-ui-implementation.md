# UI 実装ルール

実装設計書に基づいて UI を実装する際のルール。

## Design Implementation Policy [MANDATORY]

**UI 実装においては Figma が唯一の正（Single Source of Truth）です。**

| # | 原則                      | 説明                                                                                      |
| - | ------------------------- | ----------------------------------------------------------------------------------------- |
| 1 | **Figma に一致させる**    | 設計書やデザイン仕様書の記載・不記載に関わらず、実装は Figma のデザインに一致させる       |
| 2 | **困ったら Figma を確認** | 実装中に迷った場合、不明点がある場合は、まず Figma を確認する                             |
| 3 | **Figma を優先**          | 設計書と Figma で差異がある場合は、**Figma を優先**する（設計書の更新漏れの可能性が高い） |

**フォントに関する例外**: フォントファミリーはシステムフォントを使用する。Figma から参照するのはウェイト・サイズ・line-height のみ。

## 必須参照ドキュメント

実装を開始する前に、以下のドキュメントを**必ず読み込む**（取得方法は `/doc-advisor:query-rules` を使用）:

| ドキュメント           | 内容                                         | 取得方法                                                |
| ---------------------- | -------------------------------------------- | ------------------------------------------------------- |
| UI 実装パターン        | 画面構成、コンポーネント、状態管理のパターン | `/doc-advisor:query-rules` で「UI実装パターン」を検索   |
| コーディング規約       | 命名規則、ディレクトリ構造、値取得ルール     | `/doc-advisor:query-rules` で「コーディング規約」を検索 |
| コンポーネントカタログ | 既存コンポーネント一覧（存在する場合）       | プロジェクト規約を確認                                  |

## 必須実装ルール

| ルール                 | 説明                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| 既存コンポーネント優先 | 新規作成前に必ず既存コンポーネントを検索                             |
| デザイントークン使用   | 色・フォントのハードコード禁止。プロジェクトのテーマ定義を使用       |
| アセット管理の遵守     | アイコン・画像はプロジェクト規約に従った方法で参照                   |
| UI 文字列の管理        | 文字列のハードコード禁止。プロジェクトの i18n / 文字列管理方法を使用 |
| Figma 値を使用         | 設計書にない値は空想で実装しない                                     |
| 共通化方針の遵守       | 設計書に記載された共通化方針に従い、既存共通実装を優先               |

## 状態管理

状態管理ライブラリ・パターンはプロジェクト規約（`/doc-advisor:query-rules` で「状態管理」を検索）に従う。

| ケース                                | 考慮事項                           |
| ------------------------------------- | ---------------------------------- |
| シート/ダイアログ内の一時的な選択状態 | 画面ローカルの状態として管理       |
| フォーム入力の一時状態                | 画面ローカルの状態として管理       |
| API から取得したデータ                | 永続化・共有が必要な状態として管理 |
| 画面間で共有される状態                | グローバル状態として管理           |

## 設計書にない値の取得 [MANDATORY]

**設計書に記載のない値は空想で実装しない。**

設計書に値がない場合:

1. Figma MCP で `get_design_context` を実行
2. `get_variable_defs` で変数定義を取得
3. `get_metadata` でノードのサイズ情報を取得

**禁止事項**:

- 「だいたいこのくらいだろう」と推測して実装する
- 他の類似コンポーネントから値をコピーして流用する
- よくある値を根拠なく使用する

## 実装ファイルの配置

プロジェクトのディレクトリ規約（`/doc-advisor:query-rules` で「ディレクトリ構造」を検索）に従う。

## Figma MCP エラー時の対応

1. **必ずユーザーに報告する** — 何が起きたかを明示する
2. **不確かな情報で進めない** — MCP が失敗した場合、推測で実装しない
3. **代替手段を試す** — REST API での取得を試みる
4. **検証が不十分なら中断** — 「Figma 情報を取得できませんでした」と報告する

## よくある失敗パターン

1. **設計書だけ見て実装し、Figma を確認しなかった** — 実装前・中・後に Figma を確認
2. **色やフォントをハードコードした** — デザイントークンを使用
3. **既存コンポーネントを探さず新規作成した** — コンポーネントカタログと swift-selena MCP / Grep で検索
4. **プロジェクト規約のアセット管理を使わなかった** — プロジェクト規約に従ったアセット参照を使用
5. **UI 文字列をハードコードした** — プロジェクトの i18n / 文字列管理方法を使用
6. **状態バリエーションを実装し忘れた** — Figma で全状態を確認
7. **設計書にない値を空想で実装した** — Figma MCP で正確な値を取得
8. **UI 実装パターン文書を読まなかった** — 実装前に必ず読み込む
