# 要件定義書・設計書で優先すること [MANDATORY]

要件定義書・設計書を作成するとき、および要件・設計をレビューするときに優先すべき価値観と、
主目的にしてはいけない事項を定める。

**本文書のスコープ**: 要件定義書および設計書の両方に共通する普遍原則（排除ルール）。
What/How のどちらに何を書くかの**振り分け判定**は本文書の対象外であり、
`${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md` を参照すること。

---

## 1. 優先するもの（構造的設計原則）

要件定義書・設計書は以下を明示することを最優先とする:

| 原則       | 内容                                                       |
| ---------- | ---------------------------------------------------------- |
| 責務分割   | 各モジュール・コンポーネントが単一の責務を持つこと         |
| 境界設定   | モジュール間の境界が明確で、漏れ・重複がないこと           |
| 依存方向   | 依存の向きが一方向で、循環依存がないこと                   |

これらは Yes/No で判定可能な構造的性質であり、設計レビューで直接点検できる。

---

## 2. 結果として得られる品質特性（直接の目標指標にしない）

以下の品質特性は §1 を満たした**結果として**得られるものであり、
**直接の数値目標に翻訳してはならない**:

- 変更容易性
- 運用しやすさ
- テスト容易性

**理由**: Goodhart の法則。
副産物的な品質特性を直接の目標指標に据えると、構造原則を迂回して
見かけ上の指標値を改善する設計・実装が生まれ、本来の品質は損なわれる。
例: 「変更箇所数を N 以下に」という目標は、責務分割を歪めて無理に箇所を減らす設計を誘発する。

品質特性は §1 の原則が満たされているかを通じて間接的に検証する。

---

## 3. 主目的にしてはいけないもの [CRITICAL]

以下は要件定義書・設計書の主目的にしてはならない:

### 3.1 構造品質の定量化

構造品質（責務分割・境界設定・依存方向）の定量化は禁止する:

- ❌ baseline 固定 → 再計測 → 達成率判定のパイプライン
- ❌ 「flag 数を -10%」「クラス行数を ≤ N」等の定量目標
- ❌ 構造改善を数値目標に翻訳した達成率評価

**理由**: 構造品質は Yes/No で判定可能である（責務が分かれているか、境界が引けているか、依存が一方向か）。
これを数値化すると、判定可能な問いを数値の増減問題に歪める。
Yes/No で十分なものを数値化で代替しないこと。

### 3.2 品質特性の直接数値化

変更容易性・運用しやすさ・テスト容易性を直接の数値目標にしないこと（§2 の理由による）。

### 3.3 進捗管理 KPI・達成率・評価表の見栄え

- ❌ 評価表の行数・項目数を設計書の完成度の指標にする
- ❌ 「計測できるように書く」が動機化した要件・設計記述
- ❌ KPI 表・達成率表の作成自体を成果物とする

## 4. 典型的な倒錯パターン

過去に発生した倒錯の類型を列挙する。同様のパターンが発生した瞬間、設計判断を引き戻すこと:

| パターン                                                                 | 問題                                 |
| ------------------------------------------------------------------------ | ------------------------------------ |
| 構造改善目標を数値目標（-X%、≤N）に翻訳し baseline → 再計測パイプライン  | Yes/No で足りる判断を歪める          |
| 評価表の行数・項目数を設計書の完成度の指標にする                         | 見かけの整備を本質の代わりにする     |
| レビューで「定量目標がない」と指摘し、本質的でない数値目標を事後に捏造   | Goodhart の誘発                      |
| 変更容易性を直接数値目標化（修正箇所数、修正時間等）                     | 構造原則を迂回する設計を誘発         |

---

## 5. 判断基準

設計判断・レビュー時に以下を自問する:

1. **この目標は Yes/No で判定可能か？** → Yes なら定量化しない
2. **この指標は本質的な品質特性（ユーザー価値・構造原則）に直接対応しているか？** → No なら採らない
3. **この数値は「計測できるように書く」ことが動機になっていないか？** → Yes なら書き直す

---

## 6. 定量目標そのものの要件/設計への振り分け

本文書は「そもそも構造品質を定量化しない」という排除ルールを扱う。

定量目標（応答時間・スループット等、運用で検証可能なもの）を
**要件定義書と設計書のどちらに書くか**という振り分け判定は本文書の範囲外である。

→ 振り分け判定は `${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md` §4.6 を参照。
