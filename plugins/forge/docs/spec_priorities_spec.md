# 要件定義書・設計書で優先すること [MANDATORY]

要件定義書・設計書を作成するとき、および要件・設計をレビューするときに優先すべき価値観と、
主目的にしてはいけない事項を定める。

**本文書のスコープ**: 要件定義書および設計書の両方に共通する普遍原則（排除ルール）。
What/How のどちらに何を書くかの**振り分け判定**は本文書の対象外であり、
`${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md` を参照すること。

---

## 1. 優先するもの（構造的設計原則）

要件定義書・設計書は以下を明示することを最優先とする:

| 原則     | 内容                                               |
| -------- | -------------------------------------------------- |
| 責務分割 | 各モジュール・コンポーネントが単一の責務を持つこと |
| 境界設定 | モジュール間の境界が明確で、漏れ・重複がないこと   |
| 依存方向 | 依存の向きが一方向で、循環依存がないこと           |

これらは Yes/No で判定可能な構造的性質であり、設計レビューで直接点検できる。

---

## 2. 結果として得られる品質特性（直接の目標指標にしない）

以下の品質特性は §1 を満たした**結果として**得られるものであり、
**直接の数値目標に翻訳してはならない**:

- 変更容易性
- 運用しやすさ
- テスト容易性

**理由**: Goodhart の法則。
副産物的な品質特性を直接の目標指標に据えると、構造原則を迂回して
見かけ上の指標値を改善する設計・実装が生まれ、本来の品質は損なわれる。
例: 「変更箇所数を N 以下に」という目標は、責務分割を歪めて無理に箇所を減らす設計を誘発する。

品質特性は §1 の原則が満たされているかを通じて間接的に検証する。

---

## 3. 主目的にしてはいけないもの [CRITICAL]

以下は要件定義書・設計書の主目的にしてはならない:

### 3.1 構造品質の定量化

構造品質（責務分割・境界設定・依存方向）の定量化は禁止する:

- ❌ baseline 固定 → 再計測 → 達成率判定のパイプライン
- ❌ 「flag 数を -10%」「クラス行数を ≤ N」等の定量目標
- ❌ 構造改善を数値目標に翻訳した達成率評価

**理由**: 構造品質は Yes/No で判定可能である（責務が分かれているか、境界が引けているか、依存が一方向か）。
これを数値化すると、判定可能な問いを数値の増減問題に歪める。
Yes/No で十分なものを数値化で代替しないこと。

### 3.2 品質特性の直接数値化

変更容易性・運用しやすさ・テスト容易性を直接の数値目標にしないこと（§2 の理由による）。

### 3.3 進捗管理 KPI・達成率・評価表の見栄え

- ❌ 評価表の行数・項目数を設計書の完成度の指標にする
- ❌ 「計測できるように書く」が動機化した要件・設計記述
- ❌ KPI 表・達成率表の作成自体を成果物とする

### 3.4 未確定定量値の捏造

AI が未確定の定量値を推測して、要件・設計に書いてはならない:

- ❌ 実行後に人間が体感して決める値を、数値例から推測して目標化する
- ❌ システム文脈の当事者が決めるべき値を、AI が一般論で補完する
- ❌ 根拠のない目安値を、判定基準・合格基準・設計目標として採用する

**理由**: 定量目標はシステム文脈、ユーザー要求、ビジネス制約、既存契約、実測結果によって決まる。
第三者の AI が当てはめた数値は設計判断の根拠にならない。

未確定の定量目標は、要件定義書の未確定事項（TBD）として扱う。

## 4. 典型的な倒錯パターン

過去に発生した倒錯の類型を列挙する。同様のパターンが発生した瞬間、設計判断を引き戻すこと:

| パターン                                                                | 問題                             |
| ----------------------------------------------------------------------- | -------------------------------- |
| 構造改善目標を数値目標（-X%、≤N）に翻訳し baseline → 再計測パイプライン | Yes/No で足りる判断を歪める      |
| 評価表の行数・項目数を設計書の完成度の指標にする                        | 見かけの整備を本質の代わりにする |
| レビューで「定量目標がない」と指摘し、本質的でない数値目標を事後に捏造  | Goodhart の誘発                  |
| 変更容易性を直接数値目標化（修正箇所数、修正時間等）                    | 構造原則を迂回する設計を誘発     |
| 実行後に体感で判断すべき性能目標を AI が数値例で補完                    | 当事者不在の架空基準になる       |

---

## 5. 判断基準

設計判断・レビュー時に以下を自問する:

1. **この目標は Yes/No で判定可能か？** → Yes なら定量化しない
2. **この指標は本質的な品質特性（ユーザー価値・構造原則）に直接対応しているか？** → No なら採らない
3. **この数値は「計測できるように書く」ことが動機になっていないか？** → Yes なら書き直す
4. **この数値は当事者が確定したものか？** → No なら書かず、TBD として扱う

---

## 6. 定量目標そのものの要件/設計への振り分け

本文書は「そもそも構造品質を定量化しない」という排除ルールを扱う。

定量目標（応答時間・スループット等、運用で検証可能なもの）を
**要件定義書と設計書のどちらに書くか**という振り分け判定は本文書の範囲外である。

→ 振り分け判定は `${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md` §4.6 を参照。

## 7. 観点別の利用ガイド [MANDATORY] (REQ-004 FNC-401 P1/P2/P3 ルーティング)

本文書は P1 (ルール照合) / P2 (矛盾・齟齬) / P3 (不要な複雑化) のすべてから参照される。各観点で「どの規範を何の目的で見るか」を以下に固定する:

| 観点              | 本文書での扱い                                                                                                                                                                                                       |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P1 (ルール合致)   | §1 の全規範 (責務分割 / 境界設定 / 依存方向) + 重大度カタログ + 非機能要件カテゴリ網羅性節 を SoT として違反検出する。**漏れ・欠落・重複は P1 で扱う**                                                               |
| P2 (矛盾・齟齬)   | §1 の規範のうち **対象内部の相反記述** (同一論点に対する矛盾した記述) のみを検出する。**漏れ・欠落・重複は P2 観点外** (REQ-004 FNC-401 P2 定義「不足・欠落の検出は対象外」)。重大度カタログから severity を取得する |
| P3 (不要な複雑化) | §3.4 / §4 を SoT として、より少ない要素で同じ目的を達成できる代替案の存在を Yes/No で判定する                                                                                                                        |

`境界設定 (漏れ・重複の排除)` 規範は §1 に置かれるが、これは **P1 観点で違反検出するルール** である。P2 reviewer は本規範を「相反記述検出」目的のみで参照し、漏れ・欠落・重複の検出には用いない (REQ-004 FNC-401 / REQ-004 FNC-402 整合)。

本文書内の各規範を、違反時の重大度に対応付ける。レビュー時の severity 判定はこの表を SoT とする。criteria 側で重大度を判定してはならない (REQ-004 FNC-402)。

| 規範                                                                | 出典 (本文書内) | 違反時の重大度 | 理由                                                          |
| ------------------------------------------------------------------- | --------------- | -------------- | ------------------------------------------------------------- |
| 責務分割 (単一責務)                                                 | §1              | 🔴 critical    | 構造的設計原則の根本。崩れると SDD 全体が破綻する             |
| 境界設定 (漏れ・重複の排除)                                         | §1              | 🔴 critical    | 同上                                                          |
| 依存方向 (一方向・循環なし)                                         | §1              | 🔴 critical    | 循環依存はテスト不能・変更不能を生む                          |
| 品質特性の直接数値化禁止 (変更容易性 / 運用しやすさ / テスト容易性) | §2 / §3.2       | 🔴 critical    | Goodhart の罠。SDD の構造原則を直接破壊する                   |
| 構造品質の定量化禁止 (baseline → 再計測 → 達成率パイプライン)       | §3.1            | 🔴 critical    | Yes/No で判定可能なものを数値化で代替する典型的アンチパターン |
| flag 数 / クラス行数等の定量目標導入                                | §3.1            | 🔴 critical    | §3.1 と同根。直接的な数値目標化                               |
| 進捗管理 KPI・達成率・評価表の見栄え化                              | §3.3            | 🟡 major       | 動機が「計測できるように書く」になる二次的歪み                |
| 評価表の行数・項目数を完成度の指標にする                            | §3.3            | 🟡 major       | 同上                                                          |
| KPI 表・達成率表の作成自体を成果物化                                | §3.3            | 🔴 critical    | 主目的の倒錯。本来の要件・設計を空洞化する                    |
| 未確定定量値の捏造 (AI が一般論で補完する)                          | §3.4            | 🔴 critical    | 設計判断の根拠が空想になる                                    |
| 根拠のない目安値の判定基準化                                        | §3.4            | 🔴 critical    | 同上                                                          |
| 倒錯パターン: 構造改善を数値目標 (-X%/≤N) に翻訳                    | §4              | 🔴 critical    | §3.1 / §3.4 の典型化                                          |
| 倒錯パターン: 評価表の整備を本質の代替にする                        | §4              | 🟡 major       | §3.3 の典型化                                                 |
| 倒錯パターン: レビューで定量目標欠如指摘 → 事後捏造                 | §4              | 🔴 critical    | §3.4 の典型化。レビュー側の Goodhart 誘発                     |
| 倒錯パターン: 変更容易性を直接数値目標化                            | §4              | 🔴 critical    | §3.2 の典型化                                                 |
| 倒錯パターン: 実行後体感値を AI が数値例で補完                      | §4              | 🔴 critical    | §3.4 の典型化                                                 |

## 9. グレーゾーン許容範囲 [MANDATORY]

判定が割れやすい論点について、許容範囲を断定形で明示する。レビュー時の false positive 警告は principles 側で執筆者にも見える形で持つ。criteria 側に「false positive 注意」を書いてはならない (REQ-004 FNC-402)。

| 論点                     | 許容範囲 (✅ 許容)                                                                                                           | 不許容 (❌)                                                         |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| 数値化 (機能的目標)      | 性能指標 (応答時間 / スループット) / 可用性指標 (SLA / SLO) / セキュリティ閾値 / ビジネス KPI (要件側で当事者が確定したもの) | 構造品質 (保守性スコア / 凝集度スコア / 結合度スコア)               |
| 定量目標の置き場         | 要件定義書 = ユーザー要求・契約・実測由来の確定値 / 設計書 = 達成を狙う方針                                                  | 設計書に「本設計により X を達成する」等の保証表現                   |
| 未確定値の扱い           | 要件定義書の TBD として明示し、当事者判断を待つ                                                                              | AI が一般論や類似例から推測した値を判定基準・合格基準に採用すること |
| 評価軸の設計書記載       | 「定性的な確認観点リスト」「Yes/No チェックリスト」                                                                          | 「N 項目をクリアすれば合格」等のスコア化                            |
| パフォーマンス計測の記述 | 計測対象・計測方法・閾値出典 (要件由来) を明示した記述                                                                       | 計測値を持たない目標数値 (推定値) を判定基準化                      |
| カバレッジ目標           | テスト戦略上の参考値として記載 (達成義務化しない)                                                                            | カバレッジ率を品質指標の主軸として KPI 化                           |

## 10. 非機能要件カテゴリ網羅性

要件定義書レビュー時、以下のカテゴリのいずれかが文書内に一切記載されていない場合の重大度は、**対象システムの性質によってスコープ付け** する。Issue #68 (レビューを重くしない) の思想に整合させ、軽量レビューでの false positive を抑える。

| カテゴリ     | 判定                                                         |
| ------------ | ------------------------------------------------------------ |
| 性能         | 応答時間 / スループット / 同時接続数 等の言及があるか        |
| 可用性       | 稼働率 / 障害時の振る舞い / フェイルオーバ要件の言及があるか |
| セキュリティ | 認証・認可 / データ保護 / 監査ログ等の言及があるか           |
| 運用性       | 監視 / ログ / バックアップ / 復旧の言及があるか              |

### 10.1 重大度スコープ [MANDATORY]

| 対象システムの性質                                                                                  | 未記載時の重大度 |
| --------------------------------------------------------------------------------------------------- | ---------------- |
| 本番運用 / 外部公開 (顧客向けサービス・公開 API) / データ保護 (個人情報・機密情報) のいずれかを伴う | 🔴 critical      |
| 上記いずれにも該当しない (プロトタイプ / 内部 CLI ツール / 文書-only feature / 検証用スクリプト 等) | 🟡 major         |

対象システムの性質は、要件定義書の冒頭 (前提・スコープ等) に明示することを推奨する。明示がない場合、reviewer は target_files の内容 + 関連要件 ID から推定し、判定根拠を finding に記録する。

### 10.2 除外理由の明示

critical 適用範囲であっても、当該カテゴリと無関係である理由が明文化されている場合は許容 (例: 「読み取り専用の社内ツールのため可用性 SLA 不要」を明記)。major 適用範囲では、除外理由の都度明記は不要 (前段のスコープ判定で吸収する)。
