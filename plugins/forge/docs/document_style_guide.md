# 文書スタイル指針

このプラグインが生成・更新する文書（要件定義書・設計書・計画書）の汎用記述スタイル。
プロジェクト固有のスタイル指針がある場合はそちらを優先する。

---

## 1. タグの使い方

### 1.1 種類と配置

- 種類: `[MANDATORY]` / `[CRITICAL]` / `[IMPORTANT]`（3 種）
- 配置: 見出し行の末尾、タグ前のスペースは 1 個、タグ後に余分な空白を入れない

**正しい例**:

```text
## データ削除手順 [MANDATORY]
```

**誤った例**:

```text
## データ削除手順

下記を読むこと [MANDATORY] ← タグは見出しに付ける

## データ削除手順  [MANDATORY] ← スペース 2 個

## データ削除手順 [MANDATORY]  ← 末尾に余分な空白
```

### 1.2 信号価値の維持

**タグは「希少だから効く」信号**。乱用すれば AI も人間も注意を払わなくなる。
**「すべてが MANDATORY」=「どれも MANDATORY ではない」** と等価になる。

`[MANDATORY]` を付けてよい基準（いずれか 1 つを満たすときのみ）:

| # | 基準                             | 例                                                           |
| - | -------------------------------- | ------------------------------------------------------------ |
| A | 違反で不可逆な損害が発生する     | 本番データ削除、main への force push、秘密情報の commit      |
| B | アーキテクチャ前提が崩壊する     | Clean Architecture のレイヤー逆参照、Actor 境界違反          |
| C | 法的・契約的に必須               | ライセンス表示、個人情報取扱い                               |
| D | 取り返しがつかない判断で人間確認 | プロジェクトファイル変更、ビルド設定変更、ブランチ削除       |
| E | AI が高確率で間違える既知の罠    | 過去にインシデント化した規約違反、繰り返されるリグレッション |

**付けてはいけない場面**:

- 章節見出しすべてに飾りとして付与
- 「推奨」「望ましい」「ベストプラクティス」程度の事項
- 体裁・命名・プロセス整理ルール
- 上位文書で既に宣言済みの前提の再強調

**格下げと頻度**: 迷ったら `[MANDATORY]` → `[CRITICAL]` → `[IMPORTANT]` → タグなし の順で格下げ。
1 文書あたり `[MANDATORY]` は **0〜3 箇所**を目安。4 箇所以上必要に感じたら基準 A〜E に該当するか再確認する。

---

## 2. 見出しレベル

`#` 〜 `####` のみ使用する。`#####` 以降は使わない（4 階層を超えるなら章立てを再構造化する）。

| レベル | 用途                         |
| ------ | ---------------------------- |
| `#`    | ファイルタイトル（1 個のみ） |
| `##`   | 主要セクション               |
| `###`  | サブセクション               |
| `####` | 詳細項目                     |

---

## 3. 強調・表記

### 3.1 強調フレーズ

説明文中で重要事項を強調するときは英語フレーズを併記すると AI への伝達が確実:

- `NEVER` / `NEVER skip.` / `MUST` / `DO NOT` / `ALWAYS`

### 3.2 長音ルール

JIS Z 8301 準拠。語尾の長音は省略する:

| 正（長音省略） | 誤（長音あり） |
| -------------- | -------------- |
| ディレクトリ   | ディレクトリー |
| カテゴリ       | カテゴリー     |
| メモリ         | メモリー       |
| エントリ       | エントリー     |
| リポジトリ     | リポジトリー   |
| プロパティ     | プロパティー   |

固有名詞・フレームワーク公式名称はそのまま使用する。

### 3.3 強調記法の使い分け

- `**bold**`: 重要語句の強調
- `*italic*`: 用語の初出や参照名
- `` `code` ``: ファイル名・パス・コード片・コマンド・識別子

---

## 4. 構造化要素

### 4.1 コードブロック

言語指定を必ず付ける（`bash` / `yaml` / `python` / `markdown` 等）。構造説明のみで言語が無い場合は `text` を指定。

### 4.2 リスト

- 順序なし `-`: 列挙
- 順序あり `1.`: 手順・優先順位
- 入れ子は 2 階層まで（それ以上はサブセクション化する）

### 4.3 表

- 列数は 5 列以内
- 長文セルはリストや別セクションに分離
- 表セル内ではコードブロックや複数段落を使わない

---

## 5. 文書参照記法 [MANDATORY]

他文書を参照するときの記法。違反すると AI が他文書を読みに行けなくなり致命的（基準 A・E）。

### 5.1 パス表記

- フルパスで記述する: `${CLAUDE_PLUGIN_ROOT}/docs/design_format.md`
- 同一段落内で複数回参照する場合のみ 2 回目以降をファイル名のみに省略可
- 表セル内では省略しない（セル単位で完結させる）

### 5.2 例

```markdown
設計書のテンプレートは `${CLAUDE_PLUGIN_ROOT}/docs/design_format.md` を参照する。
`design_format.md` 内のメタデータセクションを必ず含めること。
```

---

## 6. 抽象例の命名

ルール文書や指針文書に登場する例文・コード例では、特定プロジェクトを連想させる固有名を使わない:

- 推奨: `Foo` / `Bar` / `Hoge` / `Fuga` + 型サフィックス（`FooService`、`BarRepository`、`HogeViewModel` 等）
- 非推奨: 「連絡先」「ユーザー管理」「商品カタログ」等の具体ドメイン

OS フレームワーク名（SwiftUI、UIKit、Combine 等）は固有名でも一般語として許容する。

## 7. 改定履歴

### 7.1 セクションの設置

文書末尾に `## 改定履歴` セクションを設け、日付は `YYYY-MM-DD` 形式で記載する。詳細な運用規約（バージョン体系、記載粒度、更新タイミング）は対応する `*_principles_spec.md` を参照する。

**初版時はセクションを設けない**。最初の意味ある改定が発生した時点で初めて追加する。

### 7.2 記載してはならないもの

改定履歴は **「公開・merge 後の意味ある変更」を後から追うための記録**。以下は記載しない:

- **初版作成中の試行錯誤** — 作って間違って直した経過は作業ログにすぎない
- **typo 修正・フォーマット整形のみ** — 内容に影響しない
- **「初版作成」「文書作成」などの自明な行** — 履歴の最初に置くだけで意味がなく、ノイズになる
- **同日内の連続修正** — 1 件に統合する

> 履歴は「希少だから読まれる」。乱用するとタグ (§1.2) と同じく信号価値を失う。\
> 全部消したくなる履歴は、書かないほうがましだった履歴である。

---

## 8. 関連文書セクションを置かない

文書末尾に「関連文書」「参考文献」「See also」等の **リスト形式の節を置かない**。理由:

- **必読文書** (前提として読まないと意味が通らない) は **冒頭または本文中** で触れる。リスト末尾に置いても読まれない
- **前提ではないが関連がある文書** は `/query-rules` / `/query-specs` で探すべきもの。明示リンクは陳腐化するうえ、検索で代替できる
- リスト形式の関連文書節は **更新されないまま腐敗** しがち (リンク切れ・SoT 移動の追従漏れ)

### 8.1 例外

以下に限り、関連リンクを置いてもよい:

- **外部 URL** (query で取得できないもの): 公式ドキュメント・RFC・外部仕様等
- **本文中で参照済みのファイル一覧** を 1 箇所にまとめて再掲したい場合 (ただし本文中の参照を省略するのと引き換え)

> 関連リンクが必要に感じたら、まず「query で見つかるか」を確認し、見つかるなら書かない。
