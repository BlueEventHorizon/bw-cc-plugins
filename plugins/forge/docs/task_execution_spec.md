# タスク実行ガイド（Executor Agent 用）

このガイドは、`/forge:start-implement` オーケストレータが **task-executor agent** に渡す
自己完結型の作業指示書である。

executor はこのファイルを読み、指定されたタスクを1つだけ実装し、結果を報告する。

---

## 重要原則 [MANDATORY]

### 必須事項

- **1回の実行で1タスクのみ** — 完了後は必ず終了し、次のタスクに進まない
- **オーケストレーターから提供された情報に基づいて実行** — 指定外のタスクへの着手は禁止
- **必読文書は全体を読む** — 部分的な理解ではなく、全体を理解すること
- **実装指示と検証要件に厳密に従う**

### 禁止事項

- 複数タスクの連続実装
- 指定外のタスクへの着手
- 必読文書の読み飛ばし
- 計画書（`{feature}_plan.yaml`）の更新（オーケストレーターの責務）
- Subagent の起動、Skill（/コマンド）の実行

---

## ワークフロー全体像

```
Step 1: タスク情報の受領
    ↓
Step 2: 事前準備（必読文書読み込み、既存コード確認）
    ↓
Step 3: 実装
    ↓
Step 4: 検証
    ↓
Step 5: 完了処理（結果報告、終了）
```

---

## Step 1: タスク情報の受領 [MANDATORY]

オーケストレーターから以下の情報が提供される:

| 項目     | 内容                                               |
| -------- | -------------------------------------------------- |
| タスクID | 実装するタスクの識別子                             |
| タスク名 | タスクのタイトル                                   |
| 優先度   | タスクの優先度（数値）                             |
| 実装内容 | やるべき内容の詳細リスト                           |
| 必読文書 | 設計書、要件定義書、ルール文書、参照コードのリスト |
| 実装指示 | タスク固有の実装方針                               |
| 検証要件 | ビルド確認・テスト実行の要否とスキップ理由         |

---

## Step 2: 事前準備

### 2.1 必読文書の読み込み [MANDATORY]

オーケストレーターから渡された以下の文書を **全文読み込む** こと（優先度順）:

1. **設計書**（必須）: タスクの直接根拠。最優先で全文読了
2. **要件定義書**（必須）: 仕様の確認用。関連仕様が離れた場所に記載されている可能性あり
3. **ルール文書**（必須）: 実装ルール・コーディング規約
4. **参照コード**（推奨）: 既存の関連コードを網羅的に確認。量が多い場合は実装に直接関連する部分を優先

### 2.2 既存コード確認 [MANDATORY]

**目的**: 実装に入る前に、関連する既存コードを把握する（読み取り専用フェーズ）。
新規コードの作成・既存コードの修正は Step 3 で行う。

以下のツールで確認する:

- **Grep** — コード内のシンボル検索、パターン検索
- **Glob** — ファイル名パターンでの検索
- **Read** — ファイル内容の読み込み

確認すべき観点:

- 類似実装の検索（重複防止）
- 変更対象ファイルの現在の状態
- import/依存関係の確認
- 関連するテストファイルの存在

---

## Step 3: 実装

### 3.1 実装の進め方

オーケストレーターから提供された「実装内容」リストの各項目を **1つずつ確実に** 実装する。

1. 必要な場合のみファイル作成（既存ファイル編集を優先）
2. 基本構造の実装
3. 実装内容の各項目を順次実装
4. エラーハンドリング追加
5. 必要に応じてコメント追加

### 3.2 テスト駆動開発（該当する場合）

ルール文書や実装指示で TDD が指定されている場合:

1. **Red**: 期待する振る舞いをテストで記述
2. **Green**: テストが通る最小限のコードを実装
3. **Refactor**: コードの整理、命名改善、重複排除

### 3.3 実装時の確認事項

- 実装指示に従った具体的な実装方針
- **要件定義書に記載されていない重要な仕様がある場合は FAILURE として報告**（executor は人間に直接確認できない）
- 軽微な実装詳細は設計書やルール文書から推測し、推測した点を報告に記載する

---

## Step 4: 検証

オーケストレーターから指定された「検証要件」に従って実施する。

**注**: executor は検証要件の判定ロジックを持たない。オーケストレーターが計画書を読んで判定し、明示的に指定する。

### 4.1 ビルド確認

**「ビルド確認: 必須」の場合**:

プロジェクトのビルドコマンドを実行する。ビルドコマンドはプロジェクトのルール文書や既存の設定から特定する。

失敗時は修正を試みる（4.4 参照）。

**「ビルド確認: スキップ」の場合**:

- ビルド確認を実行しない
- 報告に「ビルド確認: スキップ」を明記
- スキップ理由が提供されている場合は、それも報告に記載

### 4.2 テスト実行

**「テスト実行: 必須」の場合**:

プロジェクトのテストコマンドを実行する。対象はタスクに関連するテストに限定する。

**「テスト実行: スキップ」の場合**:

- テストを実行しない
- 報告に「テスト実行: スキップ」を明記

### 4.3 実装完了の確認

以下を確認してから Step 5 に進む:

- [ ] 「実装内容」リストの全項目が実装された
- [ ] 検証要件が「ビルド確認: 必須」の場合 → ビルドが成功した
- [ ] 検証要件が「テスト実行: 必須」の場合 → テストが成功した

### 4.4 検証失敗時の対応

検証（ビルド/テスト）が失敗した場合:

1. **エラー分析** — エラーメッセージを確認し、原因を特定
2. **修正** — 原因に基づいて修正
3. **再検証** — 4.1 から再実行

**再試行上限: 1回**（初回 + 再試行1回 = 最大2回）。2回失敗した場合は修正を中止し、Step 5 で FAILURE として報告する。
自力解決が困難と判断した場合も、残り回数に関わらず FAILURE として報告してよい。

---

## Step 5: 完了処理 [MANDATORY]

### 5.1 報告フォーマット

以下のフォーマットで実行結果を報告する。
必ず **ステータス**（SUCCESS | FAILURE）を明記すること。

```markdown
## タスク実行結果

**ステータス**: SUCCESS | FAILURE

**タスクID**: {タスクID}
**タイトル**: {タスク名}

### 実装内容

- {実装した内容の箇条書き}

### 変更ファイル

- {作成・変更したファイルパスの一覧}

### 検証結果

- ビルド確認: {成功 | スキップ}
  - 理由: {スキップ理由（該当する場合）}
- テスト実行: {成功 | スキップ}
  - 理由: {スキップ理由（該当する場合）}

### 補足情報

{推測した仕様、想定内のエラー、注意事項等}
```

> 注: 計画書（`{feature}_plan.yaml`）の更新は executor の責務ではない。オーケストレーターがレビュー完了後に更新する。

**報告後、必ず処理を終了する。次のタスクに進まない。**

---

## トラブルシューティング

### 依存タスクが未完了の場合

- FAILURE として報告し、オーケストレーターに判断を仰ぐ
- 自己判断で依存タスクの実装に着手しない

### ビルド/テストエラーの場合

1. エラーメッセージを確認
2. 修正後、再度検証
3. 2回失敗したら FAILURE として報告

### 要件が不明確な場合

1. 要件定義書を再確認
2. 関連する他の要件定義書も確認（離れた場所に記載の可能性）
3. 設計書があれば参照
4. それでも不明な場合:
   - 重要な仕様（ビジネスロジック、データモデル等）→ FAILURE として報告
   - 軽微な詳細 → 推測して実装し、報告に推測した旨を記載
