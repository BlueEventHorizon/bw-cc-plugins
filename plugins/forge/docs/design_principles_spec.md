# 設計書 作成原則

プロジェクト固有のルールがない場合に参照する汎用原則。
プロジェクト固有のルールがある場合はそちらを優先すること。

文書記述スタイル（タグ・見出し・参照記法）は `${CLAUDE_PLUGIN_ROOT}/docs/document_style_guide.md` を参照。本文書は設計書固有の構成要素・判断基準を扱う。

---

## 設計書の基本方針

**設計書は「How（どう作るか）」を記述する。「What（何を作るか）」は要件定義書に記載する。**

→ 境界の詳細: `${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md`

---

## 必須構成要素 [MANDATORY]

設計書には以下の要素を**必ず**含めること:

| 要素                           | 形式                       |
| ------------------------------ | -------------------------- |
| モジュールリスト               | 表形式（責務と依存を記載） |
| コンポーネント図またはクラス図 | Mermaid 推奨               |
| ユースケース一覧               | 表形式                     |
| シーケンス図                   | Mermaid 推奨               |

セクション名・番号は設計内容に応じて柔軟に調整してよい。

---

## 記載すべき内容 / 記載してはいけない内容

### 記載すべき内容

- ✅ 実装方法（クラス名、メソッド名、デザインパターン）
- ✅ 技術的詳細（フレームワーク・ライブラリの選定と**採用理由**）
- ✅ 処理フロー・アルゴリズム
- ✅ データフロー（コンポーネント間のデータの流れ）
- ✅ シーケンス図、クラス図
- ✅ テストケース設計（正常系/異常系）
- ✅ 使用する既存コンポーネント（具体的なファイルパス）

### 記載してはいけない内容

- ❌ ソースコード（説明用の小規模例・5〜10行程度は許容）
- ❌ 技術選択の理由を記載しないままフレームワークを指定
- ❌ 要件にない機能の追加
- ❌ 「本設計により X ms を切る」等、定量目標の達成を保証する記述
- ❌ 実行後に人間が体感して決める値を、AI が数値例から推測した設計目標

---

## 定量目標の扱い [MANDATORY]

設計書は、定量目標の達成を保証する文書ではない。
要件で確定した定量目標がある場合でも、設計書に書くのは達成を狙うための構造・方針・代替案までとする。

| 内容                                               | 扱い                             |
| -------------------------------------------------- | -------------------------------- |
| キャッシュ層、debounce、非同期化、冗長化などの方針 | ✅ 記載してよい                  |
| 「本設計により X ms を切る」等の保証表現           | ❌ 記載しない                    |
| 未確定値を AI が推測した設計目標                   | ❌ 記載しない                    |
| 負荷試験・プロファイリング等の実地検証結果         | 設計書ではなく検証結果として扱う |

定量目標の最終的な達成可否は、負荷試験・プロファイリング・本番計測・人間の受け入れ判断など、実行後の検証で確定する。

構造品質（責務分割・境界・依存方向）の数値化禁止、および未確定定量値の捏造禁止は `${CLAUDE_PLUGIN_ROOT}/docs/spec_priorities_spec.md` に従う。
要件定義書との振り分けは `${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md` §4.6 に従う。

## 判断基準

**迷ったら「実装者がこれで実装できるか？」で判断**

- **Yes** → 設計書に記載
- **No** → より詳細化が必要、または要件定義書の領域

**具体例**:

| シナリオ                                            | 判断      | 理由                     |
| --------------------------------------------------- | --------- | ------------------------ |
| 「検索処理を実装する」                              | ❌ 不十分 | どのように検索するか不明 |
| 「SearchService.search() でデータをフィルタリング」 | ✅ 十分   | 実装方法が明確           |
| 「300ms debounce で検索 API を呼び出す」            | ✅ 十分   | 技術的詳細が明確         |

## 階層構造ガイドライン

設計規模が大きい場合は、1つのファイルにまとめず階層で分割する:

```
トップレベル設計書（全体アーキテクチャ）
├── 子設計書（モジュールA詳細）
├── 子設計書（モジュールB詳細）
└── 子設計書（画面/機能詳細）
```

- **分割の目安**: 単一ファイルが500行を超える場合は分割を検討（トップレベルは例外）
- **子設計書のメタデータ**に親設計書 ID を記載すること
- **トップレベル設計書**に子設計書の一覧を記載すること

## ファイル命名規則

```
{設計ID}_{対象名}_design.md
```

- 設計 ID をファイル名先頭に付与
- スネークケース使用
- 末尾に `_design.md` を付与
- 英語で記載

例: `DES-001_user_list_design.md`

## 図表使用ガイドライン

設計書での図表は Mermaid 形式を推奨する。

| 図種別         | 用途                         | Mermaid 構文      |
| -------------- | ---------------------------- | ----------------- |
| フローチャート | アーキテクチャ図・処理フロー | `flowchart TB`    |
| クラス図       | モジュール構造・型関係       | `classDiagram`    |
| シーケンス図   | ユースケース・データフロー   | `sequenceDiagram` |

**注意**: flowchart で小文字の `end` は使用禁止（`End` または `END` を使用）

Mermaid 構文リファレンス:

- [Flowchart](https://mermaid.js.org/syntax/flowchart.html)
- [Class Diagram](https://mermaid.js.org/syntax/classDiagram.html)
- [Sequence Diagram](https://mermaid.js.org/syntax/sequenceDiagram.html)

## 改定履歴セクション [MANDATORY]

- セクション名は **`## 改定履歴`** を使用（「変更履歴」「承認と改訂履歴」等は使用しない）
- 設計書の**最後のセクション**として配置
- 日付は `YYYY-MM-DD` 形式

## 設計書の保守 [MANDATORY]

実装が設計書の記述と乖離した場合（バグ修正・リファクタリング・設計判断の変更）、実装を正とみなせるなら設計書を更新する。設計書が古いまま放置されると、次の実装者（AI 含む）が誤った前提で作業する。

### 更新が必要なケース

| 乖離の種類                     | 対応                                             |
| ------------------------------ | ------------------------------------------------ |
| API シグネチャ・戻り値型の変更 | 設計書のインターフェース記述を修正               |
| 設計判断の変更（選択肢 A → B） | ADR を作成または既存 ADR を更新                  |
| パラメータ・定数・閾値の変更   | 設計書の該当セクションを修正                     |
| 新機能追加（設計書に記載なし） | 設計書にセクションを追加、または新規設計書を作成 |

### 更新時のルール

- 改定履歴に変更理由を追記する
- 設計意図（なぜそう変えたか）を記述する。コードの転記は不要
- ADR（Architecture Decision Record）は設計書と同じディレクトリに配置する

### 更新不要なケース

- 内部リファクタリングで外部インターフェースが不変の場合
- コメント・ログメッセージのみの変更
- テストコードのみの変更

## よくある失敗パターン

| パターン                                                   | 問題               | 対策                                     |
| ---------------------------------------------------------- | ------------------ | ---------------------------------------- |
| 既存資産を1パターンの検索で諦め重複実装                    | 無駄な実装・不整合 | 複数キーワード・複数ツールで網羅的に検索 |
| プロジェクトの定数・デザイントークンを無視してハードコード | 不整合             | 定数定義を必ず参照                       |
| 同種機能の既存実装を確認せず独自設計                       | 重複・不整合       | 既存の類似パターンを必ず確認             |
| 設計判断の根拠を記載しない                                 | 保守困難           | 技術選択の理由・代替案を文書化           |
| 実装詳細（コード）を多量に記載                             | 設計書と実装の乖離 | 設計意図を記述し、詳細はコードで表現     |
| 実装後に設計書を更新しない                                 | 次回実装時に誤解   | 上記「設計書の保守」に従い即座に更新     |

## アーキテクチャ依存方向規範 [MANDATORY] (Appendix A.7 由来)

採用したアーキテクチャパターンの根本原則は設計書レビューで照合する。

### 規範

1. 採用したアーキテクチャパターン (レイヤード / クリーンアーキ / ヘキサゴナル等) の依存方向ルールを設計書に明示する
2. レイヤードでは下位層が上位層に依存してはならない
3. クリーンアーキでは Domain → Infrastructure を直接参照してはならない
4. 循環依存 (A → B → C → A 等) を作ってはならない

### グレーゾーン許容範囲

| 論点                                                   | ✅ 許容                                                | ❌ 不許容                                      |
| ------------------------------------------------------ | ------------------------------------------------------ | ---------------------------------------------- |
| 意図的な依存方向例外                                   | 設計書内に「この例外を採用する理由」が明文化されている | 例外理由が記載されないまま依存方向ルールを破る |
| ファサード/アダプタによる依存反転                      | 明示的なインターフェース定義を伴うもの                 | 「便利だから」の理由なき直接参照               |
| 単方向データバインディング由来の View → ViewModel 参照 | フレームワーク慣習として定着しているもの               | フレームワーク文脈外での同パターン             |

## 責務分割・凝集度規範 [MANDATORY] (Appendix A.8 由来)

### 規範

1. 各モジュール・コンポーネントは単一の責務を持つ
2. 1 コンポーネントが複数の異なる責務 (例: ユーザ認証 + レポート生成) を持つ設計は禁止
3. インターフェースを介さない直接的な内部構造への依存 (密結合) を避ける
4. 現在の利用箇所が 1 つしかない汎用フレームワーク的構造を作らない (YAGNI)

### グレーゾーン許容範囲

| 論点                                      | ✅ 許容                                         | ❌ 不許容                                                  |
| ----------------------------------------- | ----------------------------------------------- | ---------------------------------------------------------- |
| ファサードパターン (意図的な複数操作集約) | 設計書に集約意図と境界が明示されているもの      | 単に肥大化した結果としての多責務クラス                     |
| ユーティリティクラス                      | 純粋関数の集合で副作用がないもの                | 状態を持ち複数の責務 (DB アクセス + ロジック等) を持つもの |
| 抽象化の早期導入                          | 既知の 3 箇所以上で使われるもの (Rule of Three) | 1 箇所のみで使われる「将来のための」汎用化                 |

---

## 可用性・SPOF 規範 [MANDATORY] (Appendix A.9 由来)

### 規範

1. 単一障害点 (SPOF) を持つ構成では、冗長化または代替経路を設計書に明示する
2. 外部 API 連携にはリトライ戦略 / タイムアウト / フォールバックを設計レベルで定義する
3. 異常系のデータフローと障害時の振る舞いを設計書に含める

### グレーゾーン許容範囲

| 論点                         | ✅ 許容                                                                               | ❌ 不許容                          |
| ---------------------------- | ------------------------------------------------------------------------------------- | ---------------------------------- |
| シングル DB インスタンス依存 | 開発環境 / プロトタイプ / ローカルツール / 可用性要件が低いことが要件で明示されている | 本番運用前提なのに冗長化方針未記載 |
| リトライ戦略の省略           | 冪等性が保証されない処理で意図的にリトライしない判断 (記載必須)                       | リトライ要否を未検討・未記載       |

## 重大度カタログ [MANDATORY]

本文書内の各規範を、違反時の重大度に対応付ける。

### 「記載してはいけない内容」(本文書既存)

| 違反パターン                                       | 違反時の重大度 | 理由                                                   |
| -------------------------------------------------- | -------------- | ------------------------------------------------------ |
| 設計書にソースコードを多量に記載 (5〜10行を超える) | 🟡 major       | 設計書と実装の乖離源。MAJOR (≠ 即修正でないが品質問題) |
| 技術選択の理由を記載しないままフレームワーク指定   | 🔴 critical    | 設計判断の根拠喪失                                     |
| 要件にない機能の追加                               | 🔴 critical    | スコープクリープ。設計駆動の破壊                       |
| 「本設計により X を達成する」等の保証表現          | 🔴 critical    | spec_design_boundary §4.6 違反                         |
| AI が数値例から推測した設計目標                    | 🔴 critical    | spec_priorities §3.4 違反                              |

### 必須構成要素 (本文書既存)

各構成要素は「該当する場合に限り」必須とする。設計種別 (バッチ処理 / 内部 API / 文書-only feature 等) によっては不要な要素があり、画一的な必須化は false positive を生むため (Issue #68 軽量レビュー思想整合)。

| 違反パターン                                                                         | 違反時の重大度 | 理由                                                            |
| ------------------------------------------------------------------------------------ | -------------- | --------------------------------------------------------------- |
| モジュールリスト欠如 (複数モジュール構成の場合)                                      | 🟡 major       | 設計書の構造把握困難。単一モジュール構成の場合は省略可          |
| コンポーネント図/クラス図欠如 (コンポーネント関係・クラス継承構造が複数存在する場合) | 🟡 major       | 構造関係が単純 (1〜2 コンポーネントの直列構成等) な場合は省略可 |
| ユースケース一覧欠如 (ユーザ操作・外部相互作用を伴う場合)                            | 🟡 major       | バッチ処理・内部 API のみ・文書-only feature 等の場合は省略可   |
| シーケンス図欠如 (該当する処理がある場合)                                            | 🟢 minor       | 処理がシンプルな場合は省略可                                    |

スコープ条件 (例: 「複数モジュール構成の場合」) は、reviewer が target_files の内容から判定する。判定根拠を finding に記録すること。

### よくある失敗パターン (本文書既存)

| 失敗パターン                                           | 違反時の重大度 | 理由                       |
| ------------------------------------------------------ | -------------- | -------------------------- |
| 既存資産を 1 パターン検索で諦め重複実装                | 🔴 critical    | 重複・不整合の温床         |
| プロジェクトの定数・デザイントークン無視のハードコード | 🟡 major       | 一貫性・保守性低下         |
| 同種機能の既存実装未確認 → 独自設計                    | 🔴 critical    | 重複・不整合の温床         |
| 設計判断の根拠未記載                                   | 🟡 major       | 保守時の意思決定再現性低下 |
| 実装後に設計書未更新                                   | 🟡 major       | 次回実装時の誤解源         |

### アーキテクチャ・責務分割・可用性 (forge-review feature 統合で追加)

| 違反パターン                                                 | 違反時の重大度 |
| ------------------------------------------------------------ | -------------- |
| 採用アーキテクチャパターンの依存方向違反 (Domain → Infra 等) | 🔴 critical    |
| 循環依存                                                     | 🔴 critical    |
| 1 コンポーネントの多責務化 (例: 認証 + レポート生成)         | 🟡 major       |
| YAGNI 違反 (1 箇所のみ利用の汎用化)                          | 🟢 minor       |
| 本番運用前提なのに SPOF を未冗長化・未記載                   | 🔴 critical    |
| 外部 API 連携のリトライ戦略・タイムアウト未記載              | 🟡 major       |
| 異常系データフロー・障害時挙動未記載                         | 🟡 major       |

### 追加 feature 用 frontmatter (Issue #118)

判定（追加 feature か否か）は [`additive_development_spec.md`](additive_development_spec.md) §1（適用条件 / 対象外）に従う。main 初期立ち上げ・既存文書の追記更新は対象外（false positive 防止）。frontmatter の定義は [`design_format.md`](design_format.md)「追加 feature 用 frontmatter」節を参照。

| 違反パターン                                                            | 違反時の重大度 | 理由                                                                         |
| ----------------------------------------------------------------------- | -------------- | ---------------------------------------------------------------------------- |
| 追加 feature 設計書に `type: temporary-feature-design` frontmatter 欠如 | 🟡 major       | 旧設計書との優先関係・merge 予定が宣言されず、生成経路を問わず取りこぼされる |
