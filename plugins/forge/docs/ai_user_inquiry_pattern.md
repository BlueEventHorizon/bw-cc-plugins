# AI からユーザーへの複数案提示テンプレ

設計判断・方針議論・たたき台レビュー等で、AI が複数案を対等にユーザーに提示して判断を仰ぐ場面のテンプレ。

## 適用範囲

**適用する場面**:

- 設計レビューで A 案 / B 案など複数の対等な選択肢を提示する
- 実装方針や命名で複数候補がある
- 仕様変更の影響範囲が複数の選択肢を生む

**適用しない場面**:

- レビュー結果の指摘提示 (`forge:present-findings` を使う)
- 説明 / 教える場面 (本テンプレを当てず通常の説明で済ます)
- 既に判断が定まり確認だけの場面 (短い AskUserQuestion 1 つで十分)

---

## 基本構造 (3 ブロック + 選択)

各論点を以下の 4 ブロックで提示する。これより細かく分けない。

```
## 論点: <短い問い>

### 選択肢の対象
- A: <A は何か> (1 行)
- B: <B は何か> (1 行)

### 論点
A と B の違いは <何>。どちらを採るかで <何が変わる>。 (2〜3 行)

### 推奨と根拠
<A か B かを選ぶ>。理由は <1 行>。

---
(ここで AskUserQuestion を呼ぶ)
```

---

## 各ブロックの書き方

### 選択肢の対象 (1 行ずつ)

選択肢が **「何」** なのかを 1 行で書く。比較が始まる前に、選択肢そのものの定義をユーザーに理解させる。

- 関数なら **シグネチャと役割** を 1 行: `exists(key) -> bool: Entity が存在するかを真偽値で返す`
- 設計案なら **採否で起きる変化** を 1 行: `現契約維持: 旧 port を残す、内部実装だけ差し替え`
- 命名なら **その名前の含意** を 1 行: `transition_status: 状態遷移であることを明示`

**やってはいけない**:

- 1 行に収まらない長い説明をここに書く (それは「論点」ブロックの仕事)
- 「A 案」「B 案」とだけ書いて内容を省く

### 論点

A と B の **本質的な違い** と、**どちらを採るかで何が変わるか** を 2〜3 行で書く。

- 表面的な差ではなく、選択の意味を書く
- 技術的トレードオフがあれば 1 行で触れる
- 「ユーザーが何を基準に選ぶべきか」を暗に示す

### 推奨と根拠

**1 つだけ** 推奨を出す。理由は 1 行。複数の根拠を並べない (並べたい場合は質問へ降ろす)。

### 選択 (AskUserQuestion)

`AskUserQuestion` を呼ぶ。選択肢は **2〜3 個** に絞る:

- A 案 (推奨)
- B 案
- (任意) 保留 / 詳細を聞く

4 個以上の選択肢は提示しない。多すぎると判断負荷が逆に増える。

---

## やってはいけないこと (アンチパターン)

### AP1. 複数論点の連続提示

論点を 4 つ並べて「順番に AskUserQuestion で問う」をしない。1 論点提示 → ユーザーが応答 → 次の論点、の **interactive な往復** にする。

global CLAUDE.md の「Discussion Mode」原則:

> 列挙した論点を連続 AskUserQuestion で次々問わない。\
> 細部を詰めるかどうか・どの順で詰めるかは、ユーザーが明示してから動く。

### AP2. 5 章以上の章立て

「該当箇所 / 現状の判断 / SQL 比較 / 採用根拠 3 つ / 代替案 / リスク / 推奨」のような細分化はしない。基本構造の 4 ブロックに収める。詳細が必要なら **ユーザーが質問してから** 出す。

### AP3. 推奨を出さない・複数推奨

推奨は 1 つだけ。「A も B も良い」は判断を投げているだけ。「どちらでも良い」と書きたい時は、それ自体が答えなので **AskUserQuestion を呼ばずに** 短く済ます。

### AP4. 選択肢の対象を説明せず比較に入る

「A 案 (削除) と B 案 (残す) のどちらにしますか?」だけだと、A / B が **何の** 削除 / 何を残すのかが伝わらない。常に「選択肢の対象」ブロックで 1 行説明してから比較に入る。

### AP5. 過剰な技術詳細

SQL / コード片 / 表を 3 つ以上並べない。1 比較表 か 1 コード片までに絞る。詳細はユーザーから求められた時に出す。

## テンプレ (コピペ用)

```markdown
## 論点 N/M: <短い問い>

### 選択肢の対象

- A: <A は何か> (1 行)
- B: <B は何か> (1 行)

### 論点

A と B の違いは <何>。どちらを採るかで <何が変わる>。

### 推奨と根拠

<A か B>。理由は <1 行>。

---
```

その直後に `AskUserQuestion` を呼ぶ:

```
header: <選択名 (12 文字以下)>
question: 論点 N: <選択を促す文>
options:
  - label: A 案 を維持 (推奨)
    description: <A を選んだ場合に何が起きるか 1 行>
  - label: B 案 に変更
    description: <B を選んだ場合に何が起きるか 1 行>
  - label: (任意) 保留して詳細を聞く
    description: 判断保留、詳細を AI に質問する
```

## 適用例

### 例 1: 関数の削除可否

```
## 論点 1/2: exists を port から削除するか

### 選択肢の対象
- exists(key) -> bool: Entity の存在を真偽値で返す軽量関数
- get_entity(key) -> Entity | None: Entity を取得 (なければ None)

### 論点
get_entity の戻り値が None かどうかで存在確認も兼ねられる。
exists を残すか削るかは、port のメソッド数 vs クエリの軽さの判断。

### 推奨と根拠
exists を削除。get_entity で代替できるため port メソッドを増やす利益が薄い。

---
```

### 例 2: 命名

```
## 論点 1/1: lifecycle 操作の名前

### 選択肢の対象
- transition_status: 状態遷移であることを名前で明示
- update_status: status 更新一般を意味する汎用名

### 論点
状態遷移には BL-001 の制約があり、generic な update とは責務が異なる。
名前で意図を明示するかどうかの判断。

### 推奨と根拠
transition_status。状態遷移と単純更新を呼び分けることで誤用を減らせる。

---
```

## SKILL からの参照

本テンプレを SKILL から利用する場合、SKILL.md 内で本ファイルへの Read 指示を入れる。例:

```markdown

基本構造の 4 ブロックに従って提示すること。
```
