# 実装戦略策定仕様書

このファイルは、forge オーケストレータスキル（start-plan）が **実装戦略 agent** に渡す
自己完結型の作業指示書である。

agent はこのファイルを読み、設計書を分析し、実装戦略を策定して
結果を `{session_dir}/strategy_draft.md` に書き込む。

---

## 入力パラメータ

オーケストレータから agent に以下のパラメータが渡される:

| パラメータ    | 必須 | 説明                                           |
| ------------- | ---- | ---------------------------------------------- |
| `session_dir` | ○    | 結果書き込み先のセッションディレクトリパス      |
| `spec`        | ○    | この仕様書のパス                               |
| `feature`     | ○    | 対象 Feature 名                                |
| `design_docs` | ○    | 設計書のファイルパス配列                        |
| `rules_docs`  | -    | プロジェクト固有ルール文書のファイルパス配列    |

```yaml
session_dir: .claude/.temp/start-plan-a3f7b2
spec: plugins/forge/docs/strategy_formulation_spec.md
feature: "doc-db"
design_docs:
  - docs/specs/doc-db/design/DES-026_doc_db_design.md
rules_docs:
  - docs/rules/architecture.md
```

## 前提

- `session_dir` のパスが agent に渡されていること
- `design_docs` に少なくとも1つの設計書パスが含まれること
- 設計書は Read して全文を把握すること（部分読みは禁止）
- `rules_docs` が空またはなしの場合は、プロジェクト固有制約なしとして続行

---

## 処理手順

### Step 1: 設計書の全体構造分析

`design_docs` に含まれる全設計書を Read し、以下を把握する:

- モジュール数と各モジュールの責務
- モジュール間の依存関係（依存グラフの形状）
- 技術的な複雑度（新規技術の採用、外部サービス連携等）
- テスト設計の有無と範囲

### Step 2: リスク分析

設計書の内容から技術的リスクを抽出する:

| リスクカテゴリ       | 着目点                                       |
| -------------------- | -------------------------------------------- |
| 技術的不確実性       | 未検証の技術・アルゴリズム・外部 API          |
| 依存の複雑性         | 循環に近い依存、深いネスト                    |
| インテグレーション   | 複数モジュールの結合点                        |
| データ整合性         | スキーマ変更、マイグレーション                |
| パフォーマンス       | 大量データ処理、リアルタイム性要求            |

### Step 3: アプローチ選択

設計書の特性とリスク分析に基づき、実装アプローチを選択する。
複数の組み合わせも可。選択の根拠を必ず記述すること。

| アプローチ           | 概要                              | 適用場面                             |
| -------------------- | --------------------------------- | ------------------------------------ |
| ボトムアップ         | 基盤層→上位層の順に積み上げ       | 安定した基盤が必要な場合             |
| スケルトン先行       | stub で全層を通す→肉付け         | 早期 E2E 検証が重要な場合            |
| リスク駆動           | 不確実性の高い部分から着手        | 技術的リスクが大きい場合             |
| フィーチャースライス | 最小機能セットを縦断実装→拡張     | 早期デモ・フィードバックが必要な場合 |

**選択の判断基準**:

- モジュール間依存が深い → ボトムアップ
- 結合テストが重要 → スケルトン先行
- 未検証技術を含む → リスク駆動
- ユーザーフィードバックが必要 → フィーチャースライス

### Step 4: フェーズ分割

実装全体を **2〜4 フェーズ**に分割する。各フェーズについて以下を定義:

- **目標**: このフェーズ完了時に何が動作するか（具体的な検証可能条件）
- **スコープ**: 対象とするモジュール・コンポーネント（設計書のセクション参照）
- **検証ポイント**: フェーズ完了時に確認すべき事項（ビルド・テスト・動作確認）

**フェーズ分割の原則**:

- 各フェーズは独立して検証可能な成果物を持つこと
- フェーズ間の依存は一方向（フェーズ N+1 はフェーズ N の成果物に依存してよい）
- 最初のフェーズにリスクの高い要素を含めること（手戻りの早期検出）

### Step 5: 戦略書の出力

以下のテンプレートに従い `{session_dir}/strategy_draft.md` に書き込む。

---

## 出力テンプレート

```markdown
# {feature} 実装戦略

## アプローチ

**選択**: {選択したアプローチ名}

**根拠**:

{なぜこのアプローチが最適か。設計書の特性との対応を記述}

## フェーズ

### フェーズ 1: {名前}

- **目標**: {完了時に何が動作するか}
- **スコープ**: {対象モジュール — 設計書のセクション参照}
- **検証ポイント**: {ビルド成功 / テスト通過 / 特定シナリオの動作確認}

### フェーズ 2: {名前}

- **目標**: ...
- **スコープ**: ...
- **検証ポイント**: ...

(フェーズ 3, 4 は必要に応じて追加)

## リスクと対策

| リスク | 影響度 | 対策（どのフェーズで潰すか） |
| ------ | ------ | ---------------------------- |
| ...    | ...    | ...                          |
```

---

## プロジェクト固有ルールの適用

`rules_docs` が渡された場合、以下の観点でルールを戦略に反映する:

- アーキテクチャルール（レイヤー構造、依存方向）→ フェーズ分割の制約
- テストルール（カバレッジ要件、テスト種別）→ 検証ポイントに反映
- コーディング規約 → 戦略への直接影響は少ないが、リスク判断の参考

---

## エラー時の振る舞い

- 設計書の Read に失敗した場合: エラー内容を orchestrator に報告し終了
- 設計書が不完全（モジュール一覧なし等）の場合: 読み取れる情報のみで戦略を策定し、不足事項を戦略書内に明記
- フェーズ分割が困難な場合（設計が単純で2フェーズ未満）: 「単一フェーズ（一括実装）」を明示し、理由を記述
