# ID 分類カタログ

プロジェクト固有の `spec_format.md` が見つからない場合に参照するカタログ。
プロジェクトの性質に合わせて必要なものを選んで使用する。使わない ID は定義しない。

要件・設計の境界ガイド: `${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md`

---

## Feature（フィーチャー）

Feature とは、仕様の断片をグループ化した開発単位である。

### いつ使うか

| 開発パターン       | Feature の使い方                                                     |
| ------------------ | -------------------------------------------------------------------- |
| 追加開発           | 既存のメイン仕様に後から追加する機能群を Feature として分割する      |
| アジャイル開発     | イテレーションごとに Feature 単位で開発・デリバリーする              |
| 小規模プロジェクト | Feature 分割せず、プロジェクト全体を1つの Feature として扱ってもよい |

### forge での扱い

forge のスキル（start-requirements / start-design / start-plan / start-implement）は、Feature 単位で処理を実行する。スキル実行時に **どの Feature に対して処理するか** を決定する必要がある:

- Feature が1つしかない → そのまま対象とする
- 複数の Feature がある → ユーザーに確認する
- Feature が未定義 → 新規 Feature として作成する

### ディレクトリ構造

Feature ごとに要件定義書・設計書・計画書をサブディレクトリで管理する:

```
specs/
  {feature}/
    requirements/   # 要件定義書
    design/         # 設計書
    plan/           # 計画書
```

実際のパスはプロジェクトの `.doc_structure.yaml` で定義される。上記は推奨構造であり、プロジェクトの都合に合わせて変更可能。

## 要件ID カタログ

| ID prefix   | 対象               | 主な用途                                                      |
| ----------- | ------------------ | ------------------------------------------------------------- |
| `APP-xxx`   | アプリ概要         | アプリの目的・価値・主要機能の概要定義                        |
| `SCR-xxx`   | 画面               | 画面ごとの表示要件・操作要件（UI を持つプロジェクト）         |
| `CMP-xxx`   | UI コンポーネント  | 再利用可能な UI コンポーネントの要件                          |
| `THEME-xxx` | デザイントークン   | カラー・タイポグラフィ・スペーシング等のデザイン定義          |
| `NAV-xxx`   | ナビゲーション     | 画面遷移・ルーティング構造                                    |
| `FNC-xxx`   | 機能要件           | ユーザーが実行できる操作・システムの振る舞い                  |
| `BL-xxx`    | ビジネスロジック   | 計算ルール・バリデーション・業務ルール                        |
| `DM-xxx`    | データモデル       | エンティティ定義・保存データの構造                            |
| `API-xxx`   | API エンドポイント | REST/GraphQL 等のエンドポイント仕様（バックエンド・外部連携） |
| `EXT-xxx`   | 外部連携           | 外部サービス・SDK・ライブラリとの連携要件                     |
| `NFR-xxx`   | 非機能要件         | 性能・可用性・スケーラビリティ等                              |
| `SEC-xxx`   | セキュリティ       | 認証・認可・データ保護要件                                    |
| `ERR-xxx`   | エラー定義         | エラーコード・エラーハンドリング方針                          |
| `INT-xxx`   | インテグレーション | サービス間連携・イベント・メッセージング                      |
| `BUG-xxx`   | バグ報告           | 不具合修正・既知バグのトラッキング                            |

## 設計ID カタログ

| ID prefix | 対象           | 主な用途                           |
| --------- | -------------- | ---------------------------------- |
| `DES-xxx` | 設計書（汎用） | モジュール設計・画面設計・機能設計 |

---

## タスクID カタログ

| ID prefix  | 対象           | 主な用途                               |
| ---------- | -------------- | -------------------------------------- |
| `TASK-xxx` | タスク（汎用） | プロジェクト構造に依存しない汎用タスク |

プロジェクトがレイヤードアーキテクチャを採用している場合、層別にプレフィックスを定義すると依存関係の整理に役立つ（例: ドメイン層・UI 層・インフラ層等）。具体的な命名はプロジェクトのアーキテクチャルールに従うこと。

---

## 優先度 カタログ

| 範囲   | 意味                                   |
| ------ | -------------------------------------- |
| 70〜99 | 高: コア機能・共通基盤・ブロッカー解消 |
| 40〜69 | 中: 主要機能                           |
| 1〜39  | 低: UI・補助機能・後回し可能なもの     |

---

## ファイル命名規則

| 文書種別   | 命名パターン                | 例                            |
| ---------- | --------------------------- | ----------------------------- |
| 要件定義書 | `{REQ-ID}_{name}_spec.md`   | `SCR-001_user_list_spec.md`   |
| 設計書     | `{DES-ID}_{name}_design.md` | `DES-001_user_list_design.md` |
| 計画書     | `{feature}_plan.yaml`       | `user_management_plan.yaml`   |
