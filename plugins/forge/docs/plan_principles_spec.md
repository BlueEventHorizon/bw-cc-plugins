# 計画書 作成原則

プロジェクト固有のルールがない場合に参照する汎用原則。
プロジェクト固有のルールがある場合はそちらを優先すること。

文書記述スタイル（タグ・見出し・参照記法）は `${CLAUDE_PLUGIN_ROOT}/docs/document_style_guide.md` を参照。本文書は計画書固有の構成ルール・タスク設計指針を扱う。

---

## 計画書の構成ルール [MANDATORY]

計画書は YAML 形式で、以下の4セクションのみで構成する:

1. `requirements_traceability` — 要件トレーサビリティマトリクス
2. `design_traceability` — 設計トレーサビリティマトリクス
3. `tasks` — タスク一覧
4. `revision_history` — 改定履歴

**上記以外のキーは追加しないこと。** 計画書の目的は「優先度の高いタスクを検出し実行する」ことであり、タスク実行に不要な情報（依存関係図、概要説明等）は記載しない。必要な情報はすべてタスクの各フィールドに含める。

ファイル名: `{feature}_plan.yaml`（拡張子は `.yaml`）

---

## タスクの粒度 [MANDATORY]

| 基準       | 内容                                            |
| ---------- | ----------------------------------------------- |
| 単位       | 1つの Agent 実行で完結する単位                  |
| 量         | やるべき内容は5〜10項目程度                     |
| 完結性     | タスク完了時にビルド・テストが成功すること      |
| ファイル数 | 1つのファイル、または密接に関連する2〜3ファイル |

**悪い例（粒度が粗すぎる）**:

- ❌ 全モジュール実装
- ❌ 全画面実装
- ❌ 基盤層全体実装

---

## 「やるべき内容」の記載原則 [MANDATORY]

設計書を参照すればわかる実装詳細（プロパティ名、型、メソッドシグネチャ等）は計画書に書かない。設計書の**該当セクションを特定できるレベル**の記述にとどめること。

計画書に実装詳細を転記すると設計書との二重管理になり、不整合の原因となる。

---

## `required_reading` フィールドの仕様 [MANDATORY]

タスク実行前に必ず読むべき文書のファイルパスを `tasks[].required_reading` 配列に列挙する。

**記載形式**: プロジェクトルートからの相対パス文字列の YAML 配列

**正しい例**:

```yaml
required_reading:
  - specs/feature/design/DES-001_foo_design.md
  - rules/architecture/domain_core.md
```

**誤った例**:

- `foo_design` ❌（曖昧名）
- `specs/*/design/*.md` ❌（glob 禁止）
- `DES-001` ❌（ID のみ）
- `-` ❌（旧 Markdown table 表現。YAML では `null` でも `-` でもなく **空配列 `[]`** を使う）

**必読なし**: `required_reading: []`（空配列）

## タスクグループ

タスクを細かく分割すると、途中でビルドが壊れる状態が生じるのは普通のことである。
その場合、複数タスクを1つのグループとしてまとめ、グループ完了時にビルドを確認する。

### グループ化の判定基準

「タスクN完了時点でビルドが通るか？」

- **Yes** → 独立タスク
- **No** → タスクグループ化

### グループサイズ制限

- **最大10タスク**を目安とする
- 超過する場合は段階的に分割できないか検討する

### グループ化の記載方法

タスクの `group_id` と `build_check` フィールドを使用する:

| フィールド    | 値                  | 説明                     |
| ------------- | ------------------- | ------------------------ |
| `group_id`    | `"GROUP-001 (1/3)"` | グループ ID と通し番号   |
| `group_id`    | `null`              | 独立タスク               |
| `build_check` | `per_task`          | 独立タスク（デフォルト） |
| `build_check` | `skip`              | グループ途中タスク       |
| `build_check` | `on_group_complete` | グループ最終タスク       |

---

## 依存関係管理

- 依存される側を先に実装する
- 循環依存がないか確認すること
- 依存関係マップは**作業メモ**として使ってよいが、計画書本体には含めない（依存関係は各タスクの `depends_on` フィールドに落とし込む）

---

## 並列実行可能タスクの識別

依存関係がないタスク群は並列実行可能として識別する。`depends_on: []` かつ `group_id: null` のタスクは並列実行候補。

---

## タスク受け入れ基準規範 [MANDATORY] (forge-review feature 統合で追加)

### 規範

1. 各タスクに受け入れ基準 (acceptance criteria) フィールドを必ず持たせる
2. 受け入れ基準は **Yes/No で判定可能** な記述にする (主観表現禁止)
3. 主要機能の実装タスクには対応するテストタスクを必ず計画書に含める
4. テストタスクは「実装タスクと同一グループ」または「直後の独立タスク」として配置する

### 受け入れ基準の主観表現禁止例

| ❌ 不許容 (主観表現)           | ✅ 許容 (判定可能)                                                 |
| ------------------------------ | ------------------------------------------------------------------ |
| 「適切に動作する」             | 「正常系 3 ケース・異常系 2 ケースで期待値と一致する」             |
| 「問題なく処理できる」         | 「指定入力で例外を出さず指定 schema の出力を返す」                 |
| 「パフォーマンスが改善される」 | 「ベースライン処理時間に対して 50% 以内」(根拠 = 要件側で確定済み) |
| 「ユーザビリティが向上する」   | (この粒度はタスクではなく要件側で受け入れ基準を定義する)           |

---

## テストタスク必須化規範 [MANDATORY] (forge-review feature 統合で追加)

### 規範

1. 主要機能 (要件 ID に直接紐づく機能) の実装タスクには、対応するテストタスクが計画書に必ず存在する
2. テストタスクの粒度は「単体テスト」「統合テスト」のいずれかを明示する
3. テストタスクは実装タスクの `build_check` 設計に組み込む (グループ完了時 or 独立)

### 例外 (許容範囲)

| 論点                                 | ✅ 許容                                                    | ❌ 不許容                    |
| ------------------------------------ | ---------------------------------------------------------- | ---------------------------- |
| SKILL.md (テキスト規約) のみのタスク | テストタスク省略可 (SKILL.md は本リポジトリでテスト対象外) | なし                         |
| Markdown 文書編集のみのタスク        | テストタスク省略可                                         | なし                         |
| Python スクリプト追加・修正          | 単体テスト必須                                             | テストタスクなし             |
| 既存テストで網羅済みの内部リファクタ | 「既存テストで網羅」と明示するメモを持つ                   | 何も書かずテストタスクを省略 |

---

## 暗黙依存検出規範 [MANDATORY] (forge-review feature 統合で追加)

### 規範

1. タスクの `depends_on` フィールドに、実装上必要な先行タスクを必ず列挙する
2. 「フロントエンド実装」が「API 定義」に依存する等の典型的な暗黙依存パターンを見落とさない
3. 依存関係マップは作業メモとして使ってよいが、計画書本体に図として残さない (本文書「依存関係管理」の既存規範)

### 典型的な暗黙依存パターン

| 依存元タスク           | 依存先タスク                                | 重大度 (未記載時) |
| ---------------------- | ------------------------------------------- | ----------------- |
| フロントエンド画面実装 | API スキーマ確定 / API 実装                 | 🔴 critical       |
| 統合テスト             | 全機能実装完了                              | 🔴 critical       |
| マイグレーション実行   | スキーマ定義タスク                          | 🔴 critical       |
| デプロイ               | ビルド / テスト / 環境変数設定              | 🔴 critical       |
| 画面実装               | デザイントークン定義 / デザインシステム整備 | 🟡 major          |
| 通知機能               | 通知チャネル設定 / 認証情報整備             | 🟡 major          |

---

## トレーサビリティ規範 [MANDATORY] (forge-review feature 統合で追加)

### 規範

1. 計画書の `requirements_traceability` に全要件 ID を載せ、対応タスク ID を必ず付与する
2. 対応タスクが無い要件 ID には除外理由を明示する (除外理由欄)
3. タスクの内容が要件・設計と矛盾しないこと

### グレーゾーン許容範囲

| 論点                           | ✅ 許容                                                      | ❌ 不許容                            |
| ------------------------------ | ------------------------------------------------------------ | ------------------------------------ |
| 要件 ID に対応タスクが無い     | 除外理由欄に「次フェーズで実装」「TBD-xxx と一体化」等を明示 | 除外理由なし                         |
| 1 タスクが複数要件 ID をカバー | 各要件 ID への対応関係を明示 (主従関係を記載)                | 「複数要件をまとめて実装」のみの記述 |

---

## 重大度カタログ [MANDATORY] (forge-review feature 統合で追加)

本文書内の各規範を、違反時の重大度に対応付ける。

### 構成ルール (本文書既存)

| 違反パターン                                                                                      | 違反時の重大度 | 理由                                            |
| ------------------------------------------------------------------------------------------------- | -------------- | ----------------------------------------------- |
| `requirements_traceability` / `design_traceability` / `tasks` / `revision_history` 以外のキー追加 | 🟡 major       | 計画書の目的逸脱 (タスク実行に不要な情報の追加) |
| YAML 形式以外で記述                                                                               | 🔴 critical    | ツールチェーンが解釈不能                        |
| ファイル名が `{feature}_plan.yaml` 規約から逸脱                                                   | 🟡 major       | 自動解決経路が機能しない                        |

### タスクの粒度 (本文書既存)

| 違反パターン                                        | 違反時の重大度 | 理由                                             |
| --------------------------------------------------- | -------------- | ------------------------------------------------ |
| タスク内容が 5〜10 項目を大幅に超える (例: 20 項目) | 🟡 major       | Agent 1 実行で完結しない粒度                     |
| タスク完了時にビルド・テストが成功しない            | 🔴 critical    | 計画書の中核要件違反                             |
| 1 タスクが多数ファイル (4 ファイル以上) を操作      | 🟢 minor       | 粒度が粗いがケースバイケース。グループ化で対応可 |

### 「やるべき内容」記載原則 (本文書既存)

| 違反パターン                                                       | 違反時の重大度 | 理由                           |
| ------------------------------------------------------------------ | -------------- | ------------------------------ |
| 設計書から実装詳細 (プロパティ名 / 型 / メソッドシグネチャ) を転記 | 🟡 major       | 二重管理化・不整合源           |
| 設計書セクションを特定できないレベルの曖昧記述                     | 🟡 major       | 実装者が必要情報に辿り着けない |

### `required_reading` フィールド (本文書既存)

| 違反パターン                                | 違反時の重大度 | 理由                                         |
| ------------------------------------------- | -------------- | -------------------------------------------- |
| 必読パスがファイル ID のみ (例: `DES-001`)  | 🔴 critical    | 自動解決不能                                 |
| 必読パスが glob (例: `specs/*/design/*.md`) | 🔴 critical    | 自動解決不能                                 |
| 必読パスが曖昧名 (例: `foo_design`)         | 🔴 critical    | 自動解決不能                                 |
| `required_reading` フィールドが未指定       | 🟡 major       | 「必読なし」は明示的に `[]` で表現する       |
| `required_reading` に `-` 等の文字列を記載  | 🟡 major       | 旧 Markdown table 表現。YAML では空配列 `[]` |

### タスクグループ (本文書既存)

| 違反パターン                                                 | 違反時の重大度 | 理由                                         |
| ------------------------------------------------------------ | -------------- | -------------------------------------------- |
| グループ途中タスクで `build_check: per_task` を指定          | 🟡 major       | ビルド失敗を想定したグループ化の意義を無効化 |
| グループサイズが 10 タスクを大幅超過 (例: 20 タスク)         | 🟡 major       | 段階的分割の検討漏れ                         |
| グループ最終タスクに `build_check: on_group_complete` 未指定 | 🟡 major       | グループ完了時のビルド確認漏れ               |

### 依存関係管理 (本文書既存 + forge-review feature 統合で追加)

| 違反パターン                                              | 違反時の重大度          | 理由                                |
| --------------------------------------------------------- | ----------------------- | ----------------------------------- |
| 循環依存 (タスク A → B → C → A)                           | 🔴 critical             | 実行順序決定不能                    |
| 依存される側より依存する側が先に実装される順序            | 🔴 critical             | 実装不能                            |
| 暗黙依存の未記載 (上記「典型的な暗黙依存パターン」表参照) | 🔴 critical 〜 🟡 major | パターンごとに重大度差あり (上記表) |

### 受け入れ基準・テストタスク (forge-review feature 統合で追加)

| 違反パターン                                                     | 違反時の重大度 |
| ---------------------------------------------------------------- | -------------- |
| 主要機能タスクに受け入れ基準フィールドなし or 空                 | 🔴 critical    |
| 受け入れ基準が主観表現 (「適切に動作」「問題なく」等)            | 🟡 major       |
| 主要機能タスクに対応するテストタスクが計画書に存在しない         | 🔴 critical    |
| テストタスクが Markdown 文書編集等の例外条件に該当しないのに省略 | 🔴 critical    |

### トレーサビリティ (forge-review feature 統合で追加)

| 違反パターン                                         | 違反時の重大度 |
| ---------------------------------------------------- | -------------- |
| `requirements_traceability` に未掲載の要件 ID がある | 🔴 critical    |
| 対応タスクなしの要件 ID に除外理由が記載されていない | 🟡 major       |
| タスクが対応要件と矛盾する内容                       | 🔴 critical    |

### 追加 feature 用 frontmatter (Issue #118)

判定（追加 feature か否か）は [`additive_development_spec.md`](additive_development_spec.md) §1（適用条件 / 対象外）に従う。main 初期立ち上げ・既存文書の追記更新は対象外（false positive 防止）。マーカーの定義は [`plan_format.md`](plan_format.md)「追加 feature 用 frontmatter」節を参照（plan.yaml は先頭コメントブロックで表現する）。

| 違反パターン                                                              | 違反時の重大度 | 理由                                                       |
| ------------------------------------------------------------------------- | -------------- | ---------------------------------------------------------- |
| 追加 feature 計画書に `type: temporary-feature-plan` マーカーコメント欠如 | 🟡 major       | 一時計画書であることが宣言されず、merge/破棄判断を誤らせる |
