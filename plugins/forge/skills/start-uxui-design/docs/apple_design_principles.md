# Apple デザイン原則 & UX/UI 哲学知識ベース

本文書はデザイン分析時に AI が参照する知識ベースである。プラットフォーム固有の指針は別ファイル（`ios_platform_guide.md` / `macos_platform_guide.md`）を参照すること。

---

## 1. Apple HIG 4 原則

Apple Human Interface Guidelines の根幹をなす4つの設計哲学。すべてのデザイン判断はこの原則に立脚する。

### 1.1 Clarity（明瞭性）

インターフェースは**一目で理解**できなければならない。

- テキストは読みやすいサイズで表示する
- アイコンは正確かつ明快に意味を伝える
- 装飾は最小限にし、機能に奉仕させる
- 余白（ネガティブスペース）を積極的に活用して視覚的ノイズを減らす

**評価観点**: 初見のユーザーが3秒以内に画面の目的を把握できるか

### 1.2 Deference（従属性）

UI はコンテンツを引き立てる存在であり、**コンテンツと競合してはならない**。

- UI 要素はコンテンツの背景に退く
- 大胆な色やテクスチャは控え、コンテンツを主役にする
- 全画面表示を活用してコンテンツに没入させる
- アニメーションは意味を伝えるためだけに使い、装飾的なモーションは避ける

**評価観点**: UI フレームとコンテンツの視覚的重量バランスが適切か

### 1.3 Depth（深度）

視覚的なレイヤーとモーションで**空間的な階層**を表現する。

- 半透明・ブラー（Liquid Glass）で奥行きを暗示する
- 遷移アニメーションで空間的関係を伝える
- シャドウと elevation でコンテンツの優先度を示す
- z 軸の使い方で情報の重要度を直感的に理解させる

**評価観点**: 要素間の階層関係が視覚的に自明か

### 1.4 Consistency（一貫性）

Apple エコシステム全体で**馴染みのあるパターン**を維持する。

- 標準コンポーネントを優先使用する
- プラットフォーム慣習に従う（戻るジェスチャー、メニュー構成等）
- 同一アプリ内でインタラクションパターンを統一する
- SF Symbols を活用してアイコンの一貫性を保つ

**評価観点**: ユーザーが学習済みのパターンを再利用しているか

## 2. Liquid Glass（2025-2026）

Apple の最新デザイン言語。プラットフォーム横断で統一的な視覚体験を提供する。

- **素材特性**: 半透明・光の屈折・背景のぼかしで物理的なガラスを模倣
- **適応性**: 背景コンテンツに応じて動的に見え方が変わる
- **統一性**: iOS / macOS / visionOS で共通の視覚言語を実現
- **控えめさ**: Deference 原則の具現化。UI がコンテンツの上に「浮く」が主張しすぎない

**評価観点**: Liquid Glass 的な半透明・レイヤー表現がプラットフォームの最新トレンドに沿っているか

## 3. SF フォント体系

Apple プラットフォーム共通のタイポグラフィシステム。

### フォントファミリー

| フォント       | 用途                             |
| -------------- | -------------------------------- |
| SF Pro Display | 20pt 以上の見出し                |
| SF Pro Text    | 19pt 以下の本文                  |
| SF Pro Rounded | 親しみやすい印象のバリエーション |
| SF Mono        | コード・等幅テキスト             |
| New York       | セリフ体（長文読書向け）         |

### テキストスタイル階層

| スタイル    | 役割                   |
| ----------- | ---------------------- |
| Large Title | 画面タイトル（最上位） |
| Title 1-3   | セクション見出し       |
| Headline    | 強調テキスト           |
| Body        | 本文                   |
| Callout     | 補足情報               |
| Subhead     | 小見出し               |
| Footnote    | 注釈                   |
| Caption 1-2 | キャプション           |

**評価観点**: タイポグラフィ階層が明確で、2-3 段階以上のコントラストがあるか

## 4. セマンティックカラーシステム

色は**意味**で定義し、具体的な HEX 値はテーマ層で解決する。

### カラーの役割

| カテゴリ   | 例                                                                   | 目的                           |
| ---------- | -------------------------------------------------------------------- | ------------------------------ |
| Label      | primary / secondary / tertiary / quaternary                          | テキストの重要度を段階的に表現 |
| Fill       | primary / secondary / tertiary / quaternary                          | 塗りの視覚的重量を制御         |
| Background | primary / secondary / tertiary / grouped                             | Surface のレイヤー区別         |
| Separator  | opaque / non-opaque                                                  | 要素間の境界                   |
| System     | red / orange / yellow / green / blue / purple / pink / teal / indigo | 意味的アクセント               |

### Light / Dark モード対応

- すべてのカラーは light / dark の2バリアントを持つ
- ダークモードでは彩度を上げ、明度を下げる
- 純白 (#FFFFFF) と純黒 (#000000) の直接使用は避ける（system background を使用）
- コントラスト比は両モードで 4.5:1 以上を確保する

**評価観点**: セマンティックな色命名がされているか、light/dark 両対応が考慮されているか

## 5. Nielsen の 10 ユーザビリティヒューリスティクス

Jakob Nielsen（1994）が提唱した、普遍的な UI 評価フレームワーク。

### H1: システム状態の可視性

システムは適切なフィードバックを**合理的な時間内**に提供しなければならない。

- タップ/クリックへの即座のビジュアルフィードバック（100-300ms）
- 進行中の処理にはプログレスインジケーター
- ネットワーク状態・同期状態の明示

**違反例**: ボタンを押しても何も反応がない、処理中なのかフリーズなのか区別できない

### H2: システムと現実世界の一致

**ユーザーの言葉**で語り、システムの都合を押し付けない。

- 技術用語ではなく日常語を使う
- 情報はユーザーの思考順序で配置する
- 現実世界のメタファーを適切に活用する

**違反例**: エラーコード「ERR_0x4F2A」を表示する

### H3: ユーザーの制御と自由

ユーザーは常に**安全に探索**でき、間違いから回復できるべきである。

- 取り消し（Undo）を提供する
- 明確な「戻る」「閉じる」手段を用意する
- 破壊的操作には確認を求める

**違反例**: 削除操作に Undo がなく、確認ダイアログもない

### H4: 一貫性と標準

プラットフォーム慣習に従い、アプリ内でも**統一されたパターン**を使う。

- 同じアクションには同じアイコン・色・配置を使う
- プラットフォーム標準のコンポーネントを優先する
- 用語を統一する

**違反例**: 同じ「保存」操作に異なるアイコンやラベルを使っている

### H5: エラー防止

エラーが起きてから対処するより、**起きないようにする**方が良い。

- 制約による入力ミスの防止（数値のみ入力可能にする等）
- 危険な操作の前に確認ステップを挟む
- スマートなデフォルト値を設定する
- リアルタイムバリデーション

**違反例**: 自由入力テキストフィールドで日付を求める（ピッカーを使うべき）

### H6: 記憶よりも認識

ユーザーが**覚えていなくても操作できる**ようにする。

- 選択肢を表示する（ドロップダウン、サジェスト）
- 最近使った項目を提示する
- コンテキストに応じたヒントを表示する

**違反例**: コマンドラインのような暗記必須のインターフェース

### H7: 柔軟性と効率性

**初心者にも熟練者にも**対応する。

- アクセラレータ（ショートカット）を提供する
- カスタマイズ可能な UI 要素
- よく使う操作への素早いアクセス

**違反例**: 全ユーザーが同じ固定的なフローを強制される

### H8: 美的でミニマルなデザイン

**すべての要素が目的を持つ**べきである。

- 情報の優先度に応じた視覚的重量配分
- 不要な装飾の排除
- 余白の戦略的活用
- 「less is more」の原則

**違反例**: 画面に情報を詰め込みすぎて視覚的に圧倒される

### H9: エラーからの回復支援

エラーメッセージは**原因と解決策**を平易な言葉で伝える。

- 何が起きたかを具体的に説明する
- 回復のための具体的なアクションを提示する
- 技術的なエラーコードを露出しない

**違反例**: 「問題が発生しました」とだけ表示する

### H10: ヘルプとドキュメント

理想は**ヘルプなしで使える**こと。必要な場合は簡潔に提供する。

- コンテキストヘルプ（ツールチップ、インラインヒント）
- オンボーディングは必要最小限に
- 検索可能なヘルプ

**違反例**: 100 ページのユーザーマニュアルを読まないと使えない

## 6. ゲシュタルト原則

人間の視覚認知の法則。UI レイアウトの「なぜこう見えるか」を説明する根拠。

### 6.1 近接の法則（Proximity）

**近くにある要素は関連している**と知覚される。

- 関連する UI 要素はグループ化して近接配置する
- 無関係な要素間には十分な余白を確保する
- セクション間のスペーシングで情報の塊を形成する

**UI への適用**: フォームのラベルと入力フィールドの距離、カードレイアウトの内部/外部マージン

### 6.2 類似の法則（Similarity）

**見た目が似ている要素は同じグループ**と知覚される。

- 同種の操作には同じ色・形・サイズを使う
- 異なる種類の要素は視覚的に差別化する
- ステータスカラーの一貫した使用

**UI への適用**: ナビゲーションアイテムの統一スタイル、アクションボタン vs 情報表示の差別化

### 6.3 図と地の法則（Figure/Ground）

人は**前景（図）と背景（地）**を区別して知覚する。

- 重要なコンテンツを前景として際立たせる
- モーダル/オーバーレイの背景を暗くして図を明確にする
- カードの elevation でコンテンツ階層を表現する

**UI への適用**: モーダルダイアログの dim 背景、フローティングアクションボタン

### 6.4 連続の法則（Continuity）

視線は**滑らかな線やパス**に沿って流れる。

- 視線の流れを遮らないレイアウト
- リスト・テーブルの縦方向の流れ
- ステッパー/プログレスバーの水平方向の流れ

**UI への適用**: スクロール可能なリスト、ウィザードのステップ表示

### 6.5 閉合の法則（Closure）

人は**不完全な形から完全な形**を想像する。

- 要素の一部が見えていれば全体を推測できる
- 「もっと見る」の示唆（画面端で切れるコンテンツ）

**UI への適用**: 水平スクロールリストの端の切れた要素、ページネーション

### 6.6 共通領域の法則（Common Region）

**同じ領域内にある要素は関連グループ**と知覚される。

- カード、セクション、枠線で要素をグループ化する
- 背景色の変化で領域を区切る

**UI への適用**: カード UI、セクション背景、グループ化された設定項目

### 6.7 対称の法則（Symmetry）

**対称的なレイアウトは安定感**を与える。

- 中央揃え、均等配置で秩序を表現する
- 意図的な非対称で注意を引く

**UI への適用**: ダイアログのボタン配置、グリッドレイアウト

## 7. アクセシビリティ（Apple 共通）

### 7.1 視覚アクセシビリティ

- **コントラスト比**: テキストは最低 4.5:1、大テキスト（18pt 以上）は 3:1
- **色だけに依存しない**: 色に加えて形状・テキスト・パターンで情報を伝える
- **Dynamic Type 対応**: ユーザーのフォントサイズ設定を尊重する
- **代替テキスト**: 全画像・アイコンにアクセシブルな説明を付与する

### 7.2 操作アクセシビリティ

- **VoiceOver 対応**: すべてのインタラクティブ要素に適切なラベルと役割を設定
- **フォーカス管理**: 論理的なフォーカス順序を設計する
- **モーション設定の尊重**: `Reduce Motion` 設定時はアニメーションを簡素化する

### 7.3 認知アクセシビリティ

- **予測可能な動作**: 同じ操作は同じ結果を生む
- **明確なラベル**: ボタン・リンクの目的が文脈なしで理解できる
- **エラーからの回復**: 取り消し可能な操作、明確なエラーメッセージ

**評価観点**: WCAG 2.1 AA 基準を満たしているか

## 8. 8pt グリッドシステム

Apple デザインの空間リズムの基盤。

- **基本単位**: 8pt（4pt も許容）
- **スペーシング**: 4 / 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48 / 64
- **適用対象**: マージン、パディング、要素間の距離、コンポーネントの高さ
- **利点**: 一貫したリズム、様々な画面サイズへの適応性、開発者との共通言語

**評価観点**: スペーシング値が 4pt/8pt の倍数に従っているか、一貫したスケールを使用しているか

## 9. デザイン評価チェックリスト

分析時に各項目を確認し、UX ノートとして出力に付与する。

### HIG 原則準拠度

- [ ] Clarity: 画面の目的が一目で分かるか
- [ ] Deference: UI がコンテンツより目立っていないか
- [ ] Depth: レイヤー・階層が視覚的に表現されているか
- [ ] Consistency: 標準パターンを適切に使用しているか

### ヒューリスティクス

- [ ] フィードバック: 操作に対する応答が明確か
- [ ] メタファー: 現実世界との対応が適切か
- [ ] ユーザーコントロール: 取り消し・戻る手段があるか
- [ ] 一貫性: アプリ内でパターンが統一されているか
- [ ] エラー防止: 入力制約・確認が適切か
- [ ] ミニマルデザイン: 不要な要素がないか

### ゲシュタルト

- [ ] 近接: 関連要素が適切にグループ化されているか
- [ ] 類似: 同種の要素が視覚的に統一されているか
- [ ] 階層: 情報の優先度が視覚的に明確か

### アクセシビリティ

- [ ] コントラスト比: 4.5:1 以上を確保しているか
- [ ] ターゲットサイズ: プラットフォーム推奨サイズを満たしているか
- [ ] 色依存: 色以外の手段で情報を伝えているか
