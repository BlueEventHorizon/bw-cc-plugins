# APP概要要件定義書 記載基準

APP-001（アプリケーション概要要件定義書）に記載すべき項目と、各項目の目的・記載指針を定義する。
この記載基準はプロジェクトの種別を問わず汎用的に適用できる。

---

## 本文書の位置づけ

APP-001 はプロジェクトの「羅針盤」であり、以下の問いに答える文書である:

- **なぜ作るのか？** — 背景・動機・解決する問題
- **何を作るのか？** — スコープ・環境構成・機能一覧
- **誰のために作るのか？** — 対象ユーザー・ステークホルダー
- **どう使うのか？** — 利用シナリオ・前提条件
- **正しく作れているか？** — 成功基準・制約・リスク

APP-001 を読めば、プロジェクトに初めて参加するメンバーが全体像を把握でき、意思決定の根拠を追跡できる状態を目指す。

---

## 記載項目一覧

### 1. 概要

| 項目 | 内容 |
|------|------|
| 目的 | プロジェクト全体を1段落で要約する |
| 記載指針 | 「何を」「誰に」「どのような価値を」提供するかを簡潔に記述する。技術的詳細は書かない |
| 必須度 | 必須 |

### 2. 背景・動機

| 項目 | 内容 |
|------|------|
| 目的 | このプロジェクトが必要になった経緯を説明し、意思決定の根拠を残す |
| 記載指針 | 現状の問題点・課題を具体的に記述する。「なぜ既存の方法では不十分か」を明確にする |
| 必須度 | 必須 |
| 記載例 | 「AI コーディングアシスタントを利用するには各ツールを個別にインストールする必要があり、環境差異によるトラブルが頻発していた」 |

### 3. 解決する問題

| 項目 | 内容 |
|------|------|
| 目的 | 背景から導かれる具体的な問題を列挙し、プロジェクトが何を解消するかを明確にする |
| 記載指針 | 問題ごとに「誰が」「何に困っているか」を記述する。解決手段（How）ではなく問題（What）に焦点を当てる |
| 必須度 | 必須 |

### 4. プロジェクトの目的

| 項目 | 内容 |
|------|------|
| 目的 | 達成すべきゴールを定義する |
| 記載指針 | 各目的に対応する FNC-ID を紐づけ、トレーサビリティを確保する。抽象的な表現（「簡単に」「素早く」）を避け、検証可能な表現にする |
| 必須度 | 必須 |

### 5. 成功基準

| 項目 | 内容 |
|------|------|
| 目的 | プロジェクトの目的が達成されたかどうかを判定する条件を定義する |
| 記載指針 | 各目的に対して、検証可能な条件を記述する。「〜できること」「〜が動作すること」のように、合否が判定できる形にする |
| 必須度 | 必須 |
| 記載例 | 「コンテナ起動後、追加のインストール作業なしで `claude --help` が正常終了すること」 |

### 6. スコープ

| 項目 | 内容 |
|------|------|
| 目的 | プロジェクトが扱う範囲と、明示的に扱わない範囲を定義する |
| 記載指針 | **対象範囲**と**スコープ外**を分けて記述する。スコープ外は特に重要で、「やらないこと」を明示することでスコープの肥大化を防ぐ |
| 必須度 | 必須 |
| 記載例（スコープ外） | 「各 AI コーディングアシスタントの API キー取得手順はスコープ外とする」 |

### 7. 対象ユーザー

| 項目 | 内容 |
|------|------|
| 目的 | 誰がこのプロジェクトの成果物を利用するかを定義する |
| 記載指針 | ユーザー像を具体的に記述する。前提知識（例: Docker の基本操作を理解している）を明記し、文書やUIの粒度を決める基準とする |
| 必須度 | 必須 |

### 8. 対応プラットフォーム

| 項目 | 内容 |
|------|------|
| 目的 | 動作対象の OS・アーキテクチャ・ソフトウェアバージョンを定義する |
| 記載指針 | 対応する OS、必須ソフトウェアとバージョン、アーキテクチャ（x86_64 / arm64 等）を明記する。OS ごとにセットアップ手順が異なる場合はその旨も記述する |
| 必須度 | 必須 |

### 9. 前提条件

| 項目 | 内容 |
|------|------|
| 目的 | このプロジェクトの成果物を利用するために必要な環境・条件を定義する |
| 記載指針 | 「対応プラットフォーム」とは異なり、ユーザーが事前に準備すべき具体的な条件（ソフトウェアのインストール、アカウント作成等）を記述する |
| 必須度 | 必須 |

### 10. 環境構成

| 項目 | 内容 |
|------|------|
| 目的 | システムの構成要素とその関係を示し、全体像を視覚的に把握できるようにする |
| 記載指針 | 主要な構成要素を表や図で整理する。構成要素間の関係（依存、通信、データの流れ等）を示す。実装の詳細ではなく概念レベルで記述する |
| 必須度 | 必須 |

### 11. 利用シナリオ

| 項目 | 内容 |
|------|------|
| 目的 | ユーザーがどのような流れで成果物を利用するかを具体的に示す |
| 記載指針 | 典型的な利用パターンを番号付きの手順で記述する。「初回セットアップ」「日常利用」「トラブル時」など複数のシナリオを網羅する |
| 必須度 | 必須 |

### 12. 主要機能一覧

| 項目 | 内容 |
|------|------|
| 目的 | 個別の機能要件定義書（FNC-xxx）への索引を提供する |
| 記載指針 | ID・機能名・概要を表形式で列挙する。詳細は各 FNC 文書に委ねる |
| 必須度 | 必須 |

### 13. 外部依存

| 項目 | 内容 |
|------|------|
| 目的 | プロジェクトが依存する外部のサービス・ツール・ライブラリを明示し、依存リスクを可視化する |
| 記載指針 | 各依存に対して「何に依存しているか」「その依存が利用できなくなった場合の影響」を記述する |
| 必須度 | 推奨 |
| 記載例 | 「Claude Code のインストーラー（claude.ai/install.sh）に依存。提供停止時はコンテナイメージにバイナリを手動で含める必要がある」 |

### 14. 制約事項

| 項目 | 内容 |
|------|------|
| 目的 | 技術的・ビジネス的な制約を明示し、設計・実装の判断根拠とする |
| 記載指針 | 制約の内容と、その制約が存在する理由を記述する |
| 必須度 | 推奨 |
| 記載例 | 「Alpine Linux を採用するため、glibc 依存のバイナリは直接利用できない」 |

### 15. 既知のリスク

| 項目 | 内容 |
|------|------|
| 目的 | プロジェクトの目的達成を脅かす要因を事前に把握し、対策を検討する |
| 記載指針 | リスクごとに「内容」「影響度」「対策」を記述する。影響度は 高/中/低 で分類する |
| 必須度 | 推奨 |

### 16. 用語定義

| 項目 | 内容 |
|------|------|
| 目的 | プロジェクト内で使用する用語の意味を統一し、曖昧さを排除する |
| 記載指針 | 複数の解釈が可能な用語、プロジェクト固有の用語を定義する。一般的な技術用語は不要 |
| 必須度 | 必須 |

### 17. 関連文書

| 項目 | 内容 |
|------|------|
| 目的 | APP-001 から参照すべき他の文書への索引を提供する |
| 記載指針 | 設計書、計画書、外部仕様書などへのリンクを記述する |
| 必須度 | 推奨 |

---

## 記載上の注意

- APP-001 は要件定義書であり、「What（何を作るか）」を記述する。「How（どう作るか）」は設計書に委ねる
- 判断基準: 「ユーザーマニュアルに書く内容か？」→ Yes なら APP-001 に記載、No なら設計書
- 各項目は簡潔に記述し、詳細は個別の FNC/DM/BL 等の文書に委ねる
- 変更履歴を必ず記録し、変更の経緯を追跡可能にする
