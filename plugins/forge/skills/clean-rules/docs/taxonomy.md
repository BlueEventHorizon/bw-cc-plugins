# 開発文書の分類学（Taxonomy）

clean-rules スキルの判定基準。AI がルール文書を分析する際にこの分類体系に従う。

---

## 次元 1: 内容の種類（Content Type）

| 種類 | 機能 | 判別基準 | 例 |
|------|------|---------|-----|
| **Constraint** | MUST/MUST NOT の硬いルール。違反はバグ | 「〜してはならない」「〜は禁止」「[MANDATORY]」 | セキュリティ制約、外部依存禁止、レビュー観点 |
| **Convention** | SHOULD の合意事項。チームが変更可能 | 「〜を推奨」「〜に統一する」「〜を使用する」 | 命名規則、コードスタイル、用語統一 |
| **Format** | 成果物の構造テンプレート | 「〜のフォーマット」「テンプレート」「必須セクション」 | 設計書テンプレート、要件定義書フォーマット |
| **Process** | ステップバイステップの手順 | 「Step 1:〜」「〜のワークフロー」「〜の手順」 | レビュー手順、デプロイ手順、リリースフロー |
| **Decision** | 選択の根拠。ADR 相当 | 「〜を選んだ理由」「なぜ〜か」「検討した代替案」 | 技術選定理由、アーキテクチャ判断 |
| **Reference** | 参照情報。変更は事実の反映 | 一覧表、カタログ、マッピング、定義集 | API 一覧、用語集、ID 分類カタログ |

### 判別の優先順位

1 つのセクションが複数の種類に見える場合:

1. `[MANDATORY]` タグがある → **Constraint**
2. テンプレート構造を定義している → **Format**
3. 手順を記述している → **Process**
4. 「なぜ」を説明している → **Decision**
5. 一覧・カタログである → **Reference**
6. 上記いずれでもない → **Convention**

## 次元 2: 権威源（Authority Source）

| 源 | 管理者 | 判別基準 |
|---|--------|---------|
| **Tool-provided** | forge プラグイン | forge 内蔵 docs と同一または類似の内容 |
| **Project-defined** | チーム合意 | このプロジェクト固有の取り決め |
| **External standard** | 標準団体・言語仕様 | 外部仕様への参照（RFC、ISO、言語公式ドキュメント等） |

---

## 次元 3: スコープ

| スコープ | 適用範囲 | 判別基準 |
|---------|---------|---------|
| **Universal** | forge を使う全プロジェクト共通 | forge 内蔵 docs と対応する内容 |
| **Project-specific** | このプロジェクトのみ | プロジェクト名・固有技術・固有ドメインへの言及 |

---

## forge とプロジェクトの責務分離

| Content Type | forge が提供するもの | プロジェクトが定義すべきもの |
|-------------|---------------------|--------------------------|
| **Constraint** | レビュー観点（5種別） | プロジェクト固有の制約・セキュリティルール |
| **Convention** | — | 命名規則、コードスタイル、用語統一 |
| **Format** | 文書テンプレート（設計書・要件定義書・計画書） | — |
| **Process** | ワークフロー（review / start-* スキル内蔵） | デプロイ手順、リリースフロー、Git ワークフロー |
| **Decision** | — | アーキテクチャ判断（ADR）、技術選定理由 |
| **Reference** | ID 分類カタログ、仕様書フォーマット定義 | プロジェクト固有の参照情報 |

---

## 再構築の原則

1. **1 ファイル 1 関心事**: Content Type が異なるセクションは別ファイルに分離する
2. **forge 知識の可視化**: forge 内蔵 docs の内容もプロジェクトの rules/ に読める形で配置する
3. **マーキング不要**: forge 提供元の追跡コメントは付けない。スキルを再実行すれば再分析される
4. **プロジェクト固有部分を最優先で保護**: forge と重複するセクションでもプロジェクト固有の追記がある場合は必ず残す
