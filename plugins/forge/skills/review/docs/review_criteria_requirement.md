# 要件定義書レビュー観点

プロジェクト固有のレビュー観点がない場合に使用するプラグインデフォルト定義。
DocAdvisor（`/doc-advisor:query-rules`）が利用可能な場合は、プロジェクト固有の観点が追加の perspective として加わる。

## 必須参照: 知識ベース [MANDATORY]

レビュー実行前に `/forge:query-forge-rules` で要件定義に関連する知識ベースを取得し、**Read** して深く理解した上でレビューを実行すること。

```
/forge:query-forge-rules
タスク内容: 要件定義書レビュー、ID体系・フォーマット準拠性検証、仕様記述の品質評価
```

返却されたファイルを全て Read する。

**`/forge:query-forge-rules` が利用不可の場合のフォールバック**:

1. `${CLAUDE_PLUGIN_ROOT}/docs/spec_format.md`
2. `${CLAUDE_PLUGIN_ROOT}/docs/requirement_format.md`
3. `${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md`
4. `${CLAUDE_PLUGIN_ROOT}/docs/spec_priorities_spec.md`

## Perspective: completeness — 完全性 (Completeness)

必須要件の網羅性、非機能要件の不足、例外系・異常系の考慮漏れを検出する観点。

### 🔴致命的

- **必須要件の欠落**: ユーザストーリやビジネスゴールに対応する要件が存在しない
  - 判定基準: ユーザストーリに「〜できる」と記載された機能に対応する要件 ID が存在しないこと
  - 具体例: 「管理者がユーザを削除できる」というストーリに対し、削除機能の要件が未定義
  - false positive に注意: 別の要件 ID で間接的にカバーされている場合がある。要件間の参照関係を確認すること
- **非機能要件の致命的欠落**: 性能・可用性・セキュリティなど、システム運用に不可欠な非機能要件が完全に未定義
  - 判定基準: 非機能要件カテゴリ（性能、可用性、セキュリティ、運用性）のいずれかが文書内に一切記載されていない
  - 具体例: 認証を必要とするシステムでセキュリティ要件が未定義
- **例外系・異常系の考慮漏れ**: 正常系のみ定義され、エラー時やエッジケースの振る舞いが未記載
  - 判定基準: 入力値のバリデーション、タイムアウト、リソース不足等の異常系要件がない
  - 具体例: API 連携機能で接続失敗時の振る舞いが未定義

### 🟡品質問題

- **非機能要件の不十分な記述**: カテゴリは存在するが、具体的な数値目標や測定方法が不足
  - 判定基準: 「高速であること」「十分なセキュリティ」等の定量化されていない記述
  - 具体例: 「レスポンスは速いこと」→ 具体的な応答時間の基準がない
- **境界条件の未定義**: データの上限・下限、同時接続数の制限等が明示されていない
  - 判定基準: 数量に関わる機能で上限値・下限値の記載がない
- **運用要件の不足**: バックアップ、監視、ログ、移行手順等の運用に関する要件が薄い

### 🟢改善提案

- **ユースケースの追加**: 要件の理解を深めるためのユースケース記述の補完
- **図表による補足**: 要件間の関係性やデータフローの視覚的な表現
- **前提条件の明文化**: 暗黙の前提が存在する場合、明示的に記載する

## Perspective: consistency — 整合性 (Consistency)

要件間の矛盾・競合、用語定義の不統一、ビジネスゴールとの追跡性（トレーサビリティ）を検出する観点。

### 🔴致命的

- **要件間の矛盾・競合**: 同一機能や同一データに対して相反する記述が存在する
  - 判定基準: 2つ以上の要件が同一対象について異なる振る舞いを定義している
  - 具体例: 要件 A で「データは即座に削除する」、要件 B で「データは30日間保持する」と記述
  - false positive に注意: 条件分岐により異なる振る舞いが正当な場合がある。適用条件の有無を確認すること
- **ビジネスゴールとの不整合**: 要件がビジネスゴールと矛盾する、またはビジネスゴールに紐づかない要件が存在する
  - 判定基準: トレーサビリティマトリクスでビジネスゴールに紐づかない要件がある
  - 具体例: コスト削減がゴールなのに、高コストな機能が必須要件として定義されている

### 🟡品質問題

- **用語定義の不統一**: 同一概念に対して異なる用語が使用されている
  - 判定基準: 文書内で同じ意味の異なる表現が混在（例: 「ユーザ」と「利用者」と「アカウント」）
  - 具体例: ある箇所では「認証」、別の箇所では「ログイン」と記述し、同じ機能を指している
  - false positive に注意: 意図的に異なるスコープで使い分けている場合がある
- **要件間のトレーサビリティ不足**: 関連する要件間の参照が不十分で追跡が困難
  - 判定基準: 関連する要件が相互参照を持たない
- **優先度の矛盾**: 依存先の要件よりも依存元の優先度が高い等の不整合
  - 判定基準: 依存関係にある要件間で優先度の逆転がある

### 🟢改善提案

- **用語集の整備**: 主要概念の定義一覧を設けて用語を統一する
- **トレーサビリティマトリクス**: 要件間の依存関係を表形式で整理する
- **分類体系の改善**: 機能要件・非機能要件の分類を整理し、参照性を向上させる

## Perspective: verifiability — 検証可能性 (Verifiability)

曖昧な表現の排除、定量的な受け入れ基準、技術的制約との矛盾、優先度・スコープの明確さを検出する観点。

### 🔴致命的

- **検証不可能な受け入れ基準**: 受け入れ基準が曖昧で、合格・不合格を客観的に判定できない
  - 判定基準: 「適切に」「十分に」「使いやすい」等の主観的表現が受け入れ基準に含まれる
  - 具体例: 「ユーザにとって使いやすい UI であること」→ 具体的な測定方法がない
  - false positive に注意: 要件概要の説明文（受け入れ基準ではない部分）に主観的表現が含まれるのは許容範囲
- **技術的制約との矛盾**: 要件が既知の技術的制約と矛盾している
  - 判定基準: プラットフォーム制限、API 制約、ライセンス条件等と矛盾する要件
  - 具体例: オフライン必須のシステムで「リアルタイム同期」を要件としている

### 🟡品質問題

- **曖昧な表現**: 要件本文に解釈の幅が大きい表現が含まれる
  - 判定基準: 「など」「等」「必要に応じて」「可能な限り」等の曖昧語が要件の核心部分に使用されている
  - 具体例: 「必要に応じてキャッシュを利用する」→ いつ必要かの基準が不明
- **スコープの不明確さ**: 対象範囲や除外範囲が明示されていない
  - 判定基準: 「すべての」「各種」等の範囲が不明確な表現が使用されている
- **優先度の未定義**: 要件に優先度が設定されていない、または優先度の基準が不明
  - 判定基準: 優先度フィールドが空、または全要件が同一優先度

### 🟢改善提案

- **定量的基準への置き換え**: 主観的表現を具体的な数値や条件に変換する
- **受け入れテストの具体化**: 各要件に対するテストシナリオの概要を追記する
- **スコープ境界の明文化**: 対象範囲と除外範囲を明示的にリスト化する

## Perspective: priorities — 優先順位 (Priorities)

要件定義書で優先すべき価値観（責務分割・境界・依存方向）と、主目的にしてはいけない事項（構造品質の定量化・品質特性の直接数値化）の遵守を検出する観点。判断根拠: `${CLAUDE_PLUGIN_ROOT}/docs/spec_priorities_spec.md`、`${CLAUDE_PLUGIN_ROOT}/docs/spec_design_boundary_spec.md` §4.6。

### 🔴致命的

- **構造品質の定量化**: 責務分割・境界・依存方向の品質を数値目標化している
  - 判定基準: 「モジュール数 ≤ N」「依存エッジを -X%」「flag 数 ≤ N」等の定量目標が要件として定義されている
  - 具体例: 「共通モジュール A のフラグ数を 10 以下に抑える」「responsibility の数値目標を設定する」
  - false positive に注意: 応答時間・スループット等の運用で検証可能な定量目標は本項目の対象外（4.6 参照）
- **実装手段の要件混入**: 特定のフレームワーク・ライブラリ・内部構造を要件に記載している
  - 判定基準: 「X ライブラリを使用して」「Y クラスで実装」等、実装方法が要件に書かれている
  - 具体例: 「Redis を使って 200ms を切る」（実装手段 + 定量目標のセット混入）
  - false positive に注意: ユーザー要求として特定プラットフォーム・OS・SDK が指定されている場合は正当

### 🟡品質問題

- **品質特性の直接数値化**: 変更容易性・運用しやすさ・テスト容易性を直接の数値目標にしている
  - 判定基準: 「修正箇所数を N 以下」「追加作業時間を X 時間以内」等、品質特性を直接指標化している
  - 具体例: 「変更容易性の KPI として修正ファイル数を 3 以下にする」
  - false positive に注意: 可用性 SLA（稼働率 99.9% 等）は §2 の品質特性ではなく運用指標なので正当
- **計測動機の要件記述**: 「計測できるように書く」が動機化した要件
  - 判定基準: 要件の内容より「数値として表現できること」を優先した記述になっている
  - 具体例: 本質的に Yes/No で判断できる事項を、無理に %・件数に翻訳している

### 🟢改善提案

- **責務分割の明示**: 要件レベルでもユースケース・アクターごとの責務境界を明示する
- **定量目標の正当化**: 定量目標を記載する場合、ユーザー要求またはビジネス制約として出典を明示する
- **評価表の目的化回避**: 評価表・KPI 表の作成自体が目的化しないよう、原則（責務・境界・依存）を主題として記述する
