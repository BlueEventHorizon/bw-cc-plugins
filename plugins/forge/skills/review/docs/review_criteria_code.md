# コードレビュー観点

プロジェクト固有のレビュー観点がない場合に使用するプラグインデフォルト定義。
DocAdvisor（`/doc-advisor:query-rules`）が利用可能な場合は、プロジェクト固有の観点が追加の perspective として加わる。

---

## Perspective: logic — 正確性 (Logic)

ロジックエラー、エッジケース、境界値、データ損失リスク、設計書・要件定義書との不整合を検出する観点。

### 🔴致命的

- **ロジックエラー**: 意図した振る舞いと実際の実装が異なるバグ
  - 判定基準: 条件分岐の誤り、ループの終了条件の間違い、演算子の誤用等、実行結果が仕様と異なるコード
  - 具体例: `if (a > b)` とすべきところが `if (a >= b)` になっており、境界値で誤った分岐に入る
  - false positive に注意: 仕様が不明確な場合は指摘を控え、仕様確認を推奨するに留めること
- **データ損失のリスク**: ユーザデータや重要データが意図せず失われる可能性
  - 判定基準: 上書き前のバックアップがない、トランザクション外での更新、削除操作の前に確認がない
  - 具体例: 既存データを確認せずに `UPDATE` や `DELETE` を実行するコードで、WHERE 句の条件が不十分
- **設計書・要件定義書との不整合**: 実装が設計や要件と異なる振る舞いをする
  - 判定基準: 設計書で定義されたインターフェース、データモデル、処理フローと実装が一致しない
  - 具体例: 設計書では「失敗時にリトライ」と定義されているが、実装ではリトライなしで即座にエラーを返している
- **エッジケース・境界値の未処理**: null、空配列、最大値、ゼロ除算等の境界条件でクラッシュや不正動作が発生する
  - 判定基準: 入力値の境界条件に対するガード処理がないコード
  - 具体例: 配列の長さを確認せずにインデックスアクセスしている

### 🟡品質問題

- **型安全性の問題**: 暗黙の型変換やアンセーフなキャストが潜在的な実行時エラーを引き起こす
  - 判定基準: 強制アンラップ、未チェックのキャスト、暗黙的な数値変換（精度落ち）
  - 具体例: `as!` による強制キャストが型不一致時にクラッシュする
- **競合状態（Race Condition）**: 並行処理での共有リソースアクセスに適切な同期がない
  - 判定基準: 複数スレッド/タスクからアクセスされる変数に排他制御がない
- **不完全なエラー伝播**: エラーが握り潰される、または不適切に変換されて情報が失われる
  - 判定基準: catch ブロックが空、またはログ出力のみでエラーを無視している

### 🟢改善提案

- **アサーションの追加**: 前提条件の明示的な検証
- **テストケースの追加**: エッジケースに対するユニットテストの提案
- **型の明示**: 暗黙的な型推論を明示的な型注釈に置き換える

## Perspective: resilience — 堅牢性 (Resilience)

セキュリティ脆弱性（インジェクション、認証不備等）、エラーハンドリングの不足、リソース管理、入力バリデーションを検出する観点。

### 🔴致命的

- **セキュリティ脆弱性**: インジェクション攻撃、認証・認可の不備、機密情報の露出
  - 判定基準: ユーザ入力の未サニタイズ、SQL/コマンドインジェクション可能なコード、ハードコードされたシークレット
  - 具体例: ユーザ入力を直接 SQL クエリに文字列結合している、API キーがソースコード内にハードコードされている
  - false positive に注意: ORM やパラメータ化クエリを使用している場合は SQL インジェクションの指摘は不要。フレームワークの保護機構を確認すること
- **認証・認可の不備**: 保護されるべきリソースに未認証・未認可でアクセス可能
  - 判定基準: 認証チェックの欠落、権限検証のバイパス可能な経路
  - 具体例: 管理者専用 API エンドポイントに権限チェックが実装されていない
- **リソースリーク**: ファイルハンドル、ネットワーク接続、メモリ等のリソースが解放されない
  - 判定基準: `open` / `acquire` に対応する `close` / `release` がない、または例外パスでリソースが解放されない
  - 具体例: ファイルを開いた後、例外発生時に `finally` / `defer` でクローズしていない

### 🟡品質問題

- **エラーハンドリングの不足**: 失敗する可能性のある操作でエラー処理が不十分
  - 判定基準: ネットワーク通信、ファイルI/O、外部API呼び出し等でエラーハンドリングがない
  - 具体例: HTTP リクエストの結果を確認せずにレスポンスボディをパースしている
  - false positive に注意: 呼び出し元で一括キャッチしている設計の場合、個々の呼び出しでの処理は不要な場合がある
- **入力バリデーションの不足**: 外部入力（ユーザ入力、API レスポンス、ファイル読み込み）の検証が不十分
  - 判定基準: 型チェック、範囲チェック、フォーマット検証がないままデータを処理している
- **ログ・監視情報の不足**: 障害時の原因特定に必要な情報がログに記録されない
  - 判定基準: エラー発生時にコンテキスト情報（入力値、状態、操作内容）がログに含まれない
- **タイムアウト未設定**: 外部サービス呼び出しにタイムアウトが設定されていない
  - 判定基準: HTTP クライアント、DB 接続等にタイムアウト値が未設定

### 🟢改善提案

- **防御的プログラミング**: 入力値の事前検証パターンの導入
- **構造化ログの導入**: エラーコンテキストを含むログフォーマットの統一
- **リトライ戦略の導入**: 一時的な障害に対するリトライパターンの提案

## Perspective: maintainability — 保守性 (Maintainability)

コーディング規約遵守、可読性、テスト可能性（DI 等の構造）、テスト充足度、重複、パフォーマンス問題を検出する観点。

### 🔴致命的

- **テスト不可能な構造**: 依存関係がハードコードされ、テストでモックに差し替えられない
  - 判定基準: コンストラクタ内で具体的な依存を直接生成している、グローバル状態に強く依存している
  - 具体例: `class FooService { private db = new Database() }` のようにコンストラクタ内で具体的な DB 接続を生成し、テスト時にインメモリ DB に差し替えられない
  - false positive に注意: シンプルなスクリプトやユーティリティ関数で DI が過剰設計になる場合は指摘不要。プロジェクトの規模と複雑さに応じて判断すること

### 🟡品質問題

- **コーディング規約違反**: プロジェクトの規約やスタイルガイドに従っていないコード
  - 判定基準: 命名規則、インデント、ファイル構成等がプロジェクトの規約と異なる
  - 具体例: プロジェクトが camelCase を採用しているのに snake_case で命名している
  - false positive に注意: プロジェクト規約が明示されていない場合は、一般的な言語慣習を基準にする。規約の不在自体を指摘すべきであり、個別の命名を責めない
- **コードの重複**: 同一または類似のロジックが複数箇所に存在する
  - 判定基準: 3箇所以上で同一パターンのコードが繰り返されている
  - false positive に注意: 似て非なるロジック（表面的には類似だが、業務上異なるルール）の共通化は誤り。共通化すべきか個別に判断すること
- **可読性の問題**: コードの意図が理解しにくい
  - 判定基準: 過度にネストした条件分岐、意味不明な変数名、マジックナンバー
  - 具体例: 5段階以上のネスト、1文字変数名（ループカウンタ以外）、説明のない数値リテラル
- **パフォーマンス問題**: 明らかに非効率な実装
  - 判定基準: N+1 クエリ、ループ内でのリソース生成、不要な全件取得
  - 具体例: ループ内で毎回 DB コネクションを新規作成している
  - false positive に注意: パフォーマンスが問題にならない規模（少量データ、低頻度処理）では指摘の優先度を下げること
- **テスト充足度の不足**: 主要なロジックに対するテストが不足している
  - 判定基準: 分岐を含むビジネスロジックにユニットテストがない
  - false positive に注意: テストファイルが別ディレクトリに存在する場合がある。テストの有無は対象コード単体ではなくプロジェクト全体で判断すること

### 🟢改善提案

- **リファクタリング**: 長いメソッドの分割、共通処理の抽出
- **コメントの追加**: 複雑なロジックの意図を説明するコメント
- **命名の改善**: より意図が伝わる変数名・関数名への変更提案
- **定数の定義**: マジックナンバーの名前付き定数への置き換え
