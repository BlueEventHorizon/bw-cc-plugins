# 計画書レビュー観点

プロジェクト固有のレビュー観点がない場合に使用するプラグインデフォルト定義。
DocAdvisor（`/query-rules`）が利用可能な場合は、プロジェクト固有の観点が追加の perspective として加わる。

---

## Perspective: alignment — 整合性 (Alignment)

要件・設計との不整合、必須タスクの網羅、タスク間の依存関係の矛盾を検出する観点。

### 🔴致命的

- **要件・設計との不整合**: 計画書のタスクが要件定義書や設計書の内容と矛盾する
  - 判定基準: 要件 ID や設計要素に対応するタスクが存在しない、またはタスクの内容が要件・設計と矛盾している
  - 具体例: 設計書で「認証機能はOAuth2を使用」と定義されているのに、計画書では「独自認証の実装」タスクが定義されている
  - false positive に注意: 設計書の更新が計画書作成後に行われた場合、差異が生じることがある。文書の更新日時を確認すること
- **必須タスクの欠落**: 要件・設計を実現するために不可欠なタスクが計画に含まれていない
  - 判定基準: 要件定義書・設計書の主要機能に対応するタスクが計画書に存在しない
  - 具体例: データベースマイグレーションが設計で必要とされているのに、移行タスクが計画にない
- **タスク間の依存関係の矛盾**: 依存先タスクより先に依存元タスクが実行される順序になっている
  - 判定基準: タスク A がタスク B に依存すると明記されているのに、タスク A の方が先に実行される計画
  - 具体例: 「API 実装」タスクが「DB スキーマ設計」タスクに依存するのに、API 実装が先に計画されている

### 🟡品質問題

- **タスク間の暗黙的依存**: 明示されていないが実行順序に制約がある依存関係の見落とし
  - 判定基準: 依存関係フィールドに記載がないが、実装上は先行タスクの成果物が必要なタスク
  - 具体例: フロントエンド実装タスクが API 定義タスクへの依存を明示していない
- **要件カバレッジの不足**: 全要件がタスクでカバーされていることの確認が不十分
  - 判定基準: 要件定義書の要件 ID に対応するタスクがなく、かつ除外理由が記載されていない
- **テスト・検証タスクの不足**: 実装タスクに対応するテストや検証のタスクが計画されていない
  - 判定基準: 主要機能の実装タスクに対して、テストタスクや結合テストタスクが存在しない

### 🟢改善提案

- **トレーサビリティの追加**: 各タスクと要件 ID・設計要素との対応を明示する
- **依存関係図の追加**: タスク間の依存関係を視覚的に表現する
- **マイルストーンの設定**: 中間成果物の確認ポイントを計画に組み込む

## Perspective: feasibility — 現実性・リスク (Feasibility)

タスク粒度の妥当性、ボトルネックの特定、リスク対策の妥当性、受け入れ基準の明確さ、優先順位を検出する観点。

### 🔴致命的

- **実現不可能なタスク**: 技術的制約やリソース制約により実現できないタスクが計画に含まれている
  - 判定基準: 既知の技術的制約（プラットフォーム制限、API 制限等）と矛盾するタスク
  - 具体例: 使用するフレームワークがサポートしていない機能の実装タスクが計画されている
  - false positive に注意: 計画書内で代替手段や回避策が記載されている場合は問題ない
- **受け入れ基準の欠落**: タスクの完了を判定できる基準が全く定義されていない
  - 判定基準: タスクに受け入れ基準（acceptance criteria）フィールドがない、または空
  - 具体例: 「パフォーマンス改善」タスクに具体的な目標値や測定方法が記載されていない

### 🟡品質問題

- **タスク粒度の不均一**: タスクの大きさにばらつきが大きく、進捗管理が困難
  - 判定基準: 1時間で完了するタスクと数日かかるタスクが同列に並んでいる
  - 具体例: 「設定ファイル修正」と「全画面 UI 実装」が同じ粒度で定義されている
  - false positive に注意: 明確に定義された大タスクがサブタスクに分割されている場合は問題ない
- **リスク対策の不足**: 識別されたリスクに対する対策や代替計画が不十分
  - 判定基準: リスクが識別されているが、対策（回避・軽減・受容の判断と具体的アクション）が未記載
- **ボトルネックの未特定**: クリティカルパス上のリスクが識別されていない
  - 判定基準: 複数タスクの依存先となるタスクが遅延した場合の影響が未評価
- **優先順位の不適切さ**: ビジネス価値やリスクに基づかない優先順位設定
  - 判定基準: 高リスクまたは高価値のタスクが低優先度に設定されている
- **受け入れ基準の曖昧さ**: 基準は存在するが、客観的に判定できない表現で記述されている
  - 判定基準: 「適切に動作する」「問題なく処理できる」等の主観的表現

### 🟢改善提案

- **タスク分割**: 粒度が大きすぎるタスクの分割提案
- **バッファの追加**: リスクの高いタスクにバッファ時間を設定する
- **受け入れ基準の具体化**: 定量的な完了基準への書き換え提案
