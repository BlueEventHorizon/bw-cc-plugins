## Perspective: hig_compliance — HIG 準拠性 (HIG Compliance)

Apple Human Interface Guidelines の 4 原則（Clarity / Deference / Depth / Consistency）およびプラットフォーム固有の必須要件への準拠を検証する観点。

### 🔴致命的

- **HIG 必須パターンの重大な逸脱**: プラットフォーム必須のナビゲーション構造・レイアウト規則を無視している
  - 判定基準: iOS で TabBar が 5 タブ超過、macOS でメニューバーが未定義、標準ジェスチャー（戻るスワイプ等）のブロック、Safe Area の無視
  - 具体例: iOS アプリで Tab Bar に 8 個のタブを配置
  - false positive に注意: 意図的な HIG 逸脱として文書内に理由が記載されている場合は許容
- **セマンティックカラーの不在**: Light/Dark モード対応が必要なのに固定 HEX 値のみ定義されている
  - 判定基準: カラートークンに Light/Dark のバリアント定義がなく、一方のモードで視認不能になる
  - 具体例: テキスト色に #333333 のみ定義し、Dark モードでの可読性が確保されていない
- **Clarity 原則の重大な違反**: 画面の目的がまったく分からない構成
  - 判定基準: 情報の階層が存在しない、テキストが読めないサイズ、アイコンが意味不明
  - 具体例: 全テキストが同一サイズ・同一ウェイトで、見出しと本文の区別がつかない

### 🟡品質問題

- **Deference 原則の違反**: UI 要素がコンテンツより目立っている
  - 判定基準: 装飾的な UI 要素（過度なボーダー、背景パターン、アニメーション）がコンテンツの視認性を阻害
  - 具体例: カードの枠線やシャドウが強すぎて、中のテキストより目立つ
- **Depth 原則の不備**: レイヤー・階層が視覚的に表現されていない
  - 判定基準: モーダル/シートとベース画面の区別が不明確、オーバーレイの奥行き表現がない
  - 具体例: モーダルと背景が同じ elevation（シャドウなし）で表示される設計
- **Consistency 原則の違反**: アプリ内で同種の操作に異なるパターンを使用
  - 判定基準: 同じ種類のアクション（削除、保存、ナビゲーション等）に異なる UI パターンが使用されている
  - 具体例: ある画面では「保存」が右上、別の画面ではフローティングボタン
- **iOS 固有の違反**: TabBar にラベルがない、タッチターゲットが 44pt 未満、Large Title の使用が不適切
- **macOS 固有の違反**: ホバー状態が未定義、キーボードショートカットがない、コンテキストメニューが未対応

### 🟢改善提案

- **Liquid Glass 表現の検討**: 半透明・ブラー等の最新 Apple デザイン言語の採用
- **SF Symbols の活用**: カスタムアイコンの代わりに SF Symbols の使用を推奨
- **Haptic Feedback 設計**: 操作に応じた触覚フィードバックの追加（iOS）
- **ホバー状態の充実**: ポインタ操作時のビジュアルフィードバック強化（macOS）

## Perspective: usability — ユーザビリティ (Usability)

Nielsen の 10 ユーザビリティヒューリスティクスに基づき、操作性・学習容易性・エラー耐性を検証する観点。

### 🔴致命的

- **H3 違反: ユーザーコントロールの欠如**: 破壊的操作に取り消し手段も確認も存在しない
  - 判定基準: 削除・上書き等のデータ喪失操作に Undo も確認ダイアログもない
  - 具体例: スワイプ削除が確認なしで即座に実行され、Undo スナックバーもない
  - false positive に注意: 一時的なフィルタリング等の非破壊的操作には不要
- **H1 違反: システム状態の不可視**: 処理中なのか完了なのかユーザーが判断できない
  - 判定基準: 非同期処理にプログレスインジケーターやフィードバックがない
  - 具体例: データ送信ボタンを押した後、何も反応がなく処理中かフリーズか区別できない

### 🟡品質問題

- **H2 違反: 技術用語の露出**: ユーザー向けテキストにシステム内部の用語が使用されている
  - 判定基準: エラーメッセージやラベルに技術的な用語（エラーコード、内部 ID 等）が含まれる
  - 具体例: 「ERR_NETWORK_TIMEOUT」をそのまま表示
- **H4 違反: アプリ内の一貫性欠如**: 同じアクションに異なるアイコン・色・ラベルを使用
  - 判定基準: 同種の操作が画面ごとに異なる見た目・配置になっている
  - 具体例: 「お気に入り」がある画面ではハートアイコン、別の画面ではスターアイコン
- **H5 違反: エラー防止の不足**: ユーザーが容易にミスを犯せる入力設計
  - 判定基準: 自由入力テキストで制約可能な値を求めている、危険な操作に確認ステップがない
  - 具体例: 日付入力にテキストフィールドを使用（ピッカーを使うべき）
- **H6 違反: 記憶への過度な依存**: ユーザーが操作手順やコマンドを暗記する必要がある
  - 判定基準: 選択肢を表示せず自由入力を強いる、ヘルプなしでは使えない操作
- **H8 違反: 情報過多**: 画面に不要な情報が詰め込まれている
  - 判定基準: 現在のタスクに無関係な情報が画面上に表示されている
  - 具体例: 設定画面に上級者向けオプションがフィルターなしで全表示
- **H9 違反: 不明瞭なエラーメッセージ**: エラーの原因や回復方法が伝わらない
  - 判定基準: 「問題が発生しました」等の曖昧なエラーメッセージで具体的な対処法が不明
  - 具体例: バリデーションエラーで「入力値が不正です」とだけ表示し、何が不正かを示さない

### 🟢改善提案

- **H7 改善: アクセラレータの追加**: パワーユーザー向けのショートカット・クイックアクション
- **H10 改善: コンテキストヘルプの追加**: ツールチップ、インラインヒント
- **オンボーディングフローの提案**: 初回利用時のガイド設計

## Perspective: visual_system — 視覚体系 (Visual System)

ゲシュタルト原則・8pt グリッド・タイポグラフィ階層に基づき、デザイントークンとビジュアルシステムの品質を検証する観点。

### 🔴致命的

- **アクセシビリティ基準の違反**: コントラスト比が WCAG 4.5:1 を下回る
  - 判定基準: テキスト色と背景色の組み合わせのコントラスト比が 4.5:1 未満（大テキスト 18pt 以上は 3:1）
  - 具体例: 薄いグレーテキスト (#CCCCCC) を白背景 (#FFFFFF) に配置（コントラスト比 1.6:1）
  - false positive に注意: 装飾要素やプレースホルダー等の非必須テキストは基準が異なる場合がある
- **インタラクティブ要素のサイズ不足**: タッチターゲット / クリックターゲットがプラットフォーム基準未満
  - 判定基準: iOS で 44x44pt 未満、macOS で 20x20pt 未満のインタラクティブ要素
  - 具体例: 閉じるボタンが 24x24pt（iOS では 44pt 必要）

### 🟡品質問題

- **8pt グリッド非準拠のスペーシング**: スペーシング値が 4pt/8pt の倍数に従っていない
  - 判定基準: margin / padding / gap に 4pt 倍数でない値（例: 5pt, 7pt, 13pt）が使用されている
  - 具体例: カード内パディングが 15pt（16pt が適切）
  - false positive に注意: 1pt のボーダーや separator 等の例外は許容
- **タイポグラフィ階層の不明瞭**: テキストスタイルのコントラストが不十分
  - 判定基準: 見出し・本文・キャプション等の区別が不十分、サイズ差が 2pt 以下
  - 具体例: Title が 16pt、Body が 14pt で差が 2pt しかない
- **ゲシュタルト近接の法則違反**: 関連要素間の間隔が無関連要素間の間隔と同じか大きい
  - 判定基準: ラベルと入力フィールドの距離 >= フィールド間の距離
  - 具体例: ラベル-フィールド間が 16pt、フィールド-フィールド間が 12pt（逆であるべき）
- **ゲシュタルト類似の法則違反**: 同種の要素が視覚的に統一されていない
  - 判定基準: 同じ役割のボタン・カード・リスト項目で色・サイズ・形状が不統一
  - 具体例: ナビゲーションアイテムのアイコンサイズが 20pt と 28pt で混在
- **色の過剰使用**: カラーパレットが複雑すぎて一貫性がない
  - 判定基準: アクセント色が 4 色以上、類似色の区別が困難
  - 具体例: プライマリに Blue、セカンダリに Teal、ターシャリに Cyan を使用（類似しすぎ）
- **コンポーネント命名が一般的**: 画面固有の名前ではなく汎用名が使用されている
  - 判定基準: 「Card」「List」「Header」等の一般名がコンポーネント定義に使用されている
  - 具体例: CMP 定義で「Card」→「ProductDetailCard」のように具体化すべき
- **デザイントークン未参照の固定値**: 色・フォント・スペーシングが THEME トークンを参照せず固定値で記述されている
  - 判定基準: HEX 値やピクセル値が直接記述され、対応するデザイントークン名が参照されていない
  - 具体例: 「背景色: #F5F5F5」→「背景色: color.background.secondary」を参照すべき
  - false positive に注意: THEME-xxx がまだ定義されていない場合は指摘不要

### 🟢改善提案

- **Dark モードのカラーバリアント提案**: Light モードのみ定義されている場合に Dark モードの推定値を提案
- **スペーシングスケールの正規化**: 不規則なスペーシング値を 8pt グリッドに揃える提案
- **SF フォント体系への統一**: カスタムフォントの代わりに SF Pro を推奨する提案
- **コンポーネント再利用性の向上**: 類似 UI 要素の共通コンポーネント化を提案
- **状態設計の補完**: 各コンポーネントの状態（Default / Pressed / Hover / Disabled 等）定義の追加を提案
