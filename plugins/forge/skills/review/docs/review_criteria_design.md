# 設計書レビュー観点

プロジェクト固有のレビュー観点がない場合に使用するデフォルト定義。
プロジェクトに独自の観点定義がある場合は、DocAdvisor の `/query-rules` Skill または `.claude/review-config.yaml` 経由でそちらが優先される。

---

## Perspective: alignment — 整合性 (Alignment)

要件定義書との不整合、外部システム・既存コンポーネントとのインターフェース、データフローの矛盾・欠落、設計判断の根拠不足を検出する観点。

### 🔴致命的

- **要件定義書との不整合**: 設計が要件を満たさない、または要件と矛盾する設計
  - 判定基準: 要件 ID に対応する設計要素が存在しない、または設計が要件と異なる振る舞いを定義している
  - 具体例: 要件で「リアルタイム通知」と定義されているのに、設計ではポーリング方式を採用し通知遅延が発生する構成になっている
  - false positive に注意: 設計書内で代替手段による要件充足が説明されている場合は不整合ではない
- **データフローの矛盾・欠落**: データの入力元・出力先・変換処理に矛盾がある、または必要なデータフローが欠落している
  - 判定基準: あるコンポーネントの出力が次のコンポーネントの入力と型・形式が一致しない、必要なデータが途中で消失する
  - 具体例: コンポーネント A が JSON を出力するが、コンポーネント B は XML のみ受け付ける設計
- **外部インターフェースの不整合**: 外部システムや既存コンポーネントとの接続仕様が矛盾している
  - 判定基準: API 仕様、プロトコル、データ形式が接続先の仕様と一致しない

### 🟡品質問題

- **設計判断の根拠不足**: 重要な設計判断に対して「なぜそう決めたか」が記載されていない
  - 判定基準: アーキテクチャ選択、技術選定、トレードオフの判断に理由が未記載
  - 具体例: 「RDB を採用する」とだけ記載され、NoSQL や他の選択肢との比較検討がない
  - false positive に注意: 業界標準やプロジェクト規約で自明な選択は、根拠が省略されていても問題ない
- **要件カバレッジの不足**: 全ての要件が設計に反映されていることの確認が不十分
  - 判定基準: 要件定義書に記載された要件のうち、設計書内で言及されていないものがある
- **制約事項の未記載**: 技術的制約やプラットフォーム制限が設計書に反映されていない

### 🟢改善提案

- **ADR（Architecture Decision Record）の追加**: 重要な設計判断の記録を体系化する
- **トレーサビリティの強化**: 要件 ID と設計要素の対応表を追加する
- **代替案の記載**: 検討した代替設計とその棄却理由を記録する

## Perspective: architecture — 構造・品質 (Architecture)

責務分割（疎結合・高凝集）、アーキテクチャ原則の遵守、拡張性・保守性の問題を検出する観点。

### 🔴致命的

- **アーキテクチャ原則の違反**: 採用したアーキテクチャパターンの根本原則に違反する設計
  - 判定基準: レイヤードアーキテクチャで下位層が上位層に依存する等、パターンの基本制約への違反
  - 具体例: クリーンアーキテクチャを採用しているのに、Domain 層が Infrastructure 層を直接参照している
  - false positive に注意: 意図的な例外として設計書内に理由が記載されている場合は許容
- **循環依存**: コンポーネント間に循環参照が存在する
  - 判定基準: A → B → C → A のような依存の環が存在する
  - 具体例: モジュール A がモジュール B をインポートし、モジュール B もモジュール A をインポートしている

### 🟡品質問題

- **責務の曖昧さ（低凝集）**: 1つのコンポーネントが複数の異なる責務を持つ
  - 判定基準: コンポーネントの説明に「〜と〜を担当する」のように複数の異なるドメイン概念が含まれる
  - 具体例: 「ユーザ認証とレポート生成を担当する Service」のように無関係な責務が混在
  - false positive に注意: ファサードパターンのように、意図的に複数の操作をまとめる設計は正当
- **密結合**: コンポーネント間の依存が強すぎ、一方の変更が他方に広く波及する
  - 判定基準: インターフェースを介さない直接的な内部構造への依存がある
- **拡張性の問題**: 将来の変更に対して大規模な修正が必要になる設計
  - 判定基準: 新機能追加時に既存コンポーネントの大幅な変更が必要になる構造
- **過剰な抽象化**: 現在の要件に対して不必要に複雑な抽象化が施されている
  - 判定基準: 現在の利用箇所が1つしかないのに、汎用フレームワーク的な構造になっている

### 🟢改善提案

- **インターフェースの導入**: 密結合を解消するための抽象化の提案
- **コンポーネント図の追加**: 依存関係を視覚的に表現する
- **命名の改善**: コンポーネント名から責務が明確に読み取れるようにする

## Perspective: resilience — 堅牢性 (Resilience)

セキュリティ上の問題、エラーハンドリングの不足、可観測性（ログ・監視・追跡可能性）を検出する観点。

### 🔴致命的

- **セキュリティ上の問題**: 認証・認可・データ保護に関する設計上の欠陥
  - 判定基準: 機密データが暗号化されずに保存・転送される設計、認証バイパスが可能な経路の存在
  - 具体例: API エンドポイントに認証なしでアクセスできる経路が設計上存在する
  - false positive に注意: 公開 API として意図的に認証不要と設計されている場合は問題ない。設計書内の意図を確認すること
- **単一障害点（SPOF）**: システム全体の可用性に影響する単一障害点が対策なしで存在する
  - 判定基準: 1つのコンポーネントの障害がシステム全体を停止させる構成で、冗長化や代替経路が未設計
  - 具体例: 単一のデータベースインスタンスに依存し、フェイルオーバ設計がない

### 🟡品質問題

- **エラーハンドリングの不足**: 異常系のデータフローや障害時の振る舞いが未設計
  - 判定基準: 正常系のフローのみ定義され、例外・タイムアウト・リソース不足等の異常時の処理が未記載
  - 具体例: 外部 API 連携で、接続失敗時のリトライ戦略やフォールバックが未定義
- **可観測性の不足**: ログ・メトリクス・トレーシングの設計がない、または不十分
  - 判定基準: 障害発生時の原因特定に必要な情報が収集できない設計
  - 具体例: エラー発生時のログ出力設計がなく、問題の再現・追跡が困難
- **データ整合性のリスク**: 並行処理やトランザクション境界の設計が不十分
  - 判定基準: 複数コンポーネントが同一データを更新する場合に排他制御が未設計

### 🟢改善提案

- **障害シナリオの追加**: 想定される障害パターンと対応策の明文化
- **監視設計の追加**: アラート条件、ダッシュボード構成の追記
- **セキュリティ脅威モデルの明文化**: 想定する脅威と対策の対応表
