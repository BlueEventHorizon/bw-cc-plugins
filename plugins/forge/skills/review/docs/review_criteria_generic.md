# 汎用文書レビュー観点

プロジェクト固有のレビュー観点がない場合に使用するプラグインデフォルト定義。
DocAdvisor（`/doc-advisor:query-rules`）が利用可能な場合は、プロジェクト固有の観点が追加の perspective として加わる。

このファイルは Perspective セクションを持たない。全観点を単一 agent で一括適用する。

---

## 🔴致命的

- **事実の誤り**: 記載内容が客観的事実と異なる
  - 判定基準: 技術仕様、バージョン番号、API 仕様、日付等の客観的に検証可能な情報が誤っている
  - 具体例: 「Python 3.12 で導入された match 文」→ 実際には Python 3.10 で導入
  - false positive に注意: 執筆時点では正しかったが、その後変更された情報は「事実の誤り」ではなく「更新の必要」として扱うこと
- **論理矛盾**: 文書内の異なる箇所で相反する記述がある
  - 判定基準: 同一文書内で、前提と結論が矛盾する、または異なるセクションで同一事項について矛盾する記述がある
  - 具体例: 概要で「非同期処理を採用する」と記述しているのに、詳細セクションでは同期処理を前提とした手順が記載されている
- **参照切れ**: リンク、ファイルパス、コマンドが無効
  - 判定基準: 参照先のファイルが存在しない、URL が 404 を返す、コマンドが実行不可能
  - 具体例: `詳細は docs/architecture.md を参照` と記載されているが、当該ファイルが存在しない
  - false positive に注意: 将来作成予定のファイルへの参照が「TODO」等で明示されている場合は許容
- **必須情報の欠落**: 文書の目的を達成するために不可欠な情報が記載されていない
  - 判定基準: 手順書で手順が抜けている、仕様書で必須パラメータの説明がない等
  - 具体例: インストール手順書で前提条件（必要なランタイムバージョン等）が未記載

## 🟡品質問題

- **論理構成の一貫性**: 文書全体の構成や論理の流れに問題がある
  - 判定基準: セクションの順序が不自然、前提知識の説明なしに専門用語を使用、結論が唐突
  - 具体例: 手順3で使用する概念が手順5で初めて説明されている
- **用語の不統一**: 同一概念に対して異なる用語が混在している
  - 判定基準: 同じファイル内で同一の概念を指す異なる表現が使用されている
  - 具体例: 「設定ファイル」「コンフィグ」「config」が同一文書内で混在
  - false positive に注意: 初出時に「設定ファイル（config）」と定義した上で以降「config」を使う場合は統一されているとみなす
- **記述の重複**: 同じ内容が複数箇所に記載され、保守コストが高い
  - 判定基準: 同一文書内、または密接に関連する文書間で同じ情報が重複している
  - 具体例: 「前提条件」セクションと「環境構築」セクションで同じインストール手順が記載されている
- **冗長性**: 文書の目的に対して不必要に長い、または無関係な情報が含まれている
  - 判定基準: 読者が文書の目的を達成するために必要ない情報が大量に含まれている

## 🟢改善提案

- **表現の明確化**: 曖昧な記述をより具体的な表現に置き換える
- **構成改善**: セクション順序の整理、見出しの追加による可読性向上
- **不足情報の補完**: 理解を助ける補足説明や具体例の追加
- **図表の追加**: テキストだけでは理解が困難な内容を視覚化する
