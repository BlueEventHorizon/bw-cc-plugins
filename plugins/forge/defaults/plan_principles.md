# 計画書 作成原則

プロジェクト固有のルールがない場合に参照する汎用原則。
プロジェクト固有のルールがある場合はそちらを優先すること。

---

## 計画書の構成ルール [MANDATORY]

計画書は以下の4セクションのみで構成する:

1. 要件トレーサビリティマトリクス
2. 設計トレーサビリティマトリクス
3. タスク一覧
4. 改定履歴

**上記以外のセクションは追加しないこと。** 計画書の目的は「優先度の高いタスクを検出し実行する」ことであり、タスク実行に不要な情報（依存関係図、概要説明等）は記載しない。必要な情報はすべてタスク表の各列に含める。

---

## タスクの粒度 [MANDATORY]

| 基準       | 内容                                            |
| ---------- | ----------------------------------------------- |
| 単位       | 1つの Agent 実行で完結する単位                  |
| 量         | やるべき内容は5〜10項目程度                     |
| 完結性     | タスク完了時にビルド・テストが成功すること      |
| ファイル数 | 1つのファイル、または密接に関連する2〜3ファイル |

**悪い例（粒度が粗すぎる）**:

- ❌ 全モジュール実装
- ❌ 全画面実装
- ❌ 基盤層全体実装

---

## 「やるべき内容」の記載原則 [MANDATORY]

設計書を参照すればわかる実装詳細（プロパティ名、型、メソッドシグネチャ等）は計画書に書かない。設計書の**該当セクションを特定できるレベル**の記述にとどめること。

計画書に実装詳細を転記すると設計書との二重管理になり、不整合の原因となる。

---

## 「必読」列の仕様 [MANDATORY]

タスク実行前に必ず読むべき文書のファイルパスを記載する。

**記載形式**: プロジェクトルートからの相対パス

**正しい例**:

- `specs/feature/design/DES-001_foo_design.md`
- `rules/architecture/domain_core.md`

**誤った例**:

- `foo_design` ❌（曖昧名）
- `specs/*/design/*.md` ❌（glob 禁止）
- `DES-001` ❌（ID のみ）

**必読なし**: `-` を記載

---

## タスクグループ

タスクを細かく分割すると、途中でビルドが壊れる状態が生じるのは普通のことである。
その場合、複数タスクを1つのグループとしてまとめ、グループ完了時にビルドを確認する。

### グループ化の判定基準

「タスクN完了時点でビルドが通るか？」

- **Yes** → 独立タスク
- **No** → タスクグループ化

### グループサイズ制限

- **最大10タスク**を目安とする
- 超過する場合は段階的に分割できないか検討する

### グループ化の記載方法

タスク表の「グループID」列・「ビルド確認」列を使用する:

| 列           | 値                | 説明                     |
| ------------ | ----------------- | ------------------------ |
| `グループID` | `GROUP-001 (1/3)` | グループ ID と通し番号   |
| `グループID` | `-`               | 独立タスク               |
| `ビルド確認` | `タスクごと`      | 独立タスク（デフォルト） |
| `ビルド確認` | `スキップ`        | グループ途中タスク       |
| `ビルド確認` | `グループ完了時`  | グループ最終タスク       |

---

## 依存関係管理

- 依存される側を先に実装する
- 循環依存がないか確認すること
- 依存関係マップは**作業メモ**として使ってよいが、計画書本体には含めない（依存関係はタスク表の列に落とし込む）

---

## 並列実行可能タスクの識別

依存関係がないタスク群は並列実行可能として識別する。依存関係列が「-」かつ同一グループでないタスクは並列実行候補。
