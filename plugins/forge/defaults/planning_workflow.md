# 計画書作成ワークフロー

## 必読文書 [MANDATORY]

**NEVER skip.** 以下の手順で必読文書を動的に特定し、全て読み込むこと:

- `/query-rules` でプロジェクトの計画書フォーマット・タスク管理ルールを取得
- `/query-specs` で対象 Feature の要件定義書・設計書を取得

**フォーマット文書が見つからない場合**: ユーザーに「計画書フォーマットのルール文書（plan_format.md 等）が見つかりませんでした。プロジェクトの `rules/` に計画書フォーマットを用意することを推奨します」と伝え、汎用的なベストプラクティスで進める。

---

## 計画書作成の基本要件

- **作成場所**: `specs/{feature}/plan/{feature}_plan.md`
- **フォーマット**: Markdown (.md) ファイル
- **レビュー**: 計画書作成後に人間のレビューが必須 [MANDATORY]

---

## 0. 開発フローと文書更新

### 文書間の関係性

```
要件定義書 ←→ 設計書 ←→ タスク
    ↓           ↓         ↓
要件トレーサ  設計トレーサ  実装
ビリティ      ビリティ
マトリクス    マトリクス
```

### 開発フロー

1. **要件定義**: 要件定義書作成（要件ID付与）
2. **設計**: 設計書作成（設計IDと要件IDの紐付け）
3. **計画**: タスク定義（設計IDとの紐付け）
4. **実装**: タスク実行（設計書に基づく実装）

### 文書更新ルール

- **新機能追加時**: 要件定義書 → 設計書 → 計画書の順で作成/更新
- **バグ修正時**:
  - 単発: 直接タスク化（設計書不要）
  - 複数タスク: 設計書作成必須
- **設計変更時**: 設計書更新 → 計画書のマトリクス更新

## 1. 開発ルール文書・要件定義書・設計書の取得

- 開発ルール文書は `/query-rules` Skill を使って特定する
- 要件定義書・設計書は `/query-specs` Skill を使って特定する
- **Skill 失敗時**: エラー内容をユーザーに報告し、指示を待つ

## 2.1 バグ修正/新機能追加/リファクタリング

既存計画書がある場合、以下を必ず確認・実行する:

1. **要件定義書への反映確認** - その内容が要件定義書に追記または修正されているか [MANDATORY]
2. **設計書への反映確認** - 設計変更を伴う場合、設計書に反映されているか [MANDATORY]
3. **計画書にタスクを追加する** - 計画書にタスクを追加する場合は「2.2 計画書を作成」を参考に行う [MANDATORY]

---

## 2.2 設計書作成とタスク追加のループ [MANDATORY]

### 開発の進め方

**重要**: 全ての要件を一度に設計するのではなく、以下のループで段階的に進める:

```
1. 優先度の高い要件から設計書を1つ作成
   ↓
2. 人間のレビュー・承認を受ける
   ↓
3. 承認された設計書のタスクを計画書に追加
   ↓
4. 次の要件の設計書作成に戻る（ステップ1へ）
```

### なぜ段階的に進めるのか

- **レビューの質向上**: 1設計書ずつ丁寧にレビュー可能
- **早期実装開始**: 承認済み設計のタスクから順次実装開始可能
- **フィードバック反映**: 先行設計の知見を後続設計に活かせる
- **リスク低減**: 大規模な手戻りを防げる

---

## 2.3 計画書を作成・更新

計画書は以下のセクションで構成することを推奨する（プロジェクトの `plan_format.md` がある場合はそちらに従う）:

1. **要件トレーサビリティマトリクス** - 要件IDから設計IDへの対応
2. **設計トレーサビリティマトリクス** - 設計IDからタスクIDへの対応
3. **タスク一覧** - 実装タスクの詳細
4. **改定履歴** - 計画書の変更履歴

### タスク作成の詳細手順 [MANDATORY]

#### タスクの抽出方法

設計書からタスクを抽出する:

1. 設計書のモジュール一覧・クラス図から実装対象を列挙
2. 各モジュールを1Agent実行で完結する単位に分割
3. 依存関係を整理（依存される側から実装）

#### タスクIDの採番

プロジェクトのフォーマットルールに従う。ルールがない場合は `TASK-001`, `TASK-002` 等の連番を使用。

#### 優先度の設定

プロジェクトのフォーマットルールに従う。ルールがない場合は数値が大きいほど優先度が高い（例: 1〜99）。

#### task-executor実行を前提とした計画作成 [MANDATORY]

- **1タスク1Agent実行の原則**: 各タスクは1回の Agent 実行で完結する単位
- **ビルド・テスト成功必須**: タスク完了時にビルド・テストが成功する規模
- **依存関係の明確化**: 先行タスクをタスクIDで明記
- **並列実行可能なタスク識別**: 依存関係がないタスクは並列実行可能

### タスクの粒度

1. **1つのAgent実行で完結する単位**
   - 1つのモジュール、1つのクラス/ファイルなど
   - 複数の独立したモジュールを1タスクにまとめない

2. **独立して実装・テスト可能な単位**
   - 他のタスクの完了を待たずに着手できる
   - タスク完了時にビルド・テストが成功すること [MANDATORY]

3. **目安となる規模**
   - 1つのファイル、または密接に関連する2〜3つのファイル
   - やるべき内容は5〜10項目程度に抑える

### タスクグループ [MANDATORY]

複数ファイルの同期変更が技術的に必須の場合、複数タスクを1つのグループとして扱う。

#### グループ化の判定基準

「タスクN完了時点でビルドが通る状態を作れるか？」
- **Yes** → 独立タスク（タスクごとにビルド確認）
- **No** → タスクグループ化（グループ完了時にビルド確認）

#### グループ化は最終手段 [MANDATORY]

以下の回避策を先に検討すること:
1. **互換レイヤー・Adapter 導入** - 旧インターフェースを残して段階的に移行
2. **Feature Flag / 条件分岐** - Optional プロパティとして追加し段階的に必須化
3. **段階的移行（Deprecation）** - typealias 等で旧名を残して段階的に移行

#### 確認 [MANDATORY]

- 計画書がプロジェクトのフォーマット（`/query-rules` で取得）に従っていることを確認
- 要件トレーサビリティマトリクスが全要件を網羅していることを確認
- 設計トレーサビリティマトリクスが全設計書をカバーしていることを確認
- タスクの完全性を確認（全設計書がタスクに反映されているか、依存関係に循環がないか）
- 計画書の承認を人間に依頼すること

## 2.4 AIレビュー実施 [MANDATORY]

計画書作成/更新時点で `/forge:review plan` で自動レビュー:

```
/forge:review plan [作成した計画書のファイルパス]
```

致命的問題は修正を行う。結果を報告した後、次のステップへ進む。

**Skill 失敗時**: `/forge:review plan` が実行できない場合、チェック項目を手動で確認し、人間にレビューを依頼する。

---

## 2.5 specs ToC 更新について

計画書自体は specs ToC の対象外のため、計画書作成のみの場合は更新不要。
要件定義書・設計書を新規作成・更新した場合のみ、`/create-specs-toc` を実行すること。
